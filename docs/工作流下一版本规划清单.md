# 工作流下一版本规划清单

## 1. 文档目标

本文用于明确 `go-common/workflow` 在 `v1.3.0` 正式版之后的下一阶段增强范围，避免后续需求继续和正式版基线混在一起。

`v1.3.0` 已经定义为正式稳定基线，后续所有新增能力默认都属于下一版本规划，而不再回头改变本次正式版的定位。

## 2. 版本边界

当前已经纳入 `v1.3.0` 正式版的内容包括：

- 标准接口收口
- 流程发起
- 待办
- 已办
- 任务上下文
- 表单引用解析
- 进度视图
- 时间线
- 统一流程图入口
- LDAP 用户、上级、部门能力
- 单流程与父子流程基础场景支持

下面这些内容，不属于 `v1.3.0`，统一归入下一版本规划。

## 3. 下一版本目标分层

为了避免后续目标继续发散，下一版本建议按三层推进：

1. 查询性能增强
2. 复杂流程能力增强
3. 平台运维与治理能力增强

只有前一层收口后，再进入后一层，不建议并行摊大饼。

## 4. 第一层：查询性能增强

这一层的目标不是增加新接口，而是把当前已经验证正确的接口，做成更适合中大规模场景使用的实现。

### 4.1 列表默认轻量化继续收口

目标：

- 待办、已办默认只返回列表必需字段
- 完整进度、完整时间线、完整流程图继续放在详情页单独查询

需要继续确认的点：

- 列表里哪些字段属于稳定基础字段
- 哪些字段属于增强字段
- 哪些字段必须通过 `includeProgress=true` 才允许返回

### 4.2 流程摘要缓存

目标：

- 把标题、业务单号、当前节点、当前处理人、候选人等高频摘要做缓存或投影

重点：

- `processInstanceId -> progress summary` 的缓存策略
- 缓存失效与补偿刷新机制
- Redis 不可用时的降级策略

### 4.3 候选待办索引化

目标：

- 把“某个用户当前可见哪些待办”从读时推导改成写时展开

重点：

- 候选用户索引
- 候选组索引
- 用户与组变更后的重建策略

### 4.4 已办历史投影

目标：

- 让“我的已办”优先查询本地投影，而不是长期完全依赖 Flowable history 接口

重点：

- 完成事件写入
- 回调补偿
- 定时对账

## 5. 第二层：复杂流程能力增强

这一层才进入更复杂的流程语义，不建议在第一层还没收口时就直接推进。

### 5.1 更复杂 EL 表达式支持

当前已知边界：

- 简单条件表达式支持较稳定
- 复杂 EL 函数、动态表达式支持仍有限

下一步目标：

- 补齐更复杂网关条件解析
- 明确“不支持”的表达式清单

### 5.2 会签、多实例

目标：

- 明确会签节点、多实例节点在待办、已办、进度时间线中的展示规则

重点先回答三个问题：

1. 会签节点在已办里是按人展示，还是按节点汇总展示
2. 多实例节点在时间线里是重复展开，还是聚合展示
3. 流程图当前节点高亮是按任务实例，还是按活动节点

### 5.3 更深层级 `callActivity`

`v1.3.0` 已支持父子流程基础场景，但更深层嵌套仍应视为下一版本范围。

下一步目标：

- 支持多层级父子流程关系查询
- 支持更稳定的复合流程图/复合时间线聚合规则
- 明确子流程异常、回退、重入时的展示规则

### 5.4 子流程与父流程视图联动增强

目标：

- 在统一视图中更清晰展示父流程和子流程当前位置
- 继续优化复合流程图的可读性

这一块要坚持一个原则：

- 不能影响当前单流程图的稳定展示

## 6. 第三层：平台运维与治理能力增强

这一层属于平台化能力，不急着先做，但后续如果要让更多业务组统一接入，就必须补。

### 6.1 回调读模型治理

目标：

- 明确回调事件类型
- 明确回调幂等键
- 明确失败重试与死信处理

### 6.2 对账与补偿工具

目标：

- 支持按流程实例、按业务单号做对账
- 支持修复绑定表、摘要缓存、待办投影、已办投影

### 6.3 接入诊断工具

目标：

- 快速判断某个业务系统是配置问题、租户问题、表单 key 问题，还是用户组映射问题

建议后续补一类平台自检接口或诊断脚本，至少覆盖：

- Flowable REST 连通性
- LDAP 连通性
- 表单元数据库连通性
- 回调可达性
- 流程定义 / 表单 key / 租户匹配性

### 6.4 统一告警与监控指标

目标：

- 给平台层补基础监控口径

建议至少覆盖：

- Flowable REST 请求耗时
- 待办列表查询耗时
- 已办列表查询耗时
- 回调成功率
- 回调失败重试次数
- 投影写入失败次数

## 7. 建议优先级

下一版本建议优先级如下：

### P1

- 列表默认轻量化继续收口
- 流程摘要缓存
- 候选待办索引化
- 已办历史投影

### P2

- 更复杂 EL 支持
- 会签、多实例展示规则
- 更深层级 `callActivity` 聚合规则

### P3

- 对账补偿工具
- 接入诊断工具
- 平台监控与告警

## 8. 建议版本策略

后续版本建议不要直接再跳一个大而全的正式版，而是分两步：

1. 先做一个性能与复杂流程增强版
2. 再做一个治理和运维增强版

建议节奏：

- `v1.3.x`：只做 bugfix 和小幅增强，不动正式接口基线
- `v1.4.0`：承接性能优化与复杂流程增强第一阶段
- `v1.5.0`：承接治理、对账、诊断、监控能力

## 9. 当前结论

从现在开始，后续开发应遵循下面这条原则：

- `v1.3.0` 负责“正式稳定可接入”
- 《工作流下一版本规划清单》负责“下一阶段要增强什么”

这样正式版和后续规划就彻底分开了，目标会更清晰，版本边界也会更稳定。
