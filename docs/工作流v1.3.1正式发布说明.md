# go-common 工作流平台 v1.3.1 正式发布说明

## 1. 版本信息

- 版本号：`v1.3.1`
- 发布日期：`2026-04-15`
- 版本定位：`v1.3.0` 之后的易用性增强版

## 2. 发布摘要

本次版本不是推翻 `v1.3.0` 的正式能力，而是在保持主链路兼容的前提下，继续把“业务项目接入工作流”这件事做得更简单、更明确。

`v1.3.1` 重点解决的是：

- 业务项目如何更少写代码就把工作流接口注册起来
- LDAP / HTTP 目录服务如何尽量共用一套 BPMN
- 下一环节分配如何通过统一配置启用
- 多数据库 / Redis 配置如何尽量统一成结构化写法
- 新开发者拿到版本后如何更快定位配置问题
- Flowable UI 如何原生显示 LDAP 组并稳定使用 `candidateGroups`
- 业务侧 LDAP 全量同步如何改成后台批次执行，避免接口长时间阻塞

## 3. 相比 v1.3.0 的核心新增

### 3.1 业务项目推荐一行注册

当前推荐业务项目直接使用：

```go
workflowapi.MustRegisterFromConfig(server)
```

这行代码会自动完成：

- 工作流上下文解析器初始化
- 目录服务初始化
- 自动分配服务初始化
- Flowable REST client 初始化
- 表单引用服务初始化
- 标准工作流路由注册

相比之前手动 `NewDefaultModuleFromConfig()` + `module.Register(server)` 的方式，更适合作为标准接入入口。

### 3.2 目录服务 provider 统一配置化

目录服务来源正式统一为：

- `workflow.directory.provider=none`
- `workflow.directory.provider=ldap`

同时保留兼容：

- 如果未显式配置 provider，但检测到 `ldap_*` 或 `workflow.directory.ldap.*`，系统仍会自动切到 LDAP 模式

这使得新项目配置更明确，老项目升级也不会被硬性打断。

### 3.3 自动分配 provider 正式可用

自动分配服务正式统一支持：

- `workflow.assignment.provider=none`
- `workflow.assignment.provider=directory`
- `workflow.assignment.provider=ldap`

标准回写变量为：

- `nextAssignee`
- `nextCandidateUsers`
- `nextCandidateGroups`

这样业务项目在“发起流程”和“完成任务”时，不必在每个业务接口里都手动拼下一环节分配逻辑。

### 3.4 HTTP / LDAP 身份归一契约增强

本次版本继续强化：

- `workflow.identity.role_aliases`
- `workflow.identity.group_aliases`

它们的作用是：

- 把 HTTP、LDAP、JWT、业务库里的原始角色 / 组编码统一归一
- 让 Flowable BPMN 尽量只认一套稳定候选组编码
- 降低 provider 切换时修改流程图的概率

### 3.5 数据源结构化配置能力更适合业务项目

本次版本继续推荐业务项目统一使用：

- `databases.mysql.*`
- `databases.postgres.*`
- `databases.kingbase.*`
- `databases.dm.*`
- `databases.mongo.*`
- `databases.redis.*`

其中 MySQL 已支持直接通过结构化字段表达：

- `charset`
- `parse_time`
- `loc`
- `params`

也就是说，像以前这种 DSN：

- `charset=utf8mb4&parseTime=True&loc=Local`

现在可以直接拆成 YAML 字段配置，不必强依赖手写长 DSN。

### 3.6 启动期摘要日志增强

工作流模块注册时，会输出更清晰的启动摘要，帮助开发者快速判断：

- 标准接口有没有注册成功
- 当前目录服务 provider 是什么
- 当前自动分配 provider 是什么
- 当前 identity alias 是否生效

这对第一次接入和问题排查都很关键。

### 3.7 LDAP Group 投影正式补齐

本次版本把“Flowable 原生 LDAP 组模型”也补齐了。

原因是：

- 业务系统原有目录里通常只有 `people / departments / positions`
- 但 Flowable UI 查询“组”时，实际上需要真正的 LDAP Group（例如 `groupOfUniqueNames`）

因此当前版本新增：

- `workflow.directory.ldap.group_ou`
- 业务侧全量同步时，自动生成：
  - 部门组：`dept:<departmentCode>`
  - 岗位组：`pos:<positionCode>`

这样带来的直接好处是：

- Flowable UI 可以正常显示“组”
- BPMN 里的 `candidateGroups` 可以长期稳定使用这套编码
- 原有用户 / 部门 / 岗位 / 上级关系模型不受影响

### 3.8 LDAP 全量同步改为异步批次执行

此前 `/api/v1/ldap_sync/full` 在用户、部门、岗位、组较多时会阻塞数分钟。

当前版本改成：

- 调用 `/api/v1/ldap_sync/full` 后立即返回 `batch_id` / `batch_no`
- 后台 goroutine 执行真正同步
- 通过 `/api/v1/ldap_sync/history` 查看批次列表
- 通过 `/api/v1/ldap_sync/get` 查看批次详情

这使得业务前端和运维脚本不再需要长时间等待 HTTP 请求返回。

### 3.9 Flowable 权限表名大小写自动兼容

当前版本同步 Flowable UI 权限时，会自动解析实际表名：

- `ACT_ID_PRIV` / `act_id_priv`
- `ACT_ID_PRIV_MAPPING` / `act_id_priv_mapping`

从而兼容不同 MySQL 环境里的大小写差异，不再因为表名大小写不一致误报“表不存在”。

## 4. 对业务项目的直接价值

### 4.1 接入代码更少

业务项目通常只需要：

1. 配置 `workflow.flowable.*`
2. 配置 `workflow.api.*`
3. 配置 `workflow.context.*`
4. 视情况配置 `workflow.directory.*`
5. 启动时执行 `workflowapi.MustRegisterFromConfig(server)`

### 4.2 Provider 切换更稳

如果业务项目要在 `http` 和 `ldap` 之间切换目录来源，当前推荐做法是：

1. BPMN 固定候选组只写稳定编码
2. 上游原始角色 / 岗位 / 组编码通过 alias 归一
3. 分配变量名固定使用 `nextAssignee` / `nextCandidateUsers` / `nextCandidateGroups`

### 4.3 新开发更容易排障

接入失败时，开发者可以优先从以下三类问题定位：

1. `workflow.flowable.*` 是否完整
2. `workflow.directory.provider` 是否正确
3. 启动摘要日志里 provider / alias 数量是否符合预期

## 5. 推荐升级动作

从 `v1.3.0` 升到 `v1.3.1`，建议按下面顺序做：

1. 把工作流注册代码统一成 `workflowapi.MustRegisterFromConfig(server)`
2. 显式配置 `workflow.directory.provider`
3. 如有自动分配需求，显式配置 `workflow.assignment.provider`
4. 如需 HTTP / LDAP 共用 BPMN，补齐 `workflow.identity.role_aliases` / `group_aliases`
5. 业务数据库与 Redis 尽量改为 `databases.*` 结构

## 6. 兼容说明

当前版本仍兼容以下老方式：

- `workflowapi.NewDefaultModuleFromConfig()` + `module.Register(server)`
- 全局 LDAP 配置 `ldap_*`
- 历史 RBAC Redis 配置 `redis_addr` / `redis_password`

因此，`v1.3.1` 的升级目标是“更清晰、更统一”，而不是“强迫所有老项目重写配置”。

综合接入易用性、配置标准化、身份契约稳定性和兼容策略，当前版本建议正式发布为：

- `v1.3.1`

并作为：

- `v1.3.0` 之后的推荐升级版本
- 后续新业务项目优先采用的工作流接入基线版本
