# 工作流正式版发布执行步骤

## 1. 文档目标

本文用于指导 `go-common/workflow` 正式版本发布时的实际执行动作，包括发布前核对、版本号确认、tag 建议、发布说明正文使用方式以及发布后通知动作。

本文解决的问题不是“版本是否可发”，而是“确定可发之后怎么真正发出去”。

当前默认口径以 `v1.3.1` 为准；如需回看 `v1.3.0` 首次正式发布过程，请结合对应历史归档文档阅读。

## 2. 当前建议发布版本

建议当前正式版本号：

- `v1.3.1`

版本依据请参考：

- 《工作流v1.3.1正式发布说明》
- 《工作流正式版发布检查结果》
- 《工作流版本变更说明》

## 3. 发布前最后核对

发布前请再按下面顺序核对一次：

1. 《工作流正式版发布检查结果》内容已确认
2. `CHANGELOG.md` 已更新到 `v1.3.1`
3. 《工作流v1.3.1正式发布说明》已准备好
4. 《工作流配置说明》已准备好给业务组使用
5. 正式接口、兼容接口、接入文档口径一致
6. 本地代码不再包含临时调试逻辑

## 4. 建议发布材料

正式发布时，建议至少同时对外提供下面这些材料：

- `CHANGELOG.md`
- 《工作流v1.3.1正式发布说明》
- 《工作流v1.3.1业务项目最小接入手册》
- 《工作流配置说明》
- 《工作流平台标准接口说明》
- 《工作流业务接入指南》
- 《工作流后端接入交付模板》

如果是发给内部业务组，建议再附上：

- 《工作流测试验收模板》
- 《工作流正式版发布检查结果》

## 5. 建议 tag 策略

建议正式 tag 直接使用：

- `v1.3.1`

如果团队内部还有灰度、验证环境需要保留，也可以同时采用下面这种节奏：

1. 先在验证阶段使用 `v1.3.1-rc1`
2. 回归通过后打正式 tag `v1.3.1`

如果当前已经完成正式版验收，则可以直接打：

- `v1.3.1`

## 6. 建议执行命令

如果当前仓库使用 Git 管理，正式发版时建议按下面顺序执行：

```bash
git status
git add CHANGELOG.md docs/
git commit -m "release: workflow v1.3.1"
git tag -a v1.3.1 -m "go-common workflow v1.3.1"
git show v1.3.1 --stat
```

如果团队要求推送远端，再执行：

```bash
git push origin <当前分支>
git push origin v1.3.1
```

## 7. 发布说明正文使用方式

正式发版时，建议直接使用：

- 《工作流v1.3.1正式发布说明》

如果是群消息或邮件简版通知，可使用：

- 《工作流正式版发布说明模板》中的简版通知模板

## 8. 发布后建议通知内容

建议发布通知至少说明以下几点：

1. 正式版本号
2. 正式推荐接口清单
3. 统一流程图入口已改为 `diagram-view`
4. 新业务组必须先看《工作流配置说明》
5. 老业务组如果仍用兼容接口，当前仍可继续运行，但应安排迁移

## 9. 发布后第一批业务组接入建议

正式版发布后，建议不要一开始就让所有业务组同时接入，而是优先选：

1. 一个标准单流程业务
2. 一个带候选人或候选组的审批流业务
3. 一个含 `callActivity` 的复杂业务

这样可以尽快验证：

- 标准接入模板是否足够清晰
- 配置说明是否足够完整
- 兼容问题是否已经收敛

## 10. 当前执行结论

按照当前文档、版本变更说明和发布材料清单，`go-common/workflow` 已具备发布 `v1.3.1` 所需的材料完备度。

如果你决定现在发版，建议直接使用本文件中的命令和材料清单执行。
