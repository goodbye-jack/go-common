# 工作流后端接入交付模板

## 1. 文档用途

这份文档是给业务系统后端开发组直接执行的交付模板。

使用方式：

- 后端负责人先通读一遍
- 按文档里的章节建立开发任务
- 每完成一个模块就补齐对应接口和联调记录

这份文档不讨论前端实现细节，也不讨论测试排期，只关注后端如何把业务系统和 `go-common/workflow` 正确接起来。

## 2. 后端接入目标

后端接入完成后，必须具备下面这些能力：

1. 能识别当前登录用户，并解析出工作流用户上下文
2. 能把业务记录先写入业务表
3. 能带着业务标识发起流程
4. 能查询“我的申请”
5. 能查询“我的待办”
6. 能查询“我的已办”
7. 能提供任务上下文给前端
8. 能提供流程图和流程时间线给前端
9. 能接收任务处理结果并驱动流程继续流转

## 3. 后端职责边界

### 3.1 后端必须负责

- 登录态解析
- 当前用户信息透传到工作流上下文
- 业务表持久化
- 发起流程
- 业务数据和工作流数据的合并
- 任务完成时的变量映射
- 业务状态更新

### 3.2 后端不要负责

- 使用 Flowable 官方页面做业务前台
- 使用 Flowable 底表直接替代业务表
- 把业务表字段直接耦合到 `go-common/workflow`

## 4. 接入前准备

在编码前，必须先确认下面这些输入已经齐备：

### 4.1 工作流基础配置

- Flowable REST 地址
- Flowable REST 用户名
- Flowable REST 密码
- 流程定义 key 或 definitionId
- 租户标识规则
- 系统编码规则

### 4.2 LDAP 相关配置

- LDAP 地址
- bind DN
- bind 密码
- base DN
- people OU
- department OU
- position OU

### 4.3 业务侧输入

- 业务表结构
- 业务唯一标识生成规则
- 业务状态枚举
- 各节点处理变量映射规则
- 节点表单是否作为参考字段使用

## 5. 推荐数据库表结构

每个接入工作流的业务表，建议最少具备如下字段：

| 字段名 | 类型建议 | 说明 |
| --- | --- | --- |
| `biz_id` | varchar(64) | 业务单唯一标识 |
| `biz_type` | varchar(64) | 业务类型 |
| `title` | varchar(255) | 业务标题 |
| `status` | varchar(64) | 业务状态 |
| `process_instance_id` | varchar(64) | Flowable 流程实例 ID |
| `started_by_user_id` | varchar(64) | 发起人账号 |
| `started_by_user_name` | varchar(128) | 发起人姓名 |
| `created_at` | datetime | 创建时间 |
| `updated_at` | datetime | 更新时间 |

建议索引：

- `biz_id` 主键或唯一索引
- `process_instance_id` 普通索引
- `(started_by_user_id, updated_at)` 联合索引

## 6. 必要配置模板

推荐在业务系统配置文件中至少提供下面这些配置：

```yaml
workflow:
  flowable:
    base_url: "http://localhost:8087/flowable-rest/service"
    username: "rest-admin"
    password: "test"
    timeout_seconds: 15
    group_prefix: ""
    role_prefix: "role_"
  api:
    prefix: "/api"
    sso_enabled: true
    roles: ""
    callback_path: "/flowable/callback"
    callback_secret: ""
  context:
    user_name_header: "X-Workflow-User-Name"
    system_code_header: "X-System-Code"
    groups_header: "X-Workflow-Groups"
    roles_header: "X-Workflow-Roles"
  formref:
    db_instance: "flowable_meta"

ldap_addr: "ldap://127.0.0.1:8389"
ldap_bind_dn: "cn=admin,dc=example,dc=com"
ldap_bind_password: "password"
ldap_base_dn: "dc=example,dc=com"
ldap_people_ou: "ou=people"
ldap_department_ou: "ou=departments"
ldap_position_ou: "ou=positions"
ldap_use_tls: false
```

说明：

- `workflow.flowable.*` 用于工作流主链路
- `workflow.api.*` 用于标准路由前缀、登录态要求和回调入口
- `workflow.context.*` 用于定义工作流模块读取哪些请求头来补齐 `userName`、`systemCode`、`groups`、`roles`
- `workflow.formref.db_instance` 只在表单引用解析启用时需要
- LDAP 配置用于当前人、上级、部门等目录能力

如果业务系统需要待办投影缓存增强，还建议增加：

- `workflow.flowable.todo_projection_ttl_seconds`
- `workflow.flowable.todo_projection_task_prefix`
- `workflow.flowable.todo_projection_user_prefix`
- `workflow.flowable.todo_projection_group_prefix`
- `workflow.flowable.process_runtime_task_prefix`

如果业务系统要直接照着正式版模板落地，而不是只看这段示例，请同时使用《工作流配置说明》。

### 6.1 除了配置文件之外必须完成的事情

很多业务组只改了 `config.yaml`，但没做下面这些动作，最后就会出现“服务能启动，能力不能用”的问题。

必须完成：

1. 启动阶段注册工作流模块
2. 登录中间件能够解析当前登录用户
3. 租户中间件能够解析当前 `tenantId`
4. 业务系统自己的发起接口内部调用 `POST /api/process/start`
5. 如启用回调，确保 `POST /flowable/callback` 对 Flowable 可访问

### 6.2 Flowable 平台侧要同步确认的配置

后端接入不仅是业务系统本地写配置，Flowable 平台侧也必须一起确认：

1. 流程定义 key 与业务系统调用的 `processDefinitionKey` 一致
2. 节点 `assignee`、`candidateUsers`、`candidateGroups` 与 LDAP 用户、组口径一致
3. 如启用表单引用解析，节点 `formKey` 与 Flowable 表单定义 key 一致
4. 租户策略统一，不要出现流程定义、表单定义、实例运行使用不同租户
5. 如启用 `callActivity`，子流程定义必须已部署且可按当前租户策略解析

## 7. 路由注册模板

业务服务必须把 `go-common/workflow` 注册到自己的 HTTP Server 中。

参考模板：

```go
package main

import (
	workflowapi "github.com/goodbye-jack/go-common/workflow/api"
)

func registerWorkflow(server *HTTPServer) {
	module, err := workflowapi.NewDefaultModuleFromConfig()
	if err != nil {
		panic(err)
	}
	module.Register(server)
}
```

要求：

- 必须注册在正式业务服务中
- 不要再单独维护一套平行的工作流路由

### 7.1 当前用户上下文要求

工作流模块不会替业务系统做认证，它只会从现有请求上下文里读取信息。

后端必须保证：

- `commonhttp.GetUser(c)` 能返回当前登录用户
- `commonhttp.GetTenant(c)` 能返回当前租户
- 如果业务系统有显示姓名、系统编码、组、角色信息，要通过请求头传入或在中间件里补入

默认头名如下：

- `X-Workflow-User-Name`
- `X-System-Code`
- `X-Workflow-Groups`
- `X-Workflow-Roles`

如果业务系统不想使用默认头名，必须同步改 `workflow.context.*` 配置。

## 8. 发起流程实现模板

### 8.1 标准实现顺序

发起流程必须按下面顺序实现：

1. 校验业务入参
2. 生成 `bizId`
3. 写业务表
4. 调工作流发起流程
5. 保存 `processInstanceId`
6. 返回前端

### 8.2 标准请求体模板

```json
{
  "processDefinitionKey": "artifact_appraise",
  "bizId": "APPRAISE-20260311-0001",
  "bizType": "artifact_appraise",
  "title": "馆藏文物鉴定申请单-0001",
  "name": "馆藏文物鉴定申请单-0001",
  "variables": {
    "payloadRef": "APPRAISE-20260311-0001",
    "tenantId": "test-demo",
    "systemCode": "sysA",
    "callbackUrl": "http://localhost:9080/flowable/callback",
    "callbackEvents": "PROCESS_STARTED,NODE_STARTED,NODE_ENDED,PROCESS_ENDED",
    "needExpert": true
  }
}
```

### 8.3 发起成功后必须回写的字段

- `process_instance_id`
- `status`
- `updated_at`

### 8.4 发起失败时必须处理的内容

- 业务表状态应标记为失败态或待重试态
- 必须保留失败日志
- 必须能通过 `bizId` 找回这条失败记录

## 9. “我的申请”实现模板

“我的申请”一定要基于业务表实现。

### 9.1 查询方式

按当前登录人查询业务表：

- `started_by_user_id = currentUserId`

### 9.2 每条记录补充的工作流信息

- `progress-view`
- 当前节点
- 当前办理人
- 当前候选人
- 当前候选组

### 9.3 不允许的做法

- 不允许直接拿“我的已办”当“我的申请”
- 不允许直接查 Flowable 底表做业务申请列表

## 10. 待办和已办实现模板

### 10.1 我的待办

使用接口：

- `GET /api/me/tasks/todo`

后端需要做的包装：

- 补业务标题
- 补业务单号
- 补业务进度摘要

### 10.2 我的已办

使用接口：

- `GET /api/me/tasks/done`

后端需要做的包装：

- 补业务标题
- 补完成时间
- 补进度查询入口

## 11. 任务上下文实现模板

使用接口：

- `GET /api/tasks/{taskId}/context`

后端合并规则：

1. 先查工作流任务上下文
2. 再按 `payloadRef` 或 `bizId` 查业务表
3. 把业务标题、业务数据和任务信息合并成一份结构

必须保证前端从任务上下文接口里能拿到：

- `taskId`
- `activityId`
- `processInstanceId`
- `bizId`
- `payloadRef`
- `title`
- `variables`
- `formRef`

## 12. 任务处理实现模板

使用接口：

- `POST /api/tasks/{taskId}/complete`

标准顺序：

1. 接收业务处理页面提交的数据
2. 映射成工作流变量
3. 调工作流完成任务接口
4. 更新业务表状态
5. 返回处理结果

### 12.1 变量映射模板

示例：

- 页面字段 `result` -> 流程变量 `appraiseResult`
- 页面字段 `needExpert` -> 流程变量 `needExpert`
- 页面字段 `comment` -> 流程变量 `comment`
- 页面字段 `reworkComment` -> 流程变量 `reworkComment`

要求：

- 变量命名规则必须固定
- 同一业务系统内不要一会儿传中文值、一会儿传英文值

## 13. 流程进度实现模板

后端需要提供给前端的标准能力：

- 进度视图
- 进度时间线
- BPMN XML

推荐接口：

- `GET /api/process-instances/{id}/progress-view`
- `GET /api/process-instances/{id}/progress-timeline`
- `GET /api/biz/{bizId}/progress-view`
- `GET /api/biz/{bizId}/progress-timeline`
- `GET /api/process/instance/{id}/definition-xml`

要求：

- 不要在业务系统里自己再拼一套流程图逻辑
- 不要把 BPMN XML 存进业务表

## 14. 当前人员展示规则

后端必须保证把这些字段稳定返回给前端：

- `summary.currentAssignees`
- `summary.currentCandidateUsers`
- `summary.currentCandidateGroups`
- `timeline.items[].candidateUsers`
- `timeline.items[].candidateGroups`

返回语义：

1. 有 `assignee` 时，表示真实办理人
2. 没有 `assignee` 但有候选人或候选组时，表示“待认领”
3. 非人工节点才允许展示“系统”

## 15. 回调实现模板

建议保留工作流回调入口。

回调至少承担以下职责：

- 记录节点开始事件
- 记录节点结束事件
- 记录流程结束事件
- 更新业务状态

回调不是业务列表数据源，但它能帮助：

- 补充运行态任务信息
- 优化时间线构建
- 提高状态同步及时性

## 16. 推荐开发顺序

后端推荐按下面顺序开发：

1. 配置工作流和 LDAP
2. 注册工作流路由
3. 建业务表
4. 完成“业务创建 + 发起流程”
5. 完成“我的申请”
6. 完成“待办/已办”
7. 完成“任务上下文”
8. 完成“任务处理”
9. 完成“流程进度”
10. 接回调

## 17. 联调前自测要求

后端开发自测时，必须至少验证：

- 发起成功
- `processInstanceId` 已回写
- 待办能查到
- 标题不丢失
- 任务上下文能返回 `title`
- 完成任务后下一节点待办能查到
- 已办能查到
- `progress-view` 正常
- `progress-timeline` 正常

## 18. 交付物要求

后端组提交联调时，必须提供下面这些材料：

- 业务表结构说明
- 发起接口说明
- 我的申请接口说明
- 待办接口说明
- 已办接口说明
- 任务处理接口说明
- 流程变量映射表
- 一组联调成功的请求与响应示例

## 19. 验收标准

只有当下面这些结论都成立时，才说明后端接入完成：

- 发起流程稳定
- 标题链路稳定
- `bizId` 链路稳定
- `processInstanceId` 链路稳定
- 我的申请正常
- 待办正常
- 已办正常
- 任务上下文正常
- 流程进度正常
- 处理任务正常

这时才允许把工作流能力交给前端组和测试组继续推进。

## 20. 后端组可直接执行的开发任务单模板

下面这份模板可以直接复制到项目管理系统里，拆给后端开发人员执行。

### 20.1 任务一：接入工作流基础配置

任务目标：

- 完成 `workflow.flowable.*` 配置
- 完成 `workflow.api.*` 配置
- 完成 `workflow.context.*` 配置
- 完成 LDAP 配置

交付结果：

- 配置文件可启动
- 服务启动后工作流模块注册成功
- 可以调用健康检查或最小待办接口做连通性验证

完成标准：

- 服务正常启动
- Flowable REST 连通
- LDAP 连通

### 20.2 任务二：建立业务表并打通发起流程

任务目标：

- 建立业务表
- 实现业务创建接口
- 发起流程并回写 `process_instance_id`

交付结果：

- 新建一条业务记录后可自动发起流程
- 数据库存下业务记录和流程实例 ID

完成标准：

- 发起接口返回成功
- 业务表记录完整
- Flowable 中能查到对应流程实例

### 20.3 任务三：实现“我的申请”

任务目标：

- 基于业务表实现“我的申请”接口
- 每条记录补流程进度摘要

交付结果：

- 当前登录人能查到自己发起的业务单
- 每条记录能看到标题、状态、当前节点、当前人员

完成标准：

- 不依赖 Flowable 已办数据
- 服务重启后列表仍存在

### 20.4 任务四：实现待办、已办、任务上下文

任务目标：

- 对接待办
- 对接已办
- 对接任务上下文

交付结果：

- 待办可显示标题
- 已办可显示标题和完成时间
- 任务上下文可同时返回任务信息和业务数据

完成标准：

- 待办标题不丢失
- 已办标题不丢失
- 任务上下文返回 `bizId`、`title`、`formRef`

### 20.5 任务五：实现任务处理和进度查询

任务目标：

- 对接任务完成接口
- 对接流程图、时间线、BPMN XML
- 更新业务状态

交付结果：

- 任务处理后流程继续流转
- 前端可查看流程进度

完成标准：

- 处理成功后下一节点待办出现
- 时间线顺序正确
- 当前人员显示正确

## 21. 后端接口交付单模板

后端组提测时，建议按下面格式提交接口说明，而不是零散发聊天记录。

### 21.1 发起接口

- 接口名称：`POST /api/applications`
- 功能说明：创建业务单并发起流程
- 鉴权要求：登录态
- 请求字段：
  - `title`
  - 业务主字段
  - 可选扩展字段
- 响应字段：
  - `bizId`
  - `processInstanceId`
  - `status`
- 失败场景：
  - 参数校验失败
  - 业务表写入失败
  - Flowable 发起失败

### 21.2 我的申请接口

- 接口名称：`GET /api/applications`
- 功能说明：查询当前登录人发起的业务单
- 鉴权要求：登录态
- 响应字段：
  - `bizId`
  - `title`
  - `status`
  - `processInstanceId`
  - `progressSummary`

### 21.3 待办接口

- 接口名称：`GET /api/todo`
- 功能说明：查询当前登录人的待办
- 响应字段：
  - `taskId`
  - `taskName`
  - `title`
  - `bizId`
  - `processInstanceId`
  - `progressSummary`

### 21.4 已办接口

- 接口名称：`GET /api/done`
- 功能说明：查询当前登录人的已办
- 响应字段：
  - `taskId`
  - `taskName`
  - `title`
  - `bizId`
  - `completedAt`

### 21.5 任务上下文接口

- 接口名称：`GET /api/tasks/{taskId}/context`
- 功能说明：加载任务详情页所需上下文
- 响应字段：
  - `taskId`
  - `processInstanceId`
  - `bizId`
  - `title`
  - `variables`
  - `formRef`
  - `bizData`

### 21.6 任务处理接口

- 接口名称：`POST /api/tasks/{taskId}/complete`
- 功能说明：提交任务处理结果并驱动流程继续流转
- 请求字段：
  - `action`
  - `comment`
  - 业务节点扩展字段
- 响应字段：
  - `success`
  - `nextTask`
  - `status`

## 22. 流程变量映射表模板

每个业务系统都建议维护一份映射表，避免开发和测试阶段变量口径混乱。

| 页面字段 | 含义 | 流程变量名 | 变量类型 | 示例值 | 备注 |
| --- | --- | --- | --- | --- | --- |
| `result` | 审批结果 | `appraiseResult` | string | `approved` | 必填 |
| `comment` | 审批意见 | `comment` | string | `同意进入下一环节` | 选填 |
| `needExpert` | 是否专家复核 | `needExpert` | boolean | `true` | 条件网关使用 |
| `reworkComment` | 补充材料说明 | `reworkComment` | string | `请补充图片` | 驳回时使用 |

要求：

- 一张表覆盖一个业务流程
- 变量名统一使用英文
- 变量值枚举在文档里固定，不允许口头约定

## 23. 常见问题排查表

| 现象 | 优先检查项 | 常见原因 |
| --- | --- | --- |
| 待办里没有标题 | 流程变量、业务表兜底逻辑 | 发起时没传 `title`，或列表没补业务标题 |
| 已办里没有记录 | 当前登录用户、Flowable 历史任务查询条件 | 实际处理人和登录人不一致 |
| 时间线显示成系统 | `assignee`、候选人、候选组是否返回 | 当前节点未签收，但前端没展示待认领 |
| 表单引用解析失败 | `workflow.formref.db_instance` 配置 | 表单元数据库没配置或 key 不一致 |
| 流程图无法查看 | `definition-xml` 接口、流程实例 ID | 前端没传正确 `processInstanceId` |
| 发起成功但业务表没回写实例 ID | 发起接口事务顺序 | 先调工作流、后写业务表导致中断 |

## 24. 后端组提测前最终检查

提测前建议负责人按下面顺序做一次人工核对：

1. 用发起人账号创建一条新业务
2. 确认业务表里写入 `bizId`、`title`、`processInstanceId`
3. 用下一处理人账号确认待办可见
4. 确认待办里有标题和业务单号
5. 打开任务上下文确认 `bizData` 和 `formRef`
6. 完成任务后确认已办可见
7. 用业务单号查询流程图和时间线
8. 重启服务后再次核对“我的申请”和进度查询

只有这 8 步都通过，才建议正式交前端和测试联调。
