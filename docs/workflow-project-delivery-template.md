# 工作流项目推进交付模板

## 1. 文档用途

这份文档给项目经理、项目负责人、架构负责人使用。

它解决的不是代码怎么写，而是一个业务系统接入 `go-common/workflow` 时，项目应该怎么拆阶段、怎么设出口、怎么管联调。

## 2. 适用场景

适用于下面几类项目：

- 新业务系统第一次接入工作流
- 老业务系统从自研审批切到 Flowable
- 一个业务系统新增第二条或第三条审批流

## 3. 项目推进总目标

项目完成后，至少要达到下面这些结果：

1. 业务系统能发起流程
2. 用户能在业务系统里看“我的申请”
3. 用户能看自己的待办和已办
4. 用户能在业务页面中查看流程图和时间线
5. 用户能在业务页面中处理任务
6. 测试组能完成完整流转验证

## 4. 五阶段推进模型

### 4.1 第一阶段：架构确认

本阶段要确认：

- 流程定义 key
- 业务类型编码
- `bizId` 规则
- 标题字段来源
- 业务表结构
- 业务状态枚举
- LDAP 用户标识口径

阶段输出：

- 架构确认纪要
- 业务表初稿
- 接口范围初稿

出口标准：

- 后端、前端、测试都明确边界
- 不再讨论“是否使用 Flowable 官方前端页面”

### 4.2 第二阶段：后端打通

本阶段要完成：

- 工作流模块注册
- 业务表落库
- 发起流程
- 我的申请
- 待办和已办
- 任务上下文
- 任务处理
- 进度接口联通

阶段输出：

- 后端联调地址
- 接口说明
- 流程变量映射表

出口标准：

- 至少打通一条完整流程到下一节点

### 4.3 第三阶段：前端接入

本阶段要完成：

- 业务创建页
- 我的申请页
- 待办页
- 已办页
- 任务处理页
- 流程图内嵌
- 时间线展示

阶段输出：

- 可操作页面
- 前端接口映射说明
- 页面截图

出口标准：

- 用户可在业务系统页面里完成一次完整操作

### 4.4 第四阶段：联调和修正

本阶段要完成：

- 角色切换测试
- 标题链路检查
- 当前人员显示检查
- 退回、驳回、分支条件检查
- 服务重启后检查

阶段输出：

- 联调问题清单
- 修复记录
- 回归验证记录

出口标准：

- 关键缺陷清零

### 4.5 第五阶段：测试验收和上线准备

本阶段要完成：

- 测试组执行模板化测试
- 输出缺陷清单
- 输出通过结论
- 上线前清单核对

阶段输出：

- 测试报告
- 上线检查表
- 最终发布结论

出口标准：

- 满足上线条件

## 5. 角色分工建议

| 角色 | 核心职责 |
| --- | --- |
| 架构师 | 确认边界、确认标准链路、处理方案分歧 |
| 项目经理 | 排期、推动、跟踪阶段出口 |
| 后端负责人 | 落库、发起、任务处理、接口交付 |
| 前端负责人 | 页面落地、流程图、时间线、体验统一 |
| 测试负责人 | 场景覆盖、缺陷闭环、验收结论 |
| 工作流模块维护人 | 提供模板、支持联调、处理通用问题 |

## 6. 项目经理每周检查项

项目推进期间，建议项目经理每周至少检查一次下面这些内容：

1. 流程定义是否已经稳定
2. 后端是否已经能发起流程
3. 标题是否已贯穿发起、申请、待办、已办
4. 前端是否已经能展示流程图和时间线
5. 当前人员显示是否已经统一
6. 测试账号是否已准备好
7. 联调问题是否有明确责任人

## 7. 标准会议节奏

### 7.1 架构确认会

重点结论：

- 业务边界
- 接口边界
- 数据边界
- 排期边界

### 7.2 后端联调会

重点结论：

- 发起是否成功
- `bizId` 是否贯通
- `processInstanceId` 是否回写
- 待办、已办是否能查

### 7.3 前端联调会

重点结论：

- 页面是否可用
- 标题是否稳定
- 流程图展示是否正确
- 时间线是否符合预期

### 7.4 测试评审会

重点结论：

- 缺陷是否影响上线
- 是否还存在误导用户的展示问题
- 是否满足上线前标准

## 8. 可直接复制的项目任务拆解模板

### 8.1 架构任务

- 确认流程定义 key
- 确认业务状态
- 确认变量映射
- 确认标题来源

### 8.2 后端任务

- 接入工作流模块
- 建业务表
- 实现发起接口
- 实现我的申请
- 实现待办已办
- 实现任务上下文
- 实现任务处理

### 8.3 前端任务

- 建创建页
- 建我的申请
- 建待办已办
- 建任务处理页
- 接流程图
- 接时间线

### 8.4 测试任务

- 准备账号
- 编写测试数据
- 执行完整流转
- 复测关键缺陷
- 输出验收记录

## 9. 风险预警清单

项目经理需要重点防下面这些风险：

- 业务组想直接使用 Flowable 官方页面代替业务页面
- 业务组没有建设自己的业务表
- 发起流程时没有传 `title`
- 待办和已办没有补业务标题
- 当前人员显示逻辑不统一
- 流程图接口接好了，但前端用了错误的展示方式
- 表单 key、表单标识、表单引用命名混乱

## 10. 上线前最终检查表

- [ ] 发起流程成功
- [ ] 我的申请正常
- [ ] 待办正常
- [ ] 已办正常
- [ ] 任务处理正常
- [ ] 流程图正常
- [ ] 时间线正常
- [ ] 标题全链路不丢失
- [ ] 当前人员显示正确
- [ ] 服务重启后数据仍然正常

## 11. 项目经理结项标准

只有当下面这些结论都成立时，项目经理才建议结项：

- 模板化文档已经交付到位
- 后端能力已经可复用
- 前端展示已经符合规范
- 测试报告已经通过
- 新业务系统能够独立运行这条审批流

达到这一步，才说明 `go-common/workflow` 在该业务系统中真正落地，而不是只完成了接口对接。
