# 工作流待办已办优化实施路线

## 1. 文档目的

这份文档是在《工作流待办已办性能优化设计》的基础上，再往前推进一层，把“应该怎么做”收敛成“应该按什么顺序做、每一步具体改哪里、哪些接口不能动、哪些字段可以延后优化”的实施路线。

你前面已经明确确认过，待办已办优化不能一上来就直接做投影表、缓存表或者读模型，否则很容易把当前已经验证通过的权限隔离和候选任务判断搞乱。所以这份路线文档遵循一个原则：先做低风险、高收益、对外接口无破坏的优化，再做结构性演进。

这不是发布计划，也不是代码设计明细，而是后续每一阶段真正开工前都要遵守的施工顺序。

## 2. 当前代码热点在哪里

当前待办和已办的主要热点集中在 `go-common/workflow/engine/flowable/rest_client.go`。

待办查询的热点路径是：

- `ListTodo`
- `matchesCandidateRuntimeTask`
- `matchesProcessScope`
- `enrichTaskPage`

已办查询的热点路径是：

- `ListDone`
- `matchesHistoricTaskUser`
- `matchesProcessScope`
- `enrichTaskPage`

真正重的地方主要有三类。

第一类，是待办拆成了多次 Flowable runtime task 查询：

- 按 `assignee` 查
- 按 `candidateUser` 查
- 按 `candidateGroupIn` 查

第二类，是候选任务还需要再查 identity links 做二次校验。

第三类，是列表结果出来以后，还要逐条通过 `GetProgressView` 做 enrich，把标题和流程摘要补齐回去。

也就是说，当前实现正确性的代价，本质上就是“多次远程调用 + 二次过滤 + 列表后补强”。后续优化不能回避这三个热点，只能按顺序把它们拆解掉。

## 3. 总体实施原则

这一轮优化必须同时遵守下面四条原则。

第一，不能破坏当前待办隔离和已办隔离的正确性。别人的待办不能看见，别人的已办不能误查出来，这一条优先级永远高于性能。

第二，不能把过滤责任重新甩回业务系统。也就是说，业务系统后面仍然应该只调用 `go-common/workflow` 的接口，而不应该被要求自己再加一层“是不是我的任务”的过滤逻辑。

第三，不能在没有收口接口契约的情况下直接改列表结构。当前已经完成了接口收口，所以后续优化要服从那份文档，不允许随意移除稳定基础字段。

第四，列表接口应该越来越轻，详情接口应该越来越重。这条方向必须坚定，否则平台永远会陷在“列表接口什么都查”的模式里。

## 4. 第一阶段：列表默认不做重型 enrich

这是第一优先级，也是当前最应该先落地的一步。

### 4.1 目标

把待办和已办列表从“重型聚合接口”收敛成“轻量概览接口”，优先保证列表首屏查询效率和稳定性。

### 4.2 当前问题

当前 `ListTodo` 和 `ListDone` 返回列表后，还会调用 `enrichTaskPage`，对每一条任务再调一次 `GetProgressView`，把标题和 `progress` 摘要补进去。

这会带来两个直接问题：

- 一页返回多少条，就可能追加多少次进度查询
- 列表和详情职责混在一起，后面很难继续优化

### 4.3 本阶段建议动作

第一，把列表接口默认模式收敛为基础字段返回，不再默认逐条做重型 `progress-view` enrich。

第二，把标题的来源尽量前移，优先从任务本身变量、业务 key、流程变量轻量补齐，而不是依赖完整进度接口。

第三，列表中的 `progress` 字段保留为增强信息，但默认不再承诺一定完整返回。

第四，把“完整流程图”“完整时间线”“完整步骤”“当前候选人全量信息”全部视为详情页查询职责。

### 4.4 受影响代码

主要改动点会落在：

- `ListTodo`
- `ListDone`
- `enrichTaskPage`
- 可能新增一个轻量摘要补齐函数，用于替代当前逐条 `GetProgressView`

### 4.5 对外接口影响

不改接口路径，不改稳定基础字段，不改外层响应结构。

唯一的语义变化是：列表中的 `progress` 从“尽量完整 enrich”收敛为“可选增强字段”。这件事已经在《工作流接口收口说明》中提前约定过，因此属于可控变化，不算破坏性改动。

### 4.6 为什么这一步先做

因为它收益最大、风险最小，而且不需要先引入新表、新缓存、新回调机制，就能先把最明显的重复回源成本降下来。

## 5. 第二阶段：增加分页、过滤、搜索

这是紧接着第一阶段要做的事情，目的是避免列表天然失控。

### 5.1 目标

把待办和已办从“查出来再说”改成“有边界地查询”，防止用户数量和任务数量一增大就把接口拖垮。

### 5.2 当前问题

当前虽然已经支持 `start`、`size`，但过滤能力仍然比较弱，业务系统容易直接用“大页数 + 大列表 + 前端自己筛选”的方式消费接口。

这在验证阶段没问题，但在稳定版本中不够成熟。

### 5.3 本阶段建议动作

第一，明确默认分页大小和最大分页大小，避免长期出现一次拉 200 条、500 条甚至更多的调用方式。

第二，为待办和已办增加基础过滤条件，建议优先支持：

- 标题关键字
- `bizType`
- 时间范围
- 流程状态
- 流程定义 key

第三，业务前端列表默认强制分页，不允许把“全量列表”当成常态使用方式。

### 5.4 受影响代码

主要会涉及：

- `types.TaskQuery`
- `parseTaskQuery`
- `ListTodo`
- `ListDone`
- Flowable query 参数组装逻辑

### 5.5 对外接口影响

这是向前兼容增强。原有 `start`、`size` 继续保留，只是增加新的查询参数，并且把分页约束从“建议”提升到“平台默认策略”。

### 5.6 为什么排第二

因为如果不先把列表限制住，后面即使加缓存，也可能只是把不合理的大查询换一种方式放大，不是根治。

## 6. 第三阶段：流程范围信息做缓存

这是第一轮真正引入缓存能力的阶段，但它应该在接口职责和查询边界已经收口之后再做。

### 6.1 目标

减少列表查询过程中反复按 `processInstanceId` 回源取流程摘要的成本，重点缓存“范围判断 + 基础摘要”这类高频信息。

### 6.2 当前问题

当前 `matchesProcessScope` 会在待办和已办查询过程中，按 `processInstanceId` 调用 `GetProgressView` 来判断 `tenantId`、`systemCode` 是否匹配。

这意味着哪怕只是做范围过滤，也要走一次比较重的进度查询。

### 6.3 本阶段建议缓存内容

建议缓存以下轻量范围摘要：

- `processInstanceId`
- `tenantId`
- `systemCode`
- `bizId`
- `bizType`
- `title`
- `status`
- `currentActivityNames`
- `lastSyncAt`

### 6.4 本阶段建议动作

第一，在 `go-common/workflow` 内部增加流程摘要缓存层，优先使用 Redis。

第二，`matchesProcessScope` 优先查缓存，缓存未命中时再回源，并把结果回填缓存。

第三，列表轻量补齐标题和状态时，也优先使用同一份缓存，而不是额外再查完整进度。

### 6.5 受影响代码

主要会涉及：

- `matchesProcessScope`
- `enrichTaskPage` 或其替代逻辑
- `GetProgressView` 调用点
- 新增缓存服务或摘要查询服务

### 6.6 对外接口影响

这一阶段理论上不改变任何对外接口，只改变内部查询路径。

### 6.7 为什么排第三

因为如果在列表仍然很重、分页仍然不受控的前提下先上缓存，容易把缓存也打成补锅工具。只有先把列表职责收紧，再加缓存，缓存才能真正稳定。

当前这一阶段已经落地后的实现口径可以固定为：

- 缓存对象是流程范围摘要 `ProcessProgressSummary`
- 缓存 key 以 `processInstanceId` 为主键
- 默认优先使用 `go-common` 已初始化的 Redis 实例
- 默认 TTL 为 `120` 秒
- 如果 Redis 不可用，则自动退回现有回源逻辑，不影响功能正确性
- `GetProgressView` 和 `GetProgressTimeline` 在成功返回后会回填缓存
- `matchesProcessScope` 会优先查缓存
- 任务完成后会主动清理对应流程实例缓存，避免短时间内读到旧摘要

## 7. 第四阶段：候选任务判定索引化

这是第一步真正涉及结构升级的优化，也是待办性能里最关键的一步。

### 7.1 目标

把候选用户、候选组任务的可见性判断，从“每次查询时临时推导”逐步改成“写时或同步时展开索引”。

### 7.2 当前问题

当前待办查询最重的部分不是本人已签收任务，而是候选任务。因为候选任务至少要做：

- Flowable runtime task 查询
- identity links 查询
- 当前用户所属组比对
- 范围信息校验

只要某个流程普遍采用候选人模式，这部分成本就会持续存在。

### 7.3 本阶段建议动作

第一，设计并引入一张任务参与人索引表，或者等价的 Redis 索引结构。

第二，索引内容至少覆盖：

- `taskId`
- `processInstanceId`
- `bizId`
- `bizType`
- `title`
- `activityId`
- `activityName`
- `assignee`
- `candidateUser`
- `candidateGroup`
- `tenantId`
- `systemCode`
- `createdAt`
- `updatedAt`

第三，待办查询优先查索引，再在必要时回源 Flowable 做兜底校验。

### 7.4 触发机制建议

这个阶段有两种可选路线：

- 依赖 Flowable 回调事件增量维护
- 依赖定时同步任务补偿维护

更成熟的做法是两者并存：回调做实时更新，定时同步做兜底修复。

### 7.5 对外接口影响

这一阶段也不应该改待办接口路径和基础字段。它的价值是把正确性保留下来，同时显著减少待办回源查询量。

### 7.6 为什么排第四

因为这是重构量比较大的一步，需要先确保前面三步已经把列表职责、分页边界和缓存层打稳，否则一上来做索引表，很容易把问题复杂度成倍放大。

当前这一阶段已经落地后的第一版实现口径是：

- 先不直接上完整投影表
- 优先把任务的候选用户、候选组 `identity links` 做成 Redis 级别的任务索引缓存
- `matchesCandidateRuntimeTask` 优先查索引，索引未命中时再回源 Flowable
- `getTaskIdentityLinks` 在成功读取后会自动回填索引
- 任务完成后会主动清理对应任务索引，避免短时间读取旧候选关系

这意味着第四阶段当前完成的是“候选任务判定索引化第一版”，目标是先降低逐条 identity links 远程查询的频率，而不是一次性完成完整的候选任务投影体系。

## 8. 第五阶段：已办查询改成历史投影表

待办优化之后，已办也要走向本地化读模型。

### 8.1 目标

把“我的已办”从完全依赖 Flowable history API 的方式，逐步迁移到以本地历史投影为主、Flowable 为真值源和补偿源的结构。

### 8.2 当前问题

当前已办虽然正确性已经修复，但它仍然依赖：

- Flowable 历史任务接口
- 本地二次归属判断
- 范围校验
- 列表后 enrich

当历史任务数据量变大后，这条链路会变得越来越重。

### 8.3 本阶段建议动作

第一，设计并引入已办历史投影表。

第二，投影字段至少包括：

- `taskId`
- `processInstanceId`
- `bizId`
- `bizType`
- `title`
- `activityId`
- `activityName`
- `assignee`
- `owner`
- `completedAt`
- `tenantId`
- `systemCode`

第三，已办查询优先查历史投影表，Flowable history API 只用于补偿同步与抽样校验。

### 8.4 触发机制建议

建议来源包括：

- 任务完成成功后主动写入
- Flowable 回调事件消费写入
- 定时补偿同步

### 8.5 对外接口影响

不改对外接口，只替换内部数据来源。

### 8.6 为什么排第五

因为已办的业务风险比待办略低，待办是“当前能不能办”的实时入口，优先级更高。待办查询链路稳定以后，再把已办迁到投影表，是更稳妥的顺序。

当前这一阶段已经落地后的第一版实现口径是：

- 先不引入关系型历史投影表
- 优先把通过 `CompleteTask` 完成的任务写入 Redis 已办投影
- 投影内容至少覆盖任务标识、流程标识、业务标识、标题、办理人、完成时间、租户和系统范围字段
- `ListDone` 在第一页场景下优先读取 Redis 已办投影
- 如果投影不足，自动回源 Flowable 历史任务接口补齐
- 因此当前仍然是“投影优先 + Flowable 兜底”的混合模式

这意味着第五阶段当前完成的是“已办历史投影第一版”，目的是先让最近完成的已办查询逐步摆脱完全依赖 Flowable history API，而不是一次性完成最终版的历史读模型。

## 9. 第六阶段：进度摘要和列表彻底解耦

这一步你前面已经特别强调过，因此这里单独列成一个阶段，而不把它简单并进第一阶段。

### 9.1 目标

彻底固定一条平台原则：列表只展示“足够做列表判断的摘要”，完整流程图和完整时间线一律走详情接口。

### 9.2 当前问题

虽然在接口收口里已经把 `progress` 定义成增强字段，但只要系统里还存在“列表页默认就去拿完整进度”的心智，后面性能优化仍然会反复被拉回去。

### 9.3 本阶段建议动作

第一，把前端接入模板、业务接入模板和标准接口文档全部统一到这个原则上。

第二，把列表接口里的流程字段固定收敛为：

- 当前节点名
- 当前状态
- 更新时间

第三，完整时间线、流程图、候选人明细、未来路径推导，全部要求在详情页单独查。

### 9.4 对外接口影响

不删详情接口，不删按 `bizId` 查进度的能力，只是明确它们不属于列表首屏的默认负载。

### 9.5 为什么把它放在第六阶段单列

因为它不只是代码优化，而是平台接入规范和业务组使用习惯的统一。这个动作需要等前面几步都落地后，再正式固化成平台共识。

## 10. 推荐实施顺序总结

基于当前代码形态、你已经确认的优化方向，以及对正确性风险的判断，推荐的落地顺序就是下面这六步，顺序不要打乱。

第一步，列表默认不做重型 enrich。

第二步，增加分页、过滤、搜索。

第三步，流程范围信息做缓存。

第四步，候选任务判定改成索引化。

第五步，已办查询改成历史投影表。

第六步，进度摘要和列表彻底解耦，并同步收敛到所有接入模板中。

这个顺序的核心逻辑是：先减重，再限流，再缓存，再建索引，再做投影，最后把规范彻底固化。

## 11. 当前阶段建议做到哪里

如果从当前版本继续往前推进，我建议下一次真正进入代码开发时，只做前两步，不要一次跨到缓存、索引表和历史投影表。

也就是说，下一轮最合理的开发范围应该是：

- 列表默认不做重型 enrich
- 增加分页上限和基础过滤能力

这两步完成以后，再单独进入缓存阶段。这样既能尽快拿到可见收益，又不会把当前已经跑通的业务链路风险一下子抬太高。

## 12. 与版本节奏的关系

这份实施路线文档的结论，也直接决定版本节奏。

当前这版可以继续作为“接入验证版”的基础版本，因为正确性已经基本稳定。

如果后续完成了第一步和第二步，那么就可以认为平台从“功能验证可用”进入“基础性能可用”阶段。

如果后续再完成第三步到第五步，那么平台才开始真正具备中大规模场景下的稳定性基础。

等第六步连同所有接入模板一起固化以后，这套工作流平台才算真正完成了从“能接”到“好接、稳接、可扩展接”的收口。
