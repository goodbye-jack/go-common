# 工作流标准接入检查清单

## 1. 基础环境

- 已经把 `go-common/workflow` 路由注册到业务服务
- 已经配置 Flowable REST 连接信息
- 已经配置 LDAP 连接信息
- 请求中能够解析出 `userId`
- 请求中能够解析出 `tenantId`
- 如果启用表单引用解析，已经配置 `workflow.formref.db_instance`

## 2. 业务表

- 业务表中有稳定的 `biz_id`
- 业务表中有 `title`
- 业务表中有 `status`
- 业务表中有 `process_instance_id`
- 业务表支持按当前发起人查询记录

## 3. 发起流程

- 业务创建接口先写业务表，再发起流程
- 发起请求中带有 `bizId`
- 发起请求中带有 `bizType`
- 发起请求中带有 `title`
- 发起请求中带有 `payloadRef`
- 发起成功后把 `processInstanceId` 回写业务表

## 4. 我的申请

- “我的申请”来自业务表，不来自工作流待办或已办
- 每条业务记录都可以按 `bizId` 补进度
- 每条业务记录都能显示当前节点
- 每条业务记录都能显示当前办理人或候选范围

## 5. 我的待办

- `GET /api/me/tasks/todo` 可正常使用
- 待办列表能显示业务标题
- 待办列表能显示流程进度摘要
- 待办列表能进入任务处理页

## 6. 我的已办

- `GET /api/me/tasks/done` 可正常使用
- 已办列表能显示业务标题
- 已办列表能显示完成时间
- 已办列表能查看流程进度

## 7. 任务上下文

- `GET /api/tasks/{taskId}/context` 能返回任务信息
- 能返回 `bizId`
- 能返回 `payloadRef`
- 能返回业务标题
- 节点配置表单时，能返回表单引用信息

## 8. 流程进度

- `GET /api/process-instances/{id}/progress-view` 可正常使用
- `GET /api/process-instances/{id}/progress-timeline` 可正常使用
- `GET /api/biz/{bizId}/progress-view` 可正常使用
- `GET /api/biz/{bizId}/progress-timeline` 可正常使用
- `GET /api/process/instance/{id}/definition-xml` 可正常使用
- 进度摘要能返回当前办理人或候选范围
- 时间线能返回真实流转过程

## 9. 前端展示

- BPMN 流程图是页面内嵌展示
- BPMN 流程图没有内部滚动条
- 当前节点高亮清晰
- 人员显示优先展示实际办理人
- 如果没有办理人，则展示候选人和候选组，并标识为“待认领”
- 只有系统节点显示“系统”

## 10. 处理任务

- `POST /api/tasks/{taskId}/complete` 可正常使用
- 业务页面字段能够正确映射到流程变量
- 处理任务后业务表状态能按规则更新
- 流程能够正确流转到下一个节点

## 11. 最终验收

- 验证一条完整的新流程，从发起到进入待办
- 验证标题在“我的申请”和“我的待办”中都能显示
- 验证一次任务处理
- 验证一条已办记录
- 验证流程图和流程时间线
- 验证服务重启后数据仍然正确

如果以上检查项全部通过，就说明这个业务系统已经具备真实接入工作流的能力。

## 12. 业务组交付物检查

这部分建议项目负责人在提测前额外核对一次：

- [ ] 已提交业务表结构说明
- [ ] 已提交业务创建接口说明
- [ ] 已提交“我的申请”接口说明
- [ ] 已提交任务处理字段说明
- [ ] 已提交流程变量映射说明
- [ ] 已提交前端页面清单

## 13. 联调记录检查

建议每个业务组在联调时保留下面这些记录：

- [ ] 一条发起成功的请求报文
- [ ] 一条待办查询成功的返回报文
- [ ] 一条任务上下文返回报文
- [ ] 一条任务完成请求报文
- [ ] 一条流程进度返回报文
- [ ] 一条流程时间线返回报文

这样后面出问题时，能快速定位是业务问题还是工作流问题。

## 14. 上线前最终确认

上线前再核对一次下面这些结论：

- [ ] 业务标题在发起、申请、待办、已办中都不丢失
- [ ] 当前节点显示正确
- [ ] 当前办理人或候选范围显示正确
- [ ] 流程图可正常内嵌展示
- [ ] 时间线顺序和实际处理顺序一致
- [ ] 服务重启后业务列表和流程进度仍然正常
