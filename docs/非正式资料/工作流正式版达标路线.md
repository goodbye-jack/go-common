# 工作流正式版达标路线

## 1. 文档目的

这份文档用于回答一个非常具体的问题：`go-common/workflow` 从当前“接入验证版”走到“正式稳定版”还差哪些事情，必须按什么顺序推进，以及什么时候才允许对外宣称达到正式版本标准。

它不是泛泛而谈的规划文档，而是后续每一个大阶段都要对照执行的正式路线。

## 2. 当前版本处于什么位置

基于已经完成的迁移、两类真实业务流程联调、六个性能优化阶段落地以及接口收口情况，当前版本已经具备以下能力：

- 标准工作流主链路已经跑通
- LDAP 目录能力已经接入
- 业务系统和工作流平台的职责边界已经清晰
- 待办和已办的人员隔离问题已经修正
- 表单引用解析已经稳定
- 流程图、时间线、进度摘要已经可以被业务系统直接消费
- 业务接入模板、测试模板和发布准备资料已经具备基础体系

因此，当前版本已经可以定义为“工作流平台接入验证版”。

但它还不能直接定义为“正式稳定版”。原因不是功能不可用，而是正式版除了可用，还必须满足规模稳定性、复杂流程稳定性、交付稳定性和版本治理稳定性。

## 3. 正式稳定版的判定标准

如果要把 `go-common/workflow` 正式定义为稳定版本，至少必须同时满足下面五类条件。

第一类，是功能闭环稳定。也就是说，目录能力、流程发起、待办、已办、任务上下文、表单引用、进度视图、进度时间线、按业务单号查进度、业务前端内嵌流程图，这些核心能力都不能再依赖临时绕过方案。

第二类，是权限与范围隔离稳定。任何一个用户都不能看到与自己无关的待办和已办，候选任务、租户范围、系统范围的判断都必须稳定落在平台内部，而不是靠业务系统再做补救性过滤。

第三类，是性能链路进入稳定形态。当前虽然已经有缓存、候选索引和已办投影第一版，但正式版要求至少把“本地读模型优先、远端接口兜底”的路线再往前推进一层，避免在业务量上来后重新暴露出大量远端回源问题。

第四类，是复杂流程能力边界清楚且经过验证。正式版不能只支持单一路径审批流，还必须至少明确和验证常见网关、退回重提、循环重审等场景的行为边界。

第五类，是交付包和版本治理稳定。也就是说，对外文档、接入模板、发布说明、回归记录、版本变更说明都要能够独立支撑一个新业务组接入，而不是仍然依赖人工口头解释。

## 4. 当前距离正式版还差什么

如果把当前状态和正式版标准对照起来，剩余差距主要集中在四个方面。

第一，回调驱动的读模型还没有形成稳定版本。现在已经有缓存、候选索引、已办投影第一版，但还没有把待办、已办、进度摘要统一沉淀成更成熟的本地读模型体系。

第二，复杂流程结构支持还需要继续往深处收口。常见简单审批流已经没有问题，`subProcess` 和直接子流程 `callActivity` 也已经进入可用状态，但多实例、会签、深层嵌套 `callActivity`、复杂 EL 的覆盖仍然不足。

第三，正式版回归门槛还没有完全转成固定动作。现在虽然已有联调记录和发布清单，但还缺一份明确的“正式版达标门槛说明”，让后续任何一次发版都能按同一标准判定。

第四，对外交付文档的中文命名和正式交付入口还没有完全统一。虽然正文基本已经中文化，但正式版应该优先对外暴露中文命名文档，避免业务组继续混用历史英文文件名。

## 5. 推荐推进顺序

为了避免把风险一次性抬高，正式版不应该一口气跳过去，而应该按下面顺序推进。

### 5.1 第一阶段：正式版差距收口

这个阶段的目标，是把正式版达标标准、发布门槛、中文交付入口全部固定下来。

本阶段重点动作包括：

- 固化正式版达标路线
- 固化联调回归记录和版本变更说明
- 把对外交付文档优先收口到中文命名
- 明确哪些文档属于历史记录，哪些文档属于正式交付包

这个阶段完成后，版本边界会清楚，但还不能直接宣称达到正式版。

### 5.2 第二阶段：回调驱动读模型增强

这是正式版前最关键的技术阶段。

目标不是推翻当前实现，而是在现有缓存、候选索引、已办投影第一版的基础上，再往前走一步，把待办、已办和流程摘要进一步沉淀成稳定读模型。

这个阶段至少要覆盖：

- 运行中任务摘要投影
- 用户待办索引
- 用户已办索引
- 流程实例摘要投影
- 回调增量更新
- 定时补偿修复

这个阶段完成后，正式版在中大规模场景下才有基本稳定性基础。

### 5.3 第三阶段：复杂流程能力增强

这个阶段的目标，是把当前“适合标准审批流”的能力边界再往前扩。

建议至少补齐或明确验证：

- 复杂网关表达式
- 回退与重新进入同一节点的时间线语义
- 子流程内部节点展开
- 会签
- 多实例
- `callActivity` 深层嵌套与更复杂展开策略

这里的要求不是一次性做到全 BPMN 语义完美覆盖，而是把正式版承诺支持的范围说清楚，并确保这些范围已经过真实验证。

### 5.4 第四阶段：正式版回归与发布准备

这个阶段的目标，是把前面三阶段结果转成正式版发布动作。

至少要完成：

- 固定正式版回归清单
- 固定正式版发布说明模板
- 固定版本标签策略
- 至少一个新增业务组按正式交付包完整接入
- 完成一次完整回归并留档

完成这一步以后，才具备讨论正式稳定版标签的资格。

## 6. 正式版发布门槛

如果后续要真正打正式稳定版标签，我建议必须同时满足下面这些结论。

第一，两个现有业务流程和至少一个新增业务流程，都已经按标准模板完整接入并回归通过。

第二，待办、已办、进度摘要都已经不再主要依赖查询时重型回源，而是有稳定的本地读模型优先链路。

第三，复杂流程支持范围已经在文档中明确，并且其中被承诺支持的部分已经有真实验证记录。

第四，所有对外交付资料已经形成统一中文入口，业务组不需要靠聊天记录补理解。

第五，发布前检查清单中的所有关键项都已经满足，并且本轮没有残留“必须手工修数据才能工作”的问题。

只有这五条同时成立，才建议进入正式稳定版。

## 7. 当前建议的下一步

基于现在的状态，最合理的下一步不是直接打正式版标签，而是进入“回调驱动读模型增强”这一大阶段。

原因很简单：

- 当前功能链路已经够用
- 当前接口边界已经够清楚
- 当前文档体系已经初步可交付
- 离正式版最大的差距已经不是页面和字段，而是性能链路的最终稳定形态

因此，后续如果继续推进到正式版，下一阶段就应该把重点放在读模型增强，而不是再去分散做零碎接口补丁。
