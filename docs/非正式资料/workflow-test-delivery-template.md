# 工作流测试验收模板

## 1. 文档用途

这份文档是给测试组直接执行的验收模板。

测试组不需要理解所有实现细节，但必须用这份模板保证：

- 业务链路通
- 工作流链路通
- 数据不丢
- 标题不丢
- 人员展示不误导

## 2. 测试目标

至少验证以下内容：

1. 发起流程正常
2. 标题链路正常
3. 我的申请正常
4. 我的待办正常
5. 我的已办正常
6. 任务处理正常
7. 流程图正常
8. 时间线正常
9. 当前办理人或候选范围正常
10. 服务重启后数据仍正常

## 3. 测试前准备

测试前必须拿到下面这些信息：

- 测试账号清单
- 流程定义 key
- 测试业务类型
- 发起入口地址
- 待办页面地址
- 已办页面地址
- 任务处理页地址

## 4. 标准测试场景

### 4.1 场景一：发起流程

步骤：

1. 登录发起人账号
2. 新建一条业务记录
3. 发起流程

验证点：

- 返回成功
- 生成 `bizId`
- 生成 `processInstanceId`
- 我的申请中能看到这条记录
- 标题显示正确

### 4.2 场景二：下一处理人待办

步骤：

1. 登录下一处理人账号
2. 打开待办列表

验证点：

- 能看到新待办
- 待办标题正确
- 当前节点正确
- 当前人员显示正确
- 如果未签收，能看到候选范围

### 4.3 场景三：任务详情

步骤：

1. 打开任务详情页

验证点：

- 任务信息完整
- 业务数据完整
- 标题正确
- 流程图可查看
- 时间线可查看
- 节点表单参考可查看

### 4.4 场景四：处理任务

步骤：

1. 提交一次处理结果

验证点：

- 提交成功
- 当前待办消失
- 下一环节待办出现
- 已办中出现处理记录

### 4.5 场景五：已办查看

步骤：

1. 登录已处理该任务的账号
2. 打开已办页面

验证点：

- 已办记录存在
- 标题正确
- 完成时间正确
- 能查看流程图
- 能查看时间线

### 4.6 场景六：重启后验证

步骤：

1. 重启业务服务
2. 再次打开我的申请、待办、已办

验证点：

- 业务记录仍存在
- 标题仍存在
- 流程图仍可查看
- 时间线仍可查看

## 5. 重点缺陷检查项

测试组要重点关注下面这些高频问题：

- 标题丢失
- 待办显示节点名但没有业务标题
- 当前处理中显示成“系统”但实际是人工节点
- 时间线缺少当前节点
- 流程图无法完整展示
- 图内出现滚动条
- 服务重启后我的申请消失

## 6. 人员展示专项检查

这是测试组必须单独核对的一项：

### 6.1 有真实办理人时

应显示：

- `办理人：张三`

### 6.2 无真实办理人但存在候选人时

应显示：

- `待认领 / 候选人：张三`

### 6.3 无真实办理人但存在候选组时

应显示：

- `待认领 / 候选组：部门审批组`

### 6.4 系统节点时

才允许显示：

- `系统`

## 7. 流程图专项检查

测试组要验证：

- 流程图是否内嵌展示
- 是否存在图内滚动条
- 当前节点是否高亮
- 已完成节点是否高亮
- 页面缩放后流程图是否仍完整

## 8. 时间线专项检查

测试组要验证：

- 时间线顺序是否符合真实处理顺序
- 是否包含“流程发起”
- 是否包含真实已办节点
- 是否包含当前处理中的节点
- 正向未来节点是否展示合理
- 无关反向分支是否未错误展示

## 9. 验收输出要求

测试完成后，测试组至少需要输出：

- 一份测试记录
- 一份缺陷清单
- 一组关键页面截图
- 一组成功流转证明

## 10. 最终通过标准

只有当下面这些结论都成立时，测试才算通过：

- 新流程可正常发起
- 标题链路全程不丢失
- 待办正常
- 已办正常
- 当前人员展示正确
- 流程图展示正确
- 时间线展示正确
- 服务重启后数据仍然正确

满足以上条件，才允许进入正式业务接入阶段。

## 11. 测试执行记录模板

测试组建议为每个业务流程至少保留一份执行记录，推荐字段如下：

| 项目 | 内容 |
| --- | --- |
| 测试日期 | 2026-03-11 |
| 测试环境 | 开发 / 测试 / 预发 |
| 流程定义 key | 例如 `artifact_appraise` |
| 发起账号 | 例如 `gaowen` |
| 处理账号 | 例如 `test` |
| 测试业务单号 | 例如 `APPRAISE-20260311-0001` |
| 结果 | 通过 / 不通过 |
| 备注 | 缺陷编号或说明 |

## 12. 缺陷提单模板

测试组提交缺陷时，建议统一按下面格式写，避免开发侧无法快速复现。

### 12.1 标题模板

`[工作流][待办] 标题未显示 - bizId=APPRAISE-20260311-0001`

### 12.2 缺陷正文模板

- 所属页面：待办列表
- 测试账号：`test`
- 测试业务单号：`APPRAISE-20260311-0001`
- 测试任务 ID：`xxx`
- 实际结果：待办中无业务标题
- 期望结果：待办中应显示发起时填写的业务标题
- 附件：截图、接口返回、控制台日志

## 13. 回归测试重点

每次修完问题后，测试组至少回归下面这些点：

1. 原问题是否修复
2. 发起链路是否仍正常
3. 我的申请是否仍正常
4. 待办和已办是否仍正常
5. 流程图和时间线是否仍正常
6. 当前人员显示是否被误伤

## 14. 测试组与开发组的问题归属建议

为了减少扯皮，建议按下面原则快速归类：

### 14.1 更偏后端问题

- 标题接口没返回
- `bizId` 不存在
- `processInstanceId` 丢失
- 时间线顺序错误
- 当前办理人或候选范围返回错误

### 14.2 更偏前端问题

- 接口已返回标题但页面没显示
- 接口已返回人员信息但页面文案错误
- 流程图容器样式导致显示异常
- 时间线状态颜色或文案错误

### 14.3 需要共同联调的问题

- 流程节点名称和业务状态对应关系不一致
- 表单引用字段和页面字段理解不一致
- 复杂分支路径展示与预期不一致

## 15. 测试组提测结论模板

测试完成后，建议统一用下面格式给出结论：

### 15.1 通过结论模板

本次工作流接入测试已完成，以下链路验证通过：

1. 业务创建并发起流程
2. 我的申请展示
3. 我的待办展示
4. 任务处理提交
5. 我的已办展示
6. 流程图查看
7. 时间线查看
8. 服务重启后数据验证

当前未发现阻塞上线问题，建议进入下一阶段。

### 15.2 不通过结论模板

本次工作流接入测试未通过，存在以下阻塞问题：

1. 待办标题丢失
2. 当前人员显示错误
3. 时间线顺序异常

建议开发修复后重新提交测试。
