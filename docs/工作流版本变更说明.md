# 工作流版本变更说明

## 1. 文档目的

本文用于说明当前这一轮 `go-common/workflow` 相比前一阶段验证版，新增了哪些正式能力、收口了哪些接口、修复了哪些关键问题，以及为什么当前建议发布正式版本 `v1.3.0`。

## 2. 版本定位

建议版本号：

`v1.3.0`

推荐定位：

- `go-common/workflow` 首个正式稳定版

版本判断依据：

- 核心流程链路已完整
- 待办已办隔离已稳定
- 表单引用解析已稳定
- 进度视图和时间线接口已稳定
- 单流程与父子流程图已统一收口到 `diagram-view`
- 两个真实业务流程已完成联调

## 3. 本轮新增的正式能力

### 3.1 正式标准接口成型

当前正式标准接口已经收敛为：

- 用户与组织接口
- 流程发起接口
- 待办与已办接口
- 任务上下文接口
- 表单引用接口
- 任务完成接口
- 进度接口
- 统一流程图接口

### 3.2 流程图统一入口

本轮最大的正式能力收口是：

- 新增 `GET /api/process-instances/{id}/diagram-view`

这条接口自动判断：

- 单流程返回 `mode=single`
- 父子流程返回 `mode=composite`

从这一版开始，`diagram-view` 成为正式标准流程图入口。

### 3.3 流程发起分层明确

本轮正式明确了两层发起语义：

- 工作流标准接口：`POST /api/process/start`
- 业务系统推荐封装接口：如 `POST /api/applications`

这使得工作流平台与业务系统之间的职责边界更清晰。

### 3.4 父子流程图视图可用

复杂流程当前不再依赖“伪合成 XML”方案，而是通过统一视图模型提供：

- 主流程 BPMN XML
- 每个 `callActivity` 对应的子流程 BPMN XML

前端可在同一视图中展示父子流程，单流程则继续走原始 BPMN XML。

## 4. 本轮完成的关键修复

### 4.1 待办与已办隔离问题修复

当前已修复：

- 待办误查到他人任务
- 已办误查到他人历史
- 候选用户场景判定不稳定

### 4.2 表单引用解析稳定化

当前已修复：

- 表单引用解析误用业务默认库
- 表单元数据依赖与业务库耦合过深

当前支持：

- 独立 Flowable 元数据库配置
- 表单引用解析失败不阻断流程流转

### 4.3 标题、上下文、业务绑定补齐

当前已修复：

- 待办列表标题缺失
- 已办信息业务标识不清晰
- 按 `bizId` 查进度时绑定关系不稳定

### 4.4 流程图接口语义收口

当前已修复：

- 单流程和复合流程接口分裂
- 前端需要自己判断图类型
- 复杂流程图接口入口不统一

## 5. 本轮接口收口结果

### 5.1 正式主入口

当前正式主入口包括：

- `POST /api/process/start`
- `GET /api/me/tasks/todo`
- `GET /api/me/tasks/done`
- `GET /api/tasks/{id}/context`
- `GET /api/tasks/{id}/form-ref`
- `POST /api/tasks/{id}/complete`
- `GET /api/process-instances/{id}/progress-view`
- `GET /api/process-instances/{id}/progress-timeline`
- `GET /api/biz/{bizId}/progress-view`
- `GET /api/biz/{bizId}/progress-timeline`
- `GET /api/process-instances/{id}/diagram-view`

### 5.2 兼容接口

当前保留兼容但降级为非主入口：

- `GET /api/process/instance/{id}/definition-xml`
- `GET /api/process-instances/{id}/composite-diagram`

## 6. 对业务系统接入的意义

从业务系统角度，本轮版本已经可以稳定支撑以下接入模式：

- 发起业务单
- 我的申请
- 我的待办
- 我的已办
- 任务处理
- 流程进度查看
- 流程图查看
- LDAP 组织与人员信息联动

业务系统不再需要：

- 自己过滤待办归属
- 自己过滤已办归属
- 自己解析节点表单模型
- 自己区分单流程图和父子流程图接口

## 7. 当前已知边界

虽然本轮建议发布正式版，但仍需明确边界：

- 会签、多实例、深层嵌套 `callActivity` 仍属于后续增强项
- 更复杂 EL 函数与动态表达式仍属于后续增强项
- 大规模数据场景下仍建议继续推进索引化和投影表优化

这些边界不影响当前版本成为正式稳定版，但应在发布说明中明确告知。

## 8. 发布判断

综合当前能力、联调结果和接口收口状态，建议：

- 以 `v1.3.0` 发布正式版本
- 作为当前工作流平台的首个正式稳定版

## 9. 与上一阶段相比的核心变化摘要

如果需要一句话概括本轮变更，可以写成：

“本次版本完成了工作流标准接口正式收口，统一了单流程与父子流程图入口，稳定了待办已办隔离、任务上下文、表单引用解析与业务接入路径，达到正式发布条件。”
