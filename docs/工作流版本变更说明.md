# 工作流版本变更说明

## 1. 文档目的

本文用于说明 `go-common/workflow` 从 `v1.3.0` 升级到 `v1.3.1` 后，业务项目接入方式发生了哪些简化、配置方式有哪些增强、兼容边界如何控制，以及为什么当前建议将新版本发布为 `v1.3.1`。

## 2. 版本定位

建议版本号：

`v1.3.1`

推荐定位：

- `go-common/workflow` 正式稳定版后的易用性增强版

版本判断依据：

- 在 `v1.3.0` 正式能力基础上进一步降低业务项目接入成本
- 目录服务与自动分配 provider 已统一成配置驱动
- HTTP / LDAP 身份归一能力已形成稳定契约
- 启动期摘要日志与排障信息更完整
- `databases.*` 配置方式更适合多数据库、多实例业务项目

## 3. 本轮新增的正式能力

### 3.1 业务项目一行注册工作流模块

本轮对业务项目最有价值的变化是：

- 推荐统一使用 `workflowapi.MustRegisterFromConfig(server)`

这意味着业务项目不再需要自己手动拼装：

- 目录服务初始化
- 自动分配服务初始化
- Flowable REST client 初始化
- 表单引用服务初始化
- 标准工作流路由注册

### 3.2 目录服务来源标准化

当前目录服务来源统一收敛为：

- `workflow.directory.provider=none`
- `workflow.directory.provider=ldap`

并保留以下兼容能力：

- 如果未显式配置 provider，但检测到老的 `ldap_*` 或命名空间 `workflow.directory.ldap.*`，仍会自动按 LDAP 模式工作
- 旧业务项目不需要因为升级到 `v1.3.1` 就重写 LDAP 配置

### 3.3 自动分配 provider 正式可配置

本轮自动分配能力统一收敛为：

- `workflow.assignment.provider=none`
- `workflow.assignment.provider=directory`
- `workflow.assignment.provider=ldap`

其中最重要的实际意义是：

- 发起流程与处理任务时可以自动补齐下一环节分配变量
- 业务项目无需自己在每个接口里重复写“下一步找谁”的样板代码
- 支持把分配逻辑沉到通用目录服务或业务 HTTP 服务

### 3.4 LDAP 身份契约稳定化

本轮补强了：

- `workflow.identity.role_aliases`
- `workflow.identity.group_aliases`

这层能力的目标是：

- 让 LDAP 与本地登录态输出的角色 / 组编码统一归一
- 让 BPMN 固定候选组尽量只认一套稳定编码
- 降低 provider 切换时修改流程图的概率

### 3.5 启动期可观测性增强

当前工作流模块注册时会输出启动摘要，帮助开发者快速发现：

- 当前目录 provider 是什么
- 当前自动分配 provider 是什么
- 当前是否启用角色 / 组别名映射
- 标准路由是否已注册

## 4. 本轮对业务项目最直接的意义

### 4.1 新项目接入更简单

业务项目现在最推荐的接入姿势是：

1. 配置 `workflow.flowable.*`
2. 配置 `workflow.api.*`
3. 配置 `workflow.context.*`
4. 视项目情况配置 `workflow.directory.*`
5. 启动时执行 `workflowapi.MustRegisterFromConfig(server)`

到这里标准接口就已经可以用起来。

### 4.2 Provider 切换更清晰

如果业务项目未来从 LDAP 切到 HTTP，或从 HTTP 切回 LDAP，当前推荐方式是：

1. 保持 BPMN 中候选组编码不变
2. 通过 `workflow.identity.role_aliases` / `group_aliases` 做来源归一
3. 保持 `nextAssignee` / `nextCandidateUsers` / `nextCandidateGroups` 这些变量名稳定

### 4.3 数据源配置更统一

对数据库和 Redis 来说，业务项目现在更适合统一使用：

- `databases.mysql.*`
- `databases.postgres.*`
- `databases.kingbase.*`
- `databases.dm.*`
- `databases.mongo.*`
- `databases.redis.*`

尤其是 MySQL 现在可以直接通过结构化字段表达：

- `charset=utf8mb4`
- `parseTime=True`
- `loc=Local`
- 以及额外 `params`

## 5. 当前建议的升级动作

如果业务项目准备从 `v1.3.0` 升到 `v1.3.1`，建议按下面顺序执行：

1. 把工作流注册代码统一收敛到 `workflowapi.MustRegisterFromConfig(server)`
2. 明确 `workflow.directory.provider`
3. 如有自动分配需求，明确 `workflow.assignment.provider`
4. 如需 HTTP / LDAP 共用 BPMN，补齐 identity aliases
5. 业务数据库和 Redis 尽量改为 `databases.*` 配置

## 6. 兼容说明

当前版本仍保留以下兼容：

- 老的 `workflowapi.NewDefaultModuleFromConfig()` + `module.Register(server)` 用法继续可用
- 老的全局 LDAP 配置 `ldap_*` 继续可用
- 老的 RBAC Redis 配置 `redis_addr` / `redis_password` 继续可用

## 7. 发布判断

综合当前能力、接入易用性、配置标准化和兼容策略，建议：

- 将当前升级版本发布为 `v1.3.1`
- 作为 `v1.3.0` 之后的易用性增强版
- 作为后续新业务项目优先采用的推荐版本

## 8. 一句话总结

如果需要一句话概括本轮变更，可以写成：

“`v1.3.1` 在不破坏 `v1.3.0` 正式能力的前提下，进一步统一了业务项目接入方式、目录与分配 provider 配置、身份归一契约和数据源配置表达，使工作流在业务项目中的使用更简单、更明确、更容易排障。”
