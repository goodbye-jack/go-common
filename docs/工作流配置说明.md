# 工作流配置说明

## 1. 文档目标

本文用于说明业务系统接入 `go-common/workflow` 时，除了知道接口地址之外，还必须如何配置 `config.yaml`、如何注册模块、如何准备登录态上下文、如何让 Flowable 平台与业务系统对齐。

这份文档解决四类问题：

- `config.yaml` 里到底要写哪些配置
- 哪些配置是必填，哪些配置是增强项
- 除了 `config.yaml`，业务系统启动阶段还必须做什么
- Flowable 平台侧要额外配置哪些内容，业务系统才能真正跑起来

## 2. 最小可运行接入模型

一个业务系统接入工作流，最小需要满足下面五个条件：

1. 业务系统自身已经有登录态，并且能识别当前用户
2. `config.yaml` 已配置 Flowable REST、LDAP、工作流 API 基础参数
3. 服务启动时已经执行 `workflowapi.NewDefaultModuleFromConfig()` 并完成路由注册
4. 业务系统自己的“发起业务单”接口内部会调用 `POST /api/process/start`
5. Flowable 流程定义中的用户、组、表单 key、租户与业务系统配置保持一致

如果这五条有任意一条不成立，业务系统即使能启动，也只能算“代码接进去了”，不能算“工作流能力已接通”。

## 3. `config.yaml` 最小模板

下面这份模板适用于大多数业务系统，至少可以完成流程发起、待办、已办、任务上下文、进度与 LDAP 目录查询。

```yaml
service_name: "your-biz-service"
addr: ":9080"

databases:
  mysql:
    default:
      db_name: default_mysql
      mode: single
      host: 127.0.0.1
      port: 3306
      user: root
      password: 12345678
      database: your_biz_db
      max_open_conn: 100
      max_idle_conn: 10
      conn_max_life_time: 300s
      slow_time: 5
      log_mode: info
  redis:
    default:
      db_name: default_redis
      mode: single
      host: 127.0.0.1
      port: 6379
      password:
      database: 0
      timeout: 5
      max_pool_size: 100
      min_pool_size: 10

workflow:
  flowable:
    base_url: "http://localhost:8087/flowable-rest/service"
    username: "rest-admin"
    password: "test"
    timeout_seconds: 15
    group_prefix: ""
    role_prefix: "role_"
  api:
    prefix: "/api"
    sso_enabled: true
    roles: ""
    callback_path: "/flowable/callback"
    callback_secret: ""
  context:
    user_name_header: "X-Workflow-User-Name"
    system_code_header: "X-System-Code"
    groups_header: "X-Workflow-Groups"
    roles_header: "X-Workflow-Roles"

ldap_addr: "ldap://127.0.0.1:8389"
ldap_bind_dn: "cn=admin,dc=example,dc=com"
ldap_bind_password: "password"
ldap_base_dn: "dc=example,dc=com"
ldap_people_ou: "ou=people"
ldap_department_ou: "ou=departments"
ldap_position_ou: "ou=positions"
ldap_use_tls: false
```

这份最小模板里没有单独配置 `workflow.formref.db_instance`，因为不是所有业务系统都启用表单引用解析。

## 5. 配置项说明

### 5.1 `workflow.flowable.*`

- `workflow.flowable.base_url`
  - Flowable REST 服务地址
  - 必填
  - 例：`http://localhost:8087/flowable-rest/service`
- `workflow.flowable.username`
  - Flowable REST 用户名
  - 必填
- `workflow.flowable.password`
  - Flowable REST 密码
  - 必填
- `workflow.flowable.timeout_seconds`
  - 调用 Flowable REST 的超时时间，单位秒
  - 选填，默认按代码内部兜底
- `workflow.flowable.group_prefix`
  - 候选组前缀
  - 当 Flowable 模型里候选组写成 `role_xxx` 这类格式时需要配合使用
- `workflow.flowable.role_prefix`
  - 角色前缀
  - 当前已办/待办候选能力会用它识别角色型组名
- `workflow.flowable.todo_projection_ttl_seconds`
  - 待办投影缓存过期时间，单位秒
  - 启用回调增强时建议配置
- `workflow.flowable.todo_projection_task_prefix`
  - Redis 中单任务投影的 key 前缀
- `workflow.flowable.todo_projection_user_prefix`
  - Redis 中按用户索引待办的 key 前缀
- `workflow.flowable.todo_projection_group_prefix`
  - Redis 中按候选组索引待办的 key 前缀
- `workflow.flowable.process_runtime_task_prefix`
  - Redis 中按流程实例存放运行态任务索引的 key 前缀

### 5.2 `workflow.api.*`

- `workflow.api.prefix`
  - 工作流标准接口统一前缀
  - 默认建议使用 `/api`
- `workflow.api.sso_enabled`
  - 是否要求登录态
  - 业务环境建议为 `true`
  - 本地排障时可临时关闭
- `workflow.api.roles`
  - 路由允许访问的角色列表
  - 留空表示走默认角色集
- `workflow.api.callback_path`
  - Flowable 回调入口路径
  - 默认建议使用 `/flowable/callback`
- `workflow.api.callback_secret`
  - 回调签名密钥
  - 配置后，回调调用方必须附带 `X-Timestamp`、`X-Nonce`、`X-Signature`
  - 如果暂时不做签名校验，可以留空

### 5.3 `workflow.context.*`

这组配置不是“登录配置”，而是“工作流模块从请求头读取扩展上下文的头名配置”。

- `workflow.context.user_name_header`
  - 默认 `X-Workflow-User-Name`
- `workflow.context.system_code_header`
  - 默认 `X-System-Code`
- `workflow.context.groups_header`
  - 默认 `X-Workflow-Groups`
- `workflow.context.roles_header`
  - 默认 `X-Workflow-Roles`

如果业务系统已经有自己的网关或中间件在注入这些头，可以保持默认值不动；如果头名不同，就必须把这里一起改掉。

### 5.4 `workflow.formref.db_instance`

- 作用：指定表单引用解析时读取 Flowable 表单元数据所使用的数据库实例名
- 典型值：`flowable_meta`
- 必须和 `databases.mysql.<实例名>` 中的实例名保持一致
- 只有在业务系统要支持 `GET /api/tasks/{id}/form-ref` 的 Flowable 表单解析能力时才需要配置

如果不配这项：

- 流程仍然可以正常发起和流转
- 未启用表单解析的业务系统不会受影响
- 只是在节点配置了 Flowable 表单模型时，`resolved` 可能拿不到 `true`

### 5.5 LDAP 配置

- `ldap_addr`
  - LDAP 地址，例如 `ldap://113.45.4.22:8389`
- `ldap_bind_dn`
  - LDAP 管理员 DN
- `ldap_bind_password`
  - LDAP 管理员密码
- `ldap_base_dn`
  - LDAP 根 DN
- `ldap_people_ou`
  - 用户 OU
- `ldap_department_ou`
  - 部门 OU
- `ldap_position_ou`
  - 岗位 OU
- `ldap_use_tls`
  - 是否启用 TLS

这组配置会影响：

- `GET /api/me`
- `GET /api/me/manager`
- `GET /api/me/department`
- `GET /api/org/users/{userId}`
- `GET /api/org/users/{userId}/manager`
- `GET /api/org/users/{userId}/department`

## 6. 除了 `config.yaml` 之外还必须做的事

这部分是最容易被忽略的，也是最容易导致“配置看起来没问题，但业务系统还是跑不起来”的原因。

### 6.1 注册工作流模块

业务服务启动时必须执行：

```go
module, err := workflowapi.NewDefaultModuleFromConfig()
if err != nil {
	panic(err)
}
module.Register(server)
```

如果没做这一步：

- 所有 `/api/me/tasks/todo`、`/api/tasks/{id}/context`、`/api/process-instances/{id}/diagram-view` 之类的接口都不会注册出来
- `POST /flowable/callback` 也不会存在

### 6.2 准备登录态和用户上下文

工作流模块不会替业务系统做登录，所以业务系统必须自己保证：

- `commonhttp.GetUser(c)` 能拿到当前登录用户
- `commonhttp.GetTenant(c)` 能拿到当前租户
- 如果要区分系统编码、显示姓名、组、角色，需要把对应头传进来

当前默认读取关系如下：

- `userId` 来自业务系统登录态
- `tenantId` 来自业务系统租户上下文
- `userName` 来自 `X-Workflow-User-Name`
- `systemCode` 来自 `X-System-Code`
- `groups` 来自 `X-Workflow-Groups`
- `roles` 来自 `X-Workflow-Roles`

### 6.3 业务系统要封装自己的发起接口

虽然 `go-common/workflow` 已提供 `POST /api/process/start`，但正式业务接入时仍然建议业务系统保留自己的发起接口，例如：

- `POST /api/applications`
- `POST /api/alarm-applications`

业务系统自己的发起接口负责：

- 保存业务主表
- 保存业务 JSON 明细
- 保存附件
- 生成 `bizId`
- 调用 `POST /api/process/start`
- 回写 `processInstanceId`

### 6.4 如果启用回调，业务服务必须对外可访问

回调不是写在 `config.yaml` 就自动生效，还要满足下面条件：

- 业务服务对外可访问
- 发起流程时变量里传了 `callbackUrl`
- `callbackUrl` 指向当前业务服务可访问地址，例如 `http://your-host:9080/flowable/callback`
- 如果配置了 `workflow.api.callback_secret`，回调调用方要按签名规则传头

## 7. Flowable 平台侧必须配置的内容

业务系统接入工作流，不只是业务系统自己配 `config.yaml`，Flowable 平台侧也必须跟着对齐。

### 7.1 流程定义要能识别业务主键

发起流程时，至少要把下面这些变量稳定传进去：

- `bizId`
- `bizType`
- `title`
- `payloadRef`
- `tenantId`
- `systemCode`

推荐规则：

- `payloadRef` 与 `bizId` 保持一致
- `title` 始终使用业务主标题
- `bizType` 始终使用业务流程类型编码

### 7.2 人员分配规则要与 LDAP / 组织口径一致

Flowable 模型中的：

- `assignee`
- `candidateUsers`
- `candidateGroups`

必须和业务系统登录用户、LDAP 用户标识保持同一口径。

推荐口径：

- 用户统一用 LDAP `uid`
- 候选组统一用业务系统实际组编码
- 如果候选组需要带前缀，就和 `workflow.flowable.group_prefix`、`workflow.flowable.role_prefix` 一起设计

### 7.3 表单引用 key 要对齐

如果流程节点配置了 Flowable 表单模型：

- 节点上的 `formKey`
- Flowable 表单定义的 `key`
- 业务系统想看到的表单参考 key

最好保持一致。

否则会出现下面这种情况：

- 节点明明配置了表单
- `configured=true`
- 但由于 `formKey` 和表单定义 key 不一致，最终 `resolved=false`

### 7.4 租户规则要一致

`tenantId` 是否启用，不是随便填的。业务系统和 Flowable 必须选同一套规则。

推荐三种策略只选一种：

1. 全部使用统一租户，例如都用 `test-demo`
2. 全部使用空租户
3. 不同业务系统各自用自己的固定租户

不要出现下面这种混用：

- 发起时传 `tenantId=test-demo`
- 但流程定义部署在空租户
- 表单定义又部署在另一个租户

这种混用很容易导致：

- 某些流程能发起但节点表单解析不到
- 某些流程图能查到但任务查不到
- 子流程调用时提示找不到定义

### 7.5 `callActivity` 子流程要可解析

如果流程里用了 `callActivity`：

- 子流程定义必须已经部署
- 子流程 key 必须与 `callActivity` 配置一致
- 父子流程的租户策略必须一致
- 如果启用了 `sameDeployment=true`，父子流程应按同一部署策略管理

否则在处理父流程任务时，Flowable 很容易报“找不到子流程定义”。

## 8. 业务系统推荐的配置分层

为了后续方便迁移环境，建议业务系统把配置按下面三层管理：

### 8.1 固定写在配置文件中的

- `workflow.flowable.base_url`
- `workflow.api.prefix`
- `workflow.context.*`
- `workflow.formref.db_instance`
- LDAP OU 配置

### 8.2 适合环境变量覆盖的

- Flowable 用户名密码
- LDAP bind 密码
- MySQL 密码
- Redis 密码
- `workflow.api.callback_secret`

### 8.3 适合业务代码常量维护的

- `processDefinitionKey`
- `bizType`
- `systemCode`
- 业务状态枚举
- 页面字段到流程变量的映射表

## 9. 最小接入核对清单

一个业务系统准备联调前，负责人至少应核对下面这些项：

- `config.yaml` 已包含 `workflow.flowable.*`
- 如需 LDAP，LDAP 配置已填完整
- 如需表单解析，`workflow.formref.db_instance` 已配置
- 如需回调，`workflow.api.callback_path` 和 `callbackUrl` 已对齐
- 业务服务启动时已注册工作流模块
- 登录态中能解析出 `userId`
- 请求上下文中能解析出 `tenantId`
- 业务发起接口会先写业务表再发流程
- 发起时会传 `bizId`、`bizType`、`title`
- Flowable 模型里的用户、组、表单 key、租户已对齐

为了减少接入过程中的反复返工，建议业务组按下面顺序阅读文档：

1. 《工作流平台标准接口说明》
2. 《工作流配置说明》
3. 《工作流业务接入指南》
4. 《工作流后端接入交付模板》
5. 《工作流前端接入交付模板》
6. 《工作流测试验收模板》

这样先明确标准接口，再明确配置，再进入实际开发和提测阶段，效率最高。
