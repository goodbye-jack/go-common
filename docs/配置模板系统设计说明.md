# 配置模板系统设计说明

## 1. 目标

`go-common` 需要把“配置项定义”从代码字符串和分散文档中抽出来，形成可维护、可发布、可版本化的标准配置模板体系。

当前第一阶段目标：

- 沉淀配置元数据 `configspec`
- 生成 `v1.3.1` 的标准模板产物
- 形成业务项目首次接入、升级补缺、历史兼容的统一模板基础

## 2. 当前阶段边界

当前阶段只做：

- `configspec/` 配置元数据
- `templates/releases/v1.3.1/` 模板产物
- `templates/diff/` 版本差异
- 说明文档与基础校验测试

当前阶段不做：

- 不改 `config/config.go`
- 不改 `orm.InitAllDB`
- 不引入 `databases.*.*.enabled`
- 不把 `service_name / addr` 直接替换成 `service.name / host / port`
- 不自动修改业务项目真实 `config.yaml`

## 3. 当前阶段关键兼容策略

### 3.1 服务配置

由于运行时当前仍读取：

- `service_name`
- `addr`

所以第一阶段模板仍以这两个字段作为运行兼容主结构。

### 3.2 数据库配置

首次初始化真实配置时，默认生成：

- `databases.mysql.default`
- `databases.redis.default`

原因：

- 当前业务项目默认更接近 MySQL + Redis 组合
- `go-common` 中 Redis 使用频率高，迁移成本低

### 3.3 非 MySQL 关系型数据库

如果业务项目使用：

- PostgreSQL
- KingBase
- DM

则应以 `config.latest.yaml` 中对应块替换：

- `databases.mysql.default`

## 4. 模板产物说明

### `config.initial.yaml`

新业务项目首次接入时可作为真实 `config.yaml` 的初始化模板。

### `config.latest.yaml`

当前版本完整标准模板快照，用于展示当前 tag 支持的完整配置面。

### `config.full.yaml`

交付包总览模板。

### `config.workflow.yaml`

仅工作流模块模板。

### `config.databases.yaml`

仅数据库模块模板。

### `config.compatibility.yaml`

历史兼容配置模板，不建议新项目直接使用。

## 5. 后续阶段规划

第二阶段建议实现：

- 业务项目同步器
- `config.latest.yaml` / `config.missing.yaml` / meta 文件生成

第三阶段建议实现：

- `service.name / host / port` 兼容读取
- `databases.*.*.enabled`
- 更强的配置校验与启动摘要日志
