# 配置模板系统设计说明

## 1. 目标

`go-common` 需要把“配置项定义”从代码字符串和分散文档中抽出来，形成可维护、可发布、可版本化的标准配置模板体系。

当前第一阶段目标：

- 沉淀配置元数据 `configspec`
- 生成 `v1.3.3` 的标准模板产物
- 形成业务项目首次接入、升级补缺、历史兼容的统一模板基础

## 2. 当前阶段边界

当前阶段只做：

- `configspec/` 配置元数据
- `templates/releases/v1.3.3/` 模板产物
- `templates/diff/` 版本差异
- 说明文档与基础校验测试

当前阶段会同步落地：

- `config/config.go` 的基础读取规则升级
- `CONFIG_FILE` / `CONFIG_ENV` 覆盖规则
- `go-common-rules.vX.X.X.md` 自动生成
- 废弃 key 启动前检测

## 3. 当前阶段关键兼容策略

### 3.1 服务配置

`v1.3.3` 起运行时统一读取：

- `app.name`
- `server.addr`
- `security.cookie.*`
- `storage.local.*`

旧 key 仅作为迁移参考，不再作为运行时兜底读取结构。

### 3.2 数据库配置

首次初始化真实配置时，默认生成：

- `databases.mysql.default`
- `databases.redis.default`

原因：

- 当前业务项目默认更接近 MySQL + Redis 组合
- `go-common` 中 Redis 使用频率高，迁移成本低

### 3.3 非 MySQL 关系型数据库

如果业务项目使用：

- PostgreSQL
- KingBase
- DM

则应以 `config.latest.yaml` 中对应块替换：

- `databases.mysql.default`

## 4. 模板产物说明

### `config.initial.yaml`

新业务项目首次接入时可作为真实 `config.yaml` 的初始化模板。

### `config.latest.yaml`

当前版本完整标准模板快照，用于展示当前 tag 支持的完整配置面。

### `config.full.yaml`

交付包总览模板。

### `config.workflow.yaml`

仅工作流模块模板。

### `config.databases.yaml`

仅数据库模块模板。

### `config.compatibility.yaml`

历史兼容配置模板，仅用于旧项目迁移参考。

## 5. 后续阶段规划

后续建议继续补强：

- 更多业务侧通用访问器
- 更细粒度的废弃项提示
- 更强的启动摘要日志
