# 工作流 v1.3.1 业务项目最小接入手册

## 1. 这份手册解决什么问题

这份手册只回答一件事：

**一个业务项目，要怎样以最少代码、最少配置、最清晰的方式接入 `go-common/workflow`。**

如果你只是想：

- 先把 `me / todo / done / process/start / task/complete` 跑起来
- 先让业务服务把标准工作流接口注册起来
- 先保证以后能从 LDAP 切到 HTTP，或者反过来切回来

那就优先看这份手册。

如果你还需要本地启动 Flowable UI / REST、修改登录方式、处理 `8086` 访问和登录问题，请先看：

- `docs/Flowable UI与REST本地启动说明.md`

## 2. 最小结论

对大多数业务项目，最小接入只需要两件事：

### 2.1 配置好 `config.yaml`

至少要有：

- `workflow.flowable.*`
- `workflow.api.*`
- `workflow.context.*`

如果要启用目录能力，再补：

- `workflow.directory.*`

### 2.2 启动时注册这一行

```go
workflowapi.MustRegisterFromConfig(server)
```

这是 `v1.3.1` 推荐的标准入口。

## 3. 业务项目最小代码

```go
package main

import (
	"github.com/goodbye-jack/go-common/config"
	commonhttp "github.com/goodbye-jack/go-common/http"
	workflowapi "github.com/goodbye-jack/go-common/workflow/api"
)

func main() {
	addr := config.GetConfigString("addr")
	serviceName := config.GetConfigString("service_name")

	server := commonhttp.NewHTTPServer(serviceName)
	workflowapi.MustRegisterFromConfig(server)

	server.Prepare()
	server.Run(addr)
}
```

到这里，标准工作流路由就会自动注册。

## 4. 最小可运行配置

### 4.1 只启用 Flowable，不启用目录能力

适用场景：

- 先调通基础流程接口
- 先不依赖 LDAP / HTTP 目录

```yaml
service_name: "your-biz-service"
addr: ":9081"

workflow:
  api:
    enabled: true
    prefix: "/api"
    sso_enabled: true
    roles: ""
    callback_path: "/flowable/callback"
    callback_secret: ""
  flowable:
    base_url: "http://127.0.0.1:8087/flowable-rest/service"
    username: "rest-admin"
    password: "flowable-rest-password"
    timeout_seconds: 15
    group_prefix: ""
    role_prefix: "role_"
  context:
    user_name_header: "X-Workflow-User-Name"
    system_code_header: "X-System-Code"
    groups_header: "X-Workflow-Groups"
    roles_header: "X-Workflow-Roles"
  directory:
    provider: "none"
```

### 4.2 启用 LDAP 目录能力

适用场景：

- 需要 `me/manager`
- 需要 `me/department`
- 需要 `org/users/{id}` 这类目录接口

```yaml
workflow:
  api:
    enabled: true
    prefix: "/api"
    sso_enabled: true
  flowable:
    base_url: "http://127.0.0.1:8087/flowable-rest/service"
    username: "rest-admin"
    password: "flowable-rest-password"
  context:
    user_name_header: "X-Workflow-User-Name"
    system_code_header: "X-System-Code"
    groups_header: "X-Workflow-Groups"
    roles_header: "X-Workflow-Roles"
  directory:
    provider: "ldap"
    ldap:
      addr: "ldap://127.0.0.1:8389"
      bind_dn: "cn=admin,dc=example,dc=com"
      bind_password: "ldap-bind-password"
      base_dn: "dc=example,dc=com"
      people_ou: "ou=people"
      department_ou: "ou=departments"
      position_ou: "ou=positions"
      group_ou: "ou=groups"
      use_tls: false
```

### 4.3 启用 HTTP 目录能力

适用场景：

- 业务项目不想直接连 LDAP
- 组织、人、上级、部门关系已经在业务系统或统一组织服务里维护

```yaml
workflow:
  api:
    enabled: true
    prefix: "/api"
    sso_enabled: true
  flowable:
    base_url: "http://127.0.0.1:8087/flowable-rest/service"
    username: "rest-admin"
    password: "flowable-rest-password"
  context:
    user_name_header: "X-Workflow-User-Name"
    system_code_header: "X-System-Code"
    groups_header: "X-Workflow-Groups"
    roles_header: "X-Workflow-Roles"
  directory:
    provider: "ldap"
    ldap:
      addr: "ldap://127.0.0.1:8389"
      bind_dn: "cn=admin,dc=example,dc=com"
      bind_password: "ldap-bind-password"
      base_dn: "dc=example,dc=com"
      people_ou: "ou=people"
      department_ou: "ou=departments"
      position_ou: "ou=positions"
      group_ou: "ou=groups"
      use_tls: false
```

当 `provider=ldap` 时，`go-common` 会直接复用 LDAP 目录能力提供：

- 当前用户档案查询
- 直属领导查询
- 部门查询
- 标准 `/org/users/{userId}`、`/manager`、`/department` 接口

其中 `group_ou` 是给 Flowable UI 组查询和 BPMN `candidateGroups` 使用的 LDAP Group 投影 OU。它不会替代 `people_ou`、`department_ou`、`position_ou`，因此不会影响待办用户的上级、部门等目录查询。

## 5. 如果还要启用下一环节自动分配

最小配置如下：

```yaml
workflow:
  assignment:
    provider: "directory"
    variable_keys:
      assignee: "nextAssignee"
      candidate_users: "nextCandidateUsers"
      candidate_groups: "nextCandidateGroups"
```

说明：

- 如果不需要自动分配，可以不配，默认按 `none`
- 如果想直接复用目录关系，可用 `provider=directory`
- 如果需要最少改造，优先使用 `provider=directory`
- 如果希望完全复用 LDAP 目录，也可以使用 `provider=ldap`

## 6. 如果希望 LDAP / 本地登录态共用一套 BPMN

推荐最少补这段配置：

```yaml
workflow:
  flowable:
    role_prefix: "role_"
  identity:
    role_aliases:
      LDAP_ROLE_CITY_REVIEWER: APP_ROLE_CITY_REVIEW
      LDAP_ROLE_CITY_LEADER: APP_ROLE_CITY_REVIEW
      HTTP_ROLE_CITY_REVIEW: APP_ROLE_CITY_REVIEW
      HTTP_ROLE_FIELD_REPORT: APP_ROLE_BASE_REPORT
    group_aliases:
      CITY_REVIEWERS: city_reviewers
```

原则只有三个：

1. BPMN 里只写稳定候选组编码
2. HTTP / LDAP / JWT 的原始角色码都通过 alias 归一
3. 流程变量统一使用 `nextAssignee` / `nextCandidateUsers` / `nextCandidateGroups`

## 7. 业务项目至少要准备什么登录态上下文

工作流标准接口至少需要识别当前用户，因此业务项目应确保请求中能拿到：

- 用户名或用户唯一标识
- 系统编码
- 角色列表（可选但强烈建议）
- 组列表（可选）

默认请求头为：

- `X-Workflow-User-Name`
- `X-System-Code`
- `X-Workflow-Groups`
- `X-Workflow-Roles`

如果业务项目已有网关统一注入，只需要把这里的 header 名配置对齐即可。

## 8. 接入成功后的最小验证顺序

建议按下面顺序验证：

1. 启动服务，确认日志里出现工作流模块启动摘要
2. 调 `GET /api/me`
3. 调 `GET /api/me/tasks/todo`
4. 调 `GET /api/me/tasks/done`
5. 调 `POST /api/process/start`
6. 调 `GET /api/tasks/{id}/context`
7. 调 `POST /api/tasks/{id}/complete`

如果启用了目录能力，再验证：

8. 调 `GET /api/me/manager`
9. 调 `GET /api/me/department`
10. 调 `GET /api/org/users/{userId}`

## 9. 最常见的 5 个问题

### 9.1 服务启动了，但没有工作流接口

先检查：

- `workflow.api.enabled` 是否为 `true`
- 是否执行了 `workflowapi.MustRegisterFromConfig(server)`

### 9.2 `me` 能调用，但 `me/manager` 不通

先检查：

- `workflow.directory.provider` 是否正确
- LDAP 目录配置是否完整

### 9.3 Flowable 能登录，但待办 / 已办为空

先检查：

- 当前请求头里的用户名是否和 Flowable 任务归属一致
- 当前角色 / 组编码是否和 BPMN 候选组编码一致
- 是否需要补 `workflow.identity.role_aliases`

### 9.4 切换身份来源后流程图里的候选组全失效

先检查：

- BPMN 是否写死了 LDAP 原始岗位码
- 是否把不同身份来源的原始编码映射到同一个规范角色码

### 9.5 自动分配没生效

先检查：

- `workflow.assignment.provider` 是否配置
- `variable_keys` 是否和 BPMN / 业务约定一致
- 目录关系和角色映射是否能正确求解出下一环节办理人

## 10. 最后建议

如果你是新业务项目，请按下面顺序做：

1. 先用 `provider=none` 跑通基础接口
2. 再接 LDAP / directory 目录能力
3. 最后再接自动分配

这样最稳，也最容易定位问题。
