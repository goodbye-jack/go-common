# 工作流接口收口说明

## 1. 文档目的

本文用于明确当前 `go-common/workflow` 的正式接口面、兼容接口面以及字段稳定边界。

接口收口的原则只有一条：

- 后续可以优化内部实现
- 不能随意破坏已经对业务系统承诺的稳定契约

因此本文重点不是重复介绍功能，而是说明：

- 哪些路径是正式标准入口
- 哪些路径只是兼容保留
- 哪些字段属于稳定字段
- 哪些字段属于增强字段

## 2. 当前统一外层返回格式

默认 JSON 接口统一使用：

```json
{
  "data": {},
  "message": "success"
}
```

例外接口：

- `GET /api/process/instance/{id}/definition-xml`

该接口直接返回 XML，不走 JSON 包裹。此行为当前继续保留，但该接口本身已降级为兼容接口。

## 3. 当前正式路由清单

### 3.1 用户与组织接口

- `GET /api/me`
- `GET /api/me/manager`
- `GET /api/me/department`
- `GET /api/org/users/{userId}`
- `GET /api/org/users/{userId}/manager`
- `GET /api/org/users/{userId}/department`

### 3.2 流程与任务接口

- `POST /api/process/start`
- `GET /api/me/tasks/todo`
- `GET /api/me/tasks/done`
- `GET /api/tasks/{id}/context`
- `GET /api/tasks/{id}/form-ref`
- `POST /api/tasks/{id}/complete`

### 3.3 进度与流程图接口

- `GET /api/process-instances/{id}/progress-view`
- `GET /api/process-instances/{id}/progress-timeline`
- `GET /api/biz/{bizId}/progress-view`
- `GET /api/biz/{bizId}/progress-timeline`
- `GET /api/process-instances/{id}/diagram-view`

### 3.4 平台回调接口

- `POST /flowable/callback`

## 4. 当前兼容接口清单

以下接口当前保留兼容，但不建议新业务组继续直接依赖：

- `GET /api/process/instance/{id}/definition-xml`
- `GET /api/process-instances/{id}/composite-diagram`

兼容策略：

- 老页面可以继续使用
- 新页面统一走 `diagram-view`
- 后续正式版仍保留一段时间兼容期，但文档主入口不再推荐这两条接口

## 5. 当前统一用户上下文契约

工作流模块当前依赖以下上下文信息：

- `userId`
- `userName`
- `tenantId`
- `systemCode`
- `groups`
- `roles`

默认读取来源：

- `userId` 来自登录态上下文
- `tenantId` 来自租户上下文
- `userName` 默认来自 `X-Workflow-User-Name`
- `systemCode` 默认来自 `X-System-Code`
- `groups` 默认来自 `X-Workflow-Groups`
- `roles` 默认来自 `X-Workflow-Roles`

可覆写配置：

- `workflow.context.user_name_header`
- `workflow.context.system_code_header`
- `workflow.context.groups_header`
- `workflow.context.roles_header`

工作流模块只消费这些上下文，不负责替业务系统做登录。

## 6. 发起接口的收口规则

### 6.1 正式承诺的工作流发起接口

`POST /api/process/start`

当前稳定请求字段：

- `processDefinitionId`
- `processDefinitionKey`
- `businessKey`
- `bizId`
- `bizType`
- `title`
- `name`
- `variables`

当前稳定返回字段：

- `processInstanceId`
- `processDefinitionId`
- `businessKey`
- `tenantId`

### 6.2 平台自动补齐字段

当请求未显式传入时，平台会自动补齐：

- `startUserId`
- `starterId`
- `starterName`
- `tenantId`
- `systemCode`

这组自动补齐规则视为当前稳定契约的一部分。

### 6.3 对业务系统的约束

正式推荐业务系统自己再封装一层业务发起接口，例如：

- `POST /api/applications`

业务系统自己的发起接口负责：

- 业务数据落库
- 附件落库
- 生成 `bizId`
- 调用 `POST /api/process/start`
- 保存绑定关系

## 7. 列表接口的收口规则

### 7.1 稳定基础字段

`GET /api/me/tasks/todo` 和 `GET /api/me/tasks/done` 的稳定基础字段如下：

- `taskId`
- `taskName`
- `activityId`
- `activityName`
- `processInstanceId`
- `processDefinitionId`
- `bizId`
- `bizType`
- `title`
- `payloadRef`
- `createdAt`
- `completedAt`
- `assignee`
- `owner`

分页稳定字段：

- `total`
- `start`
- `size`

### 7.2 增强字段

以下字段当前仍会返回，但收口定义为增强字段：

- `tenantId`
- `formKey`
- `formName`
- `uiPage`
- `progress`

其中 `progress` 的稳定语义是：

- 列表轻量进度摘要
- 不是完整流程对象

### 7.3 列表查询参数

当前稳定支持：

- `start`
- `size`
- `includeProgress`
- `title`
- `bizType`
- `processDefinitionKey`
- `activityId`
- `status`
- `createdAfter`
- `createdBefore`
- `completedAfter`
- `completedBefore`

兼容参数：

- `keyword` 等价 `title`
- `processKey` 等价 `processDefinitionKey`
- `taskDefinitionKey` 等价 `activityId`
- `withProgress` 等价 `includeProgress`

### 7.4 默认行为

当前默认行为已经收口为：

- 列表默认轻量返回
- 默认页大小 `20`
- 最大页大小 `100`
- 不显式传 `includeProgress=true` 时，不做逐条重型 enrich

## 8. 任务上下文与表单引用收口规则

### 8.1 `GET /api/tasks/{id}/context`

稳定四段结构：

- `task`
- `business`
- `variables`
- `formRef`

这四段结构不应随意变化。

### 8.2 `GET /api/tasks/{id}/form-ref`

稳定语义：

- `configured=true` 表示节点配置了表单
- `resolved=true` 表示已经解析出表单模型
- `fields` 和 `outcomes` 只做参考

平台明确不承诺：

- 用 Flowable 表单模型直接驱动业务前端渲染

## 9. 进度接口收口规则

### 9.1 正式进度接口

- `GET /api/process-instances/{id}/progress-view`
- `GET /api/process-instances/{id}/progress-timeline`
- `GET /api/biz/{bizId}/progress-view`
- `GET /api/biz/{bizId}/progress-timeline`

### 9.2 使用边界

- 有 `processInstanceId` 时，优先按流程实例号查
- 只有 `bizId` 时，按业务单号查

### 9.3 语义边界

- `progress-view` 承担步骤视图和流程摘要职责
- `progress-timeline` 承担实际流转轨迹职责
- 列表页不要高频批量调用这两类接口

## 10. 流程图接口收口规则

### 10.1 正式标准入口

`GET /api/process-instances/{id}/diagram-view`

这是当前流程图正式标准入口。

统一返回：

- `processInstanceId`
- `mode`
- `composite`
- `parentXml`
- `children`

### 10.2 收口原则

- 调用方不需要自己判断是单流程还是父子流程
- 后端自动判断并返回统一结构
- 前端根据 `mode` 或 `composite` 决定渲染方式

### 10.3 兼容接口说明

以下接口继续保留但不再作为正式主入口：

- `definition-xml`
- `composite-diagram`

## 11. 已知边界

当前正式版前需要明确的能力边界：

- 常见条件表达式、默认分支、正向路径推导已支持
- `subProcess` 已支持内部节点展开
- `callActivity` 已支持直接子流程展开
- 单流程图和父子流程图已统一到 `diagram-view`
- 会签、多实例、深层嵌套 `callActivity`、复杂 EL 函数仍不在当前正式承诺范围

## 12. 收口结论

当前工作流接口已经收敛为一组正式主入口：

- 用户与组织接口
- 启动接口
- 待办与已办接口
- 任务上下文与表单引用接口
- 进度接口
- 统一流程图接口 `diagram-view`

后续内部无论继续做缓存、读模型、索引还是复杂流程增强，都不应破坏这组稳定契约。
