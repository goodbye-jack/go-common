# 工作流身份组契约说明

## 1. 这份文件是做什么的

这份文档用于说明 `go-common/workflow` 新增的“身份编码归一”能力。

它主要服务两类人：

- 后端开发：知道 `config.yaml` 里该怎么配 `workflow.identity.*`
- AI / 接手同事：快速理解为什么 HTTP / LDAP 切换时 BPMN 不需要跟着改

## 2. 要解决的问题

在真实项目里，不同身份源给出的编码往往不一样：

- HTTP / 业务库里可能是 `APP_ROLE_CITY_REVIEW`
- LDAP 岗位里可能是 `LDAP_ROLE_CITY_REVIEWER`
- JWT 里可能又是别的角色码

如果 BPMN 里直接写这些“来源编码”，后面一旦切换 Provider，就会出现：

- 流程图候选组需要跟着改
- 我的待办查不到
- 固定候选组节点与动态分配节点不一致

因此需要一层统一契约：

- 上游来源编码可以不同
- 进入工作流前统一归一为同一套业务角色码
- Flowable BPMN 永远只认这一套稳定编码

## 3. 推荐契约

建议把 Flowable 最终使用的角色组编码固定为业务角色编码。

例如：

- `role_APP_ROLE_CITY_REVIEW`
- `role_APP_ROLE_COUNTY_REVIEW`
- `role_APP_ROLE_BASE_REPORT`
- `role_APP_ROLE_TOWN_REVIEW`

也就是说：

- BPMN 固定候选组写规范 groupId
- 登录态角色头写原始角色也可以
- LDAP 岗位编码写原始岗位也可以
- `go-common` 负责在进入 Flowable 前统一归一

## 4. 配置方式

```yaml
workflow:
  flowable:
    role_prefix: "role_"
  identity:
    role_aliases:
      LDAP_ROLE_CITY_REVIEWER: APP_ROLE_CITY_REVIEW
      LDAP_ROLE_CITY_LEADER: APP_ROLE_CITY_REVIEW
      LDAP_ROLE_COUNTY_OWNER: APP_ROLE_COUNTY_REVIEW
      LDAP_ROLE_FIELD_REPORTER: APP_ROLE_BASE_REPORT
    group_aliases:
      CITY_REVIEWERS: city_reviewers
```

说明：

- `role_aliases`：把原始角色/岗位码映射成规范角色码
- `group_aliases`：把原始组编码映射成规范组编码
- `role_prefix` / `group_prefix`：由 Flowable groupId 规则决定

## 5. 生效位置

当前这层归一能力会用于：

- 工作流当前用户上下文解析
- 待办候选组判定
- 目录型自动分配生成 `nextCandidateGroups`
- 启动期配置摘要日志

## 6. 风险控制

这层能力默认是“非侵入”的：

- 不配置 `workflow.identity.*` 时，行为保持原样
- 配了映射时，只做编码归一，不改现有接口协议
- 业务项目仍然可以继续使用 LDAP / directory provider

## 7. 使用建议

如果你希望不同身份来源保持同一套 BPMN，建议按下面原则执行：

1. BPMN 只写规范 groupId
2. 登录态头部传原始角色码或岗位码都可以
3. 在 `workflow.identity.role_aliases` 里统一映射
4. 启动后检查日志里的 `role_alias_count` / `group_alias_count`

这样后面切换身份源时，一般不需要改 BPMN。
