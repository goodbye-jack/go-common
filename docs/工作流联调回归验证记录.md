# 工作流联调回归验证记录

## 1. 文档目的

这份文档用于固化当前 `go-common/workflow` 在真实业务联调中的回归验证结果。

它不是单次接口调试笔记，也不是研发个人备忘，而是给后续业务组、测试组、发布人员统一确认当前版本成熟度使用的正式记录。文档的核心目标只有一个：把“这版到底已经验证到了什么程度”说清楚，避免后续发布时再靠聊天记录或零散截图追溯。

## 2. 本轮验证范围

本轮验证围绕两类真实业务流程展开。

第一类，是文物鉴定业务流程。这个流程重点验证了标准审批链路、节点表单参考解析、待办处理、已办查询、按业务单号查看进度、业务前端内嵌流程图和时间线展示等能力。

第二类，是文物告警业务流程。这个流程重点验证了多角色流转、候选任务可见性、业务主表与流程绑定、业务详情页展示、按业务单号查看流程进度，以及不同业务类型共用同一套工作流平台接口的可行性。

这两类流程覆盖的不是同一套字段，但它们共用了同一套 `go-common/workflow` 接口能力，因此足以证明当前平台已经具备“平台能力和业务字段解耦”的落地基础。

## 3. 本轮验证的环境边界

本轮联调涉及三个本地项目。

工作流原始 Java 适配器项目位于：

- `/usr/local/work/workspace/dev_personal_project/zhuoue-flowable-6.8.1-adapter`

目标承载模块项目位于：

- `/usr/local/work/workspace/dev_personal_project/go-common`

业务验证后端与前端分别位于：

- `/usr/local/work/workspace/dev_personal_project/flowable-biz-demo`
- `/usr/local/work/workspace/dev_personal_project/flowable-biz-frontend`

验证过程中实际使用了 Flowable 6.8.1、LDAP、MySQL、Redis，以及业务系统本地的 `warmcity` 业务库。

## 4. 已验证通过的核心链路

从联调结果看，下面这些核心链路已经形成完整闭环。

首先，业务系统能够完成登录态解析，并把当前用户上下文稳定传给 `go-common/workflow`。用户资料、上级、部门信息已经可以通过目录能力返回，说明当前工作流接口不再强依赖 Java adapter 才能完成目录解析。

其次，业务系统已经能够先写业务主表，再发起流程，并把返回的 `processInstanceId` 回写到业务表和本地绑定表。这意味着“业务数据归业务系统，流程实例归工作流引擎”的边界已经跑通，而不是把业务明细强塞到 Flowable 底表里。

再次，当前用户能够正确查询自己的待办和已办，并且候选用户、候选组的可见性判断已经回到 `go-common/workflow` 内部处理，不需要业务系统自己额外做二次过滤。

然后，任务处理页已经能够同时拿到任务上下文、流程变量和节点表单参考信息。这里的表单模型语义已经固定为“节点字段参考”，不是前端渲染协议，这一点在真实联调里也已经验证可行。

最后，业务详情页和任务处理页都已经能够按流程实例号或按业务单号查看流程图和流程时间线。流程图已经改为业务前端内嵌渲染 BPMN XML，避免了旧版图形服务带来的中文乱码和新页签跳转问题。

## 5. 已修正并回归通过的关键问题

这一轮联调不是一次性平滑完成的，中间实际上修过几类关键问题，而这些问题已经完成回归验证。

第一类，是表单引用解析问题。前期出现过 Flowable 表单 key、标识、引用配置不一致时 `resolved=false` 的情况，后续通过明确“节点 formKey 对应表单定义 key”的口径，并引入独立 Flowable 元数据库配置后，表单参考解析链路已经稳定。

第二类，是待办和已办的人员隔离问题。前期出现过候选任务查询不准、已办归属判断不严、某些用户能看到与自己无关历史任务的问题。后续通过把范围判断、候选任务判定、已办归属过滤统一收敛在 `go-common/workflow` 内部，并补充候选索引和已办投影后，这个问题已经在真实账号联调中回归通过。

第三类，是 `tenantId` 相关的范围判断问题。前期如果完全依赖 Flowable 原生运行时数据中的 `tenantId`，会出现部分流程查不到或误判的问题。现在已经调整为写链路继续保留 `tenantId`，读链路对 `tenantId` 宽容处理，并结合流程摘要与本地绑定关系做范围校验，因此“有租户”和“部分运行态无租户”两种场景都已经能够稳定兼容。

第四类，是业务标题和业务详情回填问题。前期某些待办、已办列表中标题展示不完整，后续已经把标题补齐逻辑前移，优先从流程变量、业务绑定和业务主表中补充，现已回归通过。

第五类，是流程进度展示语义问题。前期列表、详情、流程图、时间线混用同一套重型数据语义，容易引发性能和展示混乱。现在已经完成第六阶段收口：列表只展示轻量进度摘要，完整流程图和完整时间线一律走详情接口，这一点已经在代码和文档层统一。

## 6. 六个性能优化阶段的落地结果

到当前为止，最初约定的六个优化阶段已经全部完成，并且已经落实到代码和文档，而不是停留在讨论层面。

第一，待办和已办列表默认不再做逐条重型 enrich。列表已经从“每条都回源拉完整进度”收敛成“优先返回基础字段和轻量摘要”。

第二，列表分页、过滤、搜索能力已经补齐，默认分页大小和最大分页上限都已经在平台侧控制，避免业务前端长期依赖大页拉全量数据。

第三，流程范围摘要缓存已经接入，`matchesProcessScope` 以及部分摘要补齐路径会优先使用 Redis 中的流程摘要缓存，Redis 不可用时自动降级。

第四，候选任务判定已经完成索引化第一版。当前待办查询会优先使用候选任务索引，降低了逐条回源 Flowable `identity links` 的频率。

第五，已办查询已经完成历史投影第一版。当前第一页场景下会优先读取已办投影，不足部分再回源 Flowable 历史接口补齐。

第六，列表中的 `progress` 语义已经彻底收口为“轻量进度摘要”，不再把完整流程对象默认混入列表结果。

这六步完成后，当前版本已经从“功能验证版”进入“具备基础性能可用性和清晰接口边界的接入验证版”。

## 7. 当前版本仍然存在的边界

虽然这一版已经可以交付给业务组接入验证，但仍然不应该把它描述成最终正式稳定版。

从流程推导能力看，当前对常见审批流已经足够，并且已经增强了常见网关条件表达式与 `inclusiveGateway` 的正向路径推导能力，`subProcess` 和直接子流程 `callActivity` 也已经能够展开到实际节点级别；但会签、多实例、深层嵌套 `callActivity` 和更复杂 EL 语义仍然需要后续继续增强。

从性能演进路径看，当前已经有缓存、候选索引、已办投影的第一版，但还没有完全进入“回调驱动的本地读模型”阶段。在中小规模业务场景里，这版已经够用；如果进入更大规模场景，仍然建议后续继续向本地读模型和补偿同步演进。

从发布节奏看，当前更合适的标签仍然是 `dev` 或 `beta`，而不是直接定义成最终稳定正式版。

## 8. 当前结论

综合本轮真实联调、问题修复、回归结果和接口收口情况，可以得出一个明确结论：

`go-common/workflow` 已经具备作为工作流平台验证版对外提供给业务组接入的条件。

这里的“具备条件”指的是：

- 核心业务链路已经跑通
- 权限与人员隔离问题已经修正
- 两个真实业务流程已经完成联调闭环
- 接口契约已经收口
- 前后端接入模板已经补齐
- 性能优化路线的前六阶段已经落地

因此下一步最合理的动作，不是继续无边界扩展功能，而是进入版本说明固化和验证版发布准备阶段。
