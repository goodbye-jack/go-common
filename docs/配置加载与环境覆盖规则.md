# 配置加载与环境覆盖规则

本文说明 `go-common v1.3.5` 起统一采用的配置加载顺序，以及 `CONFIG_FILE` / `CONFIG_ENV` 的使用方式。

## 1. 总体原则

- 永远先读取基础配置：`config.yaml`
- 再按优先级选择一个覆盖来源
- 运行时不再直接兼容旧的平铺 key，如 `service_name`、`addr`、`cookie_token`

## 2. 覆盖优先级

覆盖优先级固定如下：

1. `CONFIG_FILE`
2. `CONFIG_ENV`
3. 无覆盖

### 2.1 同时存在 `CONFIG_FILE` 和 `CONFIG_ENV`

如果两个变量同时存在：

- 仍然先读取 `config.yaml`
- 再读取 `CONFIG_FILE` 指定的覆盖文件
- `CONFIG_ENV` 被忽略
- 启动日志会明确打印“已按优先级使用 CONFIG_FILE，忽略 CONFIG_ENV”

这样做的目的是保证覆盖规则单一、可预期，避免出现：

`config.yaml + config.dev.yaml + custom.yaml`

这类难以排查的三层叠加。

## 3. `CONFIG_ENV` 的使用方式

`CONFIG_ENV` 适合开发环境、脚本启动或按约定命名的部署方式。

例如：

```bash
CONFIG_ENV=dev
```

运行时读取顺序为：

```text
config.yaml
config.dev.yaml
```

如果 `config.dev.yaml` 不存在，程序会直接报错，而不是静默跳过。

## 4. `CONFIG_FILE` 的使用方式

`CONFIG_FILE` 适合：

- Docker / Kubernetes 挂载外部配置文件
- 本地开发者使用个人覆盖文件
- 运维明确指定某一份部署配置

例如：

```bash
CONFIG_FILE=./config/config.duyc.yaml
```

运行时读取顺序为：

```text
config.yaml
./config/config.duyc.yaml
```

再例如 Docker 场景：

```bash
CONFIG_FILE=/app/runtime/pano-material-prod.yaml
```

运行时读取顺序为：

```text
config.yaml
/app/runtime/pano-material-prod.yaml
```

如果 `CONFIG_FILE` 指定的文件不存在，程序会直接报错。

## 5. 默认行为

如果两个变量都没有配置：

- 只读取 `config.yaml`

这也是本地开发默认推荐方式。

## 6. 推荐实践

### 6.1 本地开发

- 默认只用 `config.yaml`
- 需要个人差异配置时，优先使用 `CONFIG_FILE=./config/config.<name>.yaml`

### 6.2 常规环境切换

- 使用 `CONFIG_ENV=dev/test/prod`
- 对应维护 `config.dev.yaml`、`config.test.yaml`、`config.prod.yaml`

### 6.3 Docker / Kubernetes

- 优先使用 `CONFIG_FILE`
- 通过挂载卷或 ConfigMap / Secret 把覆盖文件映射到容器内固定路径

## 7. 启动日志要求

`go-common` 在启动时会明确输出：

- 读取的基础配置路径
- 是否读取了覆盖配置
- 如果同时设置了 `CONFIG_FILE` 与 `CONFIG_ENV`，会打印忽略说明

这样做的目标是：开发、运维、排障时都能快速确定“最终到底用了哪两份文件”。
