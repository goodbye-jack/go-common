# 工作流平台标准接口说明

## 1. 文档目的

本文用于定义 `go-common/workflow` 当前正式对外提供的标准接口、字段语义、职责边界和推荐接入方式。

这份文档重点回答四个问题：

- 业务系统接入时，应该依赖哪些工作流标准接口
- 哪些接口属于兼容接口，不建议新系统继续直接依赖
- “流程发起”到底应该由工作流接口承接，还是由业务系统自己的发起接口承接
- 单流程图和父子流程图如何通过统一入口获取

当前文档内容基于已经完成联调验证的两类真实业务流程沉淀：

- 文物鉴定流程
- 文物告警流程

因此本文不是概念稿，而是面向实际交付的接入基线。

## 2. 职责边界

### 2.1 `go-common/workflow` 负责的能力

- 当前用户工作流上下文解析
- LDAP 用户、上级、部门、岗位信息查询
- 标准流程启动能力
- 我的待办查询
- 我的已办查询
- 任务上下文查询
- 任务完成
- 节点表单引用解析
- 流程进度视图查询
- 流程时间线查询
- 统一流程图视图查询

### 2.2 业务系统负责的能力

- 业务主表、业务明细表、附件表设计
- 业务字段校验
- 业务详情页、发起页、处理页、我的申请页
- 业务状态流转
- `bizId` 与本地业务记录的持久化绑定
- 业务字段到流程变量的映射

一句话概括：`go-common/workflow` 提供通用流程能力，业务系统保留自己的业务数据和业务页面。

## 3. 标准接入链路

推荐业务系统按下面的顺序接入：

1. 登录后，把 `userId`、`userName`、`tenantId`、`systemCode`、`groups`、`roles` 放入请求上下文。
2. 用户点击“发起”时，业务系统先写自己的业务主表和绑定表。
3. 业务系统调用工作流标准启动能力，拿到 `processInstanceId`。
4. 业务系统把 `bizId` 与 `processInstanceId` 建立稳定关联。
5. 我的申请页查询业务系统自己的业务表。
6. 我的待办、我的已办直接查询工作流标准接口。
7. 处理页同时加载业务详情、任务上下文、流程进度、流程图视图。
8. 任务完成时，由业务系统把业务页面字段映射为流程变量，再调用任务完成接口。

## 4. 流程发起的两层接口

这部分必须明确，否则业务组很容易误解。

### 4.1 工作流标准启动接口

`POST /api/process/start`

这是 `go-common/workflow` 提供的标准流程启动能力，负责：

- 根据 `processDefinitionId` 或 `processDefinitionKey` 启动流程
- 注入 `bizId`、`bizType`、`title`、`variables`
- 返回 `processInstanceId`

标准请求模型核心字段：

- `processDefinitionId`
- `processDefinitionKey`
- `businessKey`
- `bizId`
- `bizType`
- `title`
- `name`
- `variables`

标准返回字段：

- `processInstanceId`
- `processDefinitionId`
- `businessKey`
- `tenantId`

### 4.2 业务系统推荐封装发起接口

推荐每个业务系统都再封装一层自己的“发起业务单”接口，例如：

- `POST /api/applications`

这类接口负责：

- 写业务主表
- 生成业务单号
- 保存业务 JSON 明细
- 保存附件
- 调用 `POST /api/process/start`
- 回写 `processInstanceId`

因此正式接入时，业务组通常不应直接从前端裸调 `POST /api/process/start`，而应先调业务系统自己的发起接口。

## 5. 正式标准接口清单

以下接口定义为当前正式推荐接入面。

### 5.1 用户与组织接口

- `GET /api/me`
- `GET /api/me/manager`
- `GET /api/me/department`
- `GET /api/org/users/{userId}`
- `GET /api/org/users/{userId}/manager`
- `GET /api/org/users/{userId}/department`

### 5.2 流程与任务接口

- `POST /api/process/start`
- `GET /api/me/tasks/todo`
- `GET /api/me/tasks/done`
- `GET /api/tasks/{id}/context`
- `GET /api/tasks/{id}/form-ref`
- `POST /api/tasks/{id}/complete`

### 5.3 进度与图视图接口

- `GET /api/process-instances/{id}/progress-view`
- `GET /api/process-instances/{id}/progress-timeline`
- `GET /api/biz/{bizId}/progress-view`
- `GET /api/biz/{bizId}/progress-timeline`
- `GET /api/process-instances/{id}/diagram-view`

其中：

- `progress-view` 负责流程步骤视图和节点状态摘要
- `progress-timeline` 负责实际流转轨迹
- `diagram-view` 负责统一流程图入口，自动判断单流程还是父子复合流程

## 6. 兼容接口清单

以下接口当前保留，但定义为兼容接口，不建议新系统继续直接依赖：

- `GET /api/process/instance/{id}/definition-xml`
- `GET /api/process-instances/{id}/composite-diagram`

兼容语义如下：

- `definition-xml` 只返回单份 BPMN XML，适合旧前端
- `composite-diagram` 是 `diagram-view` 统一前的过渡接口

推荐做法：

- 新系统统一使用 `diagram-view`
- 老系统可以逐步从 `definition-xml` 或 `composite-diagram` 平滑迁移

## 7. `diagram-view` 统一接口说明

### 7.1 设计目的

`diagram-view` 是当前正式推荐的流程图标准入口。

它解决的问题是：

- 调用方无需先判断流程是否包含 `callActivity`
- 调用方无需分别调用单流程图接口和复合流程图接口
- 单流程和父子流程都走同一个 URL

### 7.2 返回结构

统一返回结构如下：

```json
{
  "processInstanceId": "xxx",
  "mode": "single",
  "composite": false,
  "parentXml": "<definitions>...</definitions>",
  "children": []
}
```

如果是父子复合流程，则返回：

```json
{
  "processInstanceId": "xxx",
  "mode": "composite",
  "composite": true,
  "parentXml": "<definitions>...</definitions>",
  "children": [
    {
      "callActivityId": "call_external_review",
      "callActivityName": "外部联动审核子流程",
      "processInstanceId": "child-instance-id",
      "processDefinitionId": "child-definition-id",
      "processDefinitionKey": "workflow_feature_child_test",
      "xml": "<definitions>...</definitions>"
    }
  ]
}
```

### 7.3 字段语义

- `mode`
  - `single` 表示单流程图
  - `composite` 表示父子复合流程图
- `composite`
  - 布尔兼容字段
- `parentXml`
  - 主流程 BPMN XML
- `children`
  - 每个 `callActivity` 对应一个子流程图块

## 8. 待办与已办接口的稳定字段

列表接口：

- `GET /api/me/tasks/todo`
- `GET /api/me/tasks/done`

以下字段定义为稳定基础字段：

- `taskId`
- `taskName`
- `activityId`
- `activityName`
- `processInstanceId`
- `processDefinitionId`
- `bizId`
- `bizType`
- `title`
- `payloadRef`
- `createdAt`
- `completedAt`
- `assignee`
- `owner`

分页稳定字段：

- `total`
- `start`
- `size`

增强字段：

- `tenantId`
- `formKey`
- `formName`
- `uiPage`
- `progress`

注意：

- `progress` 在列表页中只表示轻量进度摘要
- 完整流程详情必须走 `progress-view`、`progress-timeline`、`diagram-view`

## 9. 任务上下文与表单引用接口

### 9.1 `GET /api/tasks/{id}/context`

返回结构固定为四段：

- `task`
- `business`
- `variables`
- `formRef`

这四段结构视为当前稳定契约。

### 9.2 `GET /api/tasks/{id}/form-ref`

关键字段：

- `configured`
- `resolved`
- `formKey`
- `formName`
- `fields`
- `outcomes`

架构原则：

- 表单模型只做节点字段参考
- 不作为业务前端渲染协议
- 解析失败不应阻断流程流转

## 10. 进度接口说明

### 10.1 `progress-view`

适合：

- 步骤条
- 节点状态概览
- 当前节点摘要
- 流程图高亮输入

关键字段：

- `summary`
- `steps`
- `completedActivityIds`
- `currentActivityIds`

### 10.2 `progress-timeline`

适合：

- 实际流转轨迹
- 已办记录追踪
- 当前正在处理节点展示
- 正向未来节点展示

关键字段：

- `summary`
- `items`

增强字段：

- `items[].occurrence`
- `steps[].occurrenceCount`

## 11. 推荐前端页面结构

推荐业务前端至少包含：

1. 发起页
2. 我的申请页
3. 我的待办页
4. 我的已办页
5. 业务详情页
6. 业务处理页

推荐接口组合：

- 发起页：业务系统自定义发起接口
- 我的申请页：业务系统自己的业务表 + `progress-view` + `diagram-view`
- 我的待办页：`GET /api/me/tasks/todo`
- 我的已办页：`GET /api/me/tasks/done`
- 处理页：`GET /api/tasks/{id}/context` + `progress-view` + `progress-timeline` + `diagram-view`

## 12. 当前结论

当前正式推荐的工作流标准接口已经收敛为一组稳定接入面：

- 启动
- 待办
- 已办
- 任务上下文
- 任务处理
- 组织目录
- 进度视图
- 时间线
- 统一流程图视图

对新业务系统而言，优先按这组标准接口接入即可，不需要再直接依赖 Flowable 原始 REST 接口，也不应继续围绕旧的 `definition-xml` 设计新页面。
