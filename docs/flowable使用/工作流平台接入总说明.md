   - 面向流程设计人员、平台架构人员

## 2. 先说结论

业务系统接入工作流时，只需要记住下面 8 条：

1. 业务数据必须落自己的业务表，不能指望 Flowable 表单代替业务存储。
2. 发起流程必须分两步：
   - 先调业务保存接口
   - 再调工作流启动接口
3. `claim / unclaim / delegate / resolve / transfer` 只负责“谁来办”。
4. `complete` 只负责“流程往哪走”。
5. 退回、重审、驳回、补正必须通过 BPMN 网关和变量表达，不新增“任意跳转”接口。
6. BPMN 中不要直接写业务私有变量如 `${customApproverId}` 作为长期平台契约，统一用标准变量。
7. 流程轨迹现在分三类：
   - `progress-view` 看流程摘要、BPMN 结构、当前节点总览
   - `progress-timeline` 看真正走过或当前正在处理的主链路步骤
   - `task-records` 看每个步骤下发生过哪些办理动作
   - `action-timeline` 只看办理权动作审计，不建议直接拿来做主进度页
8. “我的已办”和“我的办理记录”不是一回事：
   - `done` 只看真正完成过的任务
   - `task-records` 看认领、委派、转办、完成等实际办理动作

## 2.1 5 分钟接入清单

如果你是第一次接入，不要一上来就翻所有接口。按下面顺序做，基本不会走偏。

### 第 1 步：先把平台启动起来

你至少要有下面两类服务：

1. `flowable-ui`
   - 给流程设计人员画流程图、发布流程
2. `flowable-rest`
   - 给业务系统运行时调用

如果本地还没部署，先看：

- `Flowable UI与REST本地启动说明.md`

验收标准：

- 能打开 `flowable-ui`
- 能登录 `flowable-ui`
- 能在 `flowable-ui` 发布流程
- `flowable-rest` 的 `/service/management/engine` 可以返回结果

### 第 2 步：先把流程图画对

在 `flowable-ui` 里完成下面事情：

1. 创建 BPMN 流程
2. 配置用户任务
3. 配置候选人 / 候选组 / 固定办理人
4. 发布流程定义
5. 记住流程定义 `key`

如果你不确定节点到底该怎么配，先看：

- `工作流BPMN建模与变量规范.md`

验收标准：

- 流程已经发布，不是草稿
- 你知道自己要调用的 `processDefinitionKey`
- 节点办理人配置符合 Flowable 标准写法

### 第 3 步：业务系统补 `config.yaml`

业务系统至少补齐下面这些配置：

```yaml
workflow:
  api:
    enabled: true
    prefix: "/api/v1/workflow"
    sso_enabled: true
    callback_path: "/flowable/callback"
    callback_secret: ""
  flowable:
    base_url: "http://127.0.0.1:8087/flowable-rest/service"
    username: "rest-admin"
    password: "PLEASE_CHANGE"
  directory:
    provider: "ldap"
  assignment:
    provider: "directory"
```

如果需要完整字段解释，继续看本文第 5 章“推荐配置”。

验收标准：

- 业务系统已经知道 `flowable-rest` 地址、账号、密码
- 业务系统已经决定目录来源是 `ldap` 还是 `none`
- 工作流接口前缀已经明确

### 第 4 步：业务后端注册工作流接口

这一步是最容易漏掉的。

只配 YAML 不会自动对外暴露工作流接口。你还必须在业务代码里注册：

```go
workflowapi.MustRegisterFromConfig(server)
```

推荐顺序：

```go
server := commonhttp.NewHTTPServer(serviceName)
registerBizRoutes(server)
workflowapi.MustRegisterFromConfig(server)
server.Prepare()
server.Run(addr)
```

完整说明看本文第 `5.3` 节“业务程序代码如何接入”。

验收标准：

- 启动日志里能看到 `WORKFLOW ROUTES REGISTERED`
- `/api/v1/workflow/...` 相关接口已经对外可访问

### 第 5 步：业务接口和流程接口分开

标准接入不是只做一个“万能 applications 接口”。

标准做法是拆成两步：

1. 先调业务保存接口
   - 保存自己的业务表
   - 生成 `bizId`
2. 再调工作流启动接口
   - `POST /api/v1/workflow/process/start`

验收标准：

- 业务保存失败时，不会误启动流程
- 流程启动失败时，可以明确知道问题在工作流侧
- `bizId` 与 `processInstanceId` 能建立映射关系

### 第 6 步：前端按标准接口接页面

前端最常用的页面，一般这样接：

- 发起页
  - 业务保存接口
  - `POST /api/v1/workflow/process/start`
- 我的待办
  - `GET /api/v1/workflow/me/tasks/todo`
- 我的已办
  - `GET /api/v1/workflow/me/tasks/done`
- 处理页
  - `GET /api/v1/workflow/tasks/{id}/context`
  - `GET /api/v1/workflow/tasks/{id}/form-ref`
  - `GET /api/v1/workflow/biz/{bizId}/progress-timeline`
  - `GET /api/v1/workflow/biz/{bizId}/task-records`
- 办理动作
  - `claim / unclaim / delegate / resolve / transfer / complete`

验收标准：

- 前端没有直接调用 `flowable-ui`
- 前端没有直接调用 `flowable-rest`
- 前端统一调用业务系统暴露出来的接口

### 第 7 步：最后联调整条链路

至少走完一次完整流程：

1. 登录业务系统
2. 保存业务
3. 启动流程
4. 查询待办
5. 打开处理页
6. 完成一次 `claim / complete` 或 `delegate / resolve / complete`
7. 查看进度时间线和办理记录
8. 验证业务状态是否同步更新

如果这 8 步都通了，说明这套工作流接入已经具备可交付基础。

## 2.2 常见接入错误 Top 10

这一节不是补充阅读，而是高频踩坑清单。很多“接口不通”“待办不见了”“流程启动了但没人办”本质上都属于下面这些问题。

### 1. 只配了 `config.yaml`，没有在代码里注册工作流接口

典型表现：

- 项目启动成功了
- 但访问 `/api/v1/workflow/...` 全部 `404`
- 启动日志里没有 `WORKFLOW ROUTES REGISTERED`

根因：

- 你只写了 `workflow.*` 配置
- 但没有在代码里调用：

```go
workflowapi.MustRegisterFromConfig(server)
```

正确做法：

- 在业务路由注册完成后调用 `workflowapi.MustRegisterFromConfig(server)`
- 然后执行 `server.Prepare()`

### 2. 把 `flowable-ui` 当成运行时接口服务

典型表现：

- 前端或后端去调 `flowable-ui` 地址
- 明明流程图已经能打开，但流程接口就是不通

根因：

- `flowable-ui` 是设计平台，不是给业务系统跑运行态接口的

正确做法：

- `flowable-ui` 只负责画图、配图、发布
- `flowable-rest` 才是 go-common 调用的运行时服务
- 业务系统配置里 `workflow.flowable.base_url` 必须指向 `flowable-rest/service`

### 3. `flowable-rest` 地址配错了

典型表现：

- 启动流程时报 `404`
- 查询引擎信息时报认证弹窗或找不到路径

常见错法：

- 少写了 `/flowable-rest/service`
- 错把 `flowable-ui` 地址填进 `base_url`

正确示例：

```yaml
workflow:
  flowable:
    base_url: "http://127.0.0.1:8087/flowable-rest/service"
```

### 4. 前端直接调用 `flowable-rest`

典型表现：

- 页面里直接写死 Flowable 地址
- 遇到跨域、鉴权、密码暴露、接口风格不一致等问题

根因：

- 没把业务系统当成统一接入层

正确做法：

- 前端只调业务系统接口
- 业务系统通过 go-common 统一代理工作流能力

### 5. 把“业务保存”和“流程启动”硬写成一个黑盒接口

典型表现：

- 一个接口里既落业务表，又起流程
- 一旦中间失败，不知道该回滚业务还是补流程

根因：

- 职责边界没有拆开

正确做法：

1. 先调业务保存接口
2. 拿到 `bizId`
3. 再调 `POST /api/v1/workflow/process/start`

这样失败定位和补偿都会简单很多。

### 6. BPMN 节点里直接依赖业务私有变量

典型表现：

- 流程图里大量出现 `${bureauAdminId}`、`${areaOwnerId}`、`${stationManagerId}` 这类写法
- 流程只能在某一个业务系统里勉强复用

根因：

- 把业务字段直接当成平台级流程契约

正确做法：

- 优先使用标准 Flowable 人员配置能力
- 需要变量驱动时，优先使用标准变量：
  - `nextAssignee`
  - `nextCandidateUsers`
  - `nextCandidateGroups`

### 7. 流程图发布了，但调用的不是已发布版本

典型表现：

- UI 里明明改了流程
- 但运行时还是走旧逻辑

根因：

- 流程图只是保存了，没有真正发布
- 调用时 `processDefinitionKey` 不是你以为的那个流程

正确做法：

- 确认流程已经 `Publish`
- 确认启动接口里传的是正确的 `processDefinitionKey`
- 必要时去 `flowable-rest` 查定义列表确认版本

### 8. 角色、组、用户编码没有统一

典型表现：

- 节点配置了候选组，但待办没人收到
- 待办里显示一堆看不懂的组编码

根因：

- LDAP 组编码、业务角色码、Flowable 候选组编码没有归一

正确做法：

- 明确谁是“用户 ID”
- 明确谁是“角色码”
- 通过 `workflow.identity.role_aliases` 和 `group_aliases` 做统一映射
- BPMN 中使用统一后的稳定编码

### 9. 回调地址能配，但业务系统没真正处理业务状态

典型表现：

- `callbackUrl` 配了
- `flowable/callback` 也收到了
- 但业务表状态完全没变

根因：

- 以为 callback 会自动修改你的业务表
- 但实际上 go-common 只负责接收和解析回调，不会凭空知道你的业务状态字段怎么更新

正确做法：

- 把 `flowable/callback` 当成“流程事件通知入口”
- 在业务系统里根据事件类型、节点 ID、流程状态去更新自己的业务表

### 10. 不区分待办、已办、办理记录、进度时间线

典型表现：

- 前端把所有接口混着用
- 页面上要么重复显示，要么信息断裂

正确理解如下：

- `todo`
  - 看“现在等谁处理”
- `done`
  - 看“我真正完成过哪些任务”
- `progress-timeline`
  - 看“流程主链路实际走到了哪”
- `task-records`
  - 看“某个节点发生过哪些办理动作，如认领、委派、转办、完成”
- `action-timeline`
  - 看“动作审计流”，不建议直接当主流程步骤页

如果前端只想做一个主进度页，优先组合：

- `progress-timeline`
- `task-records`

而不是直接拿 `action-timeline` 当主页面。

## 3. 总体架构与职责边界

推荐架构固定为三层：

1. `go-common/workflow`
   - 标准流程接口
   - 目录能力
   - 身份归一
   - 自动分配能力
   - Flowable REST 适配
2. `flowable-ui + flowable-rest`
   - `flowable-ui` 负责流程图、表单模型、部署发布、权限管理
   - `flowable-rest` 负责流程运行态和历史态
3. 业务系统
   - 业务表
   - 业务接口
   - 业务页面
   - 业务状态
   - 业务字段校验

### 3.1 `go-common/workflow` 负责什么

- 当前用户工作流上下文解析
- 启动流程
- 我的待办
- 我的已办
- 任务上下文
- 任务动作
- 流程进度视图
- 流程进度时间线
- 流程动作时间线
- 流程办理记录
- 我的办理记录
- 流程图视图
- 目录查询
- 自动分配

### 3.2 `go-common/workflow` 不负责什么

- 业务主表设计
- 业务详情组装
- 业务表单渲染
- 业务状态机
- 业务权限模型
- 行业专属字段语义

### 3.3 业务系统必须自己负责什么

- 业务保存接口
- 业务详情接口
- 业务列表接口
- 业务字段校验
- 附件管理
- 业务状态流转
- 流程变量与业务字段映射

## 4. 标准接入顺序

标准调用链路如下：

1. 用户登录，业务系统在请求上下文里放入：
   - `userId`
   - `userName`
   - `tenantId`
   - `systemCode`
   - `groups`
   - `roles`
2. 前端点击“提交申请”。
3. 前端先调用业务保存接口。
4. 业务后端写自己的业务表，生成 `bizId`。
5. 前端或业务后端调用 `POST /api/v1/workflow/process/start`。
6. 工作流返回 `processInstanceId`。
7. 业务后端把 `bizId` 与 `processInstanceId` 绑定。
8. 后续：
   - 我的待办、我的已办走工作流标准接口
   - 业务详情走业务系统自己的详情接口
   - 处理页同时加载业务详情 + 工作流上下文 + 进度 + 动作时间线

## 5. 推荐配置

下面只保留工作流接入真正需要关心的配置。

### 5.1 最小配置示例

```yaml
workflow:
  flowable:
    base_url: "http://127.0.0.1:8087/flowable-rest/service"
    username: "rest-admin"
    password: "PLEASE_CHANGE"
    timeout_seconds: 15
    group_prefix: ""
    role_prefix: "role_"
    summary_cache_ttl_seconds: 120
    action_timeline_cache_ttl_seconds: 60
    action_timeline_cache_prefix: "workflow:process-action-timeline:"
  api:
    enabled: true
    prefix: "/api/v1/workflow"
    sso_enabled: true
    callback_path: "/flowable/callback"
    callback_secret: ""
  context:
    user_name_header: "X-Workflow-User-Name"
    system_code_header: "X-System-Code"
    groups_header: "X-Workflow-Groups"
    roles_header: "X-Workflow-Roles"
  identity:
    role_aliases: {}
    group_aliases: {}
  directory:
    provider: "ldap"
  assignment:
    provider: "directory"
    variable_keys:
      assignee: "nextAssignee"
      candidate_users: "nextCandidateUsers"
      candidate_groups: "nextCandidateGroups"
  formref:
    db_instance: "flowable_meta"
  records:
    db_instance: "flowable_meta"
```

### 5.2 配置项含义

#### `workflow.flowable.*`

- `base_url`
  - Flowable REST 根地址。
  - 示例：`http://127.0.0.1:8087/flowable-rest/service`
- `username`
  - Flowable REST 服务账号。
- `password`
  - Flowable REST 服务密码。
- `timeout_seconds`
  - 调用 Flowable REST 的超时时间。
- `group_prefix`
  - 候选组前缀。
- `role_prefix`
  - 角色型候选组前缀。
- `summary_cache_ttl_seconds`
  - 流程摘要缓存秒数。
- `action_timeline_cache_ttl_seconds`
  - 动作时间线缓存秒数。
- `action_timeline_cache_prefix`
  - 动作时间线缓存前缀。

#### `workflow.api.*`

- `enabled`
  - 是否注册标准工作流接口。
- `prefix`
  - 工作流接口前缀，正式环境推荐统一为 `/api/v1/workflow`。
- `sso_enabled`
  - 是否要求登录态。
- `callback_path`
  - Flowable 回调接收路径。
- `callback_secret`
  - 回调签名密钥。

#### `workflow.context.*`

- `user_name_header`
  - 读取当前用户显示名的 Header。
- `system_code_header`
  - 读取系统编码的 Header。
- `groups_header`
  - 读取用户组列表的 Header。
- `roles_header`
  - 读取角色列表的 Header。

#### `workflow.identity.*`

- `role_aliases`
  - 用于把 LDAP 角色、JWT 角色、业务角色统一归一到 BPMN 使用的角色码。
- `group_aliases`
  - 用于统一非角色型组编码。

#### `workflow.directory.*`

- `provider`
  - 目录服务来源。
  - 常见值：`none`、`ldap`

#### `workflow.assignment.*`

- `provider`
  - 自动分配来源。
  - 常见值：`none`、`directory`、`ldap`
- `variable_keys.assignee`
  - 标准下一办理人变量名，推荐 `nextAssignee`
- `variable_keys.candidate_users`
  - 标准下一候选用户变量名，推荐 `nextCandidateUsers`
- `variable_keys.candidate_groups`
  - 标准下一候选组变量名，推荐 `nextCandidateGroups`

#### `workflow.formref.*`

- `db_instance`
  - 用于解析表单引用的数据库实例名。

#### `workflow.records.*`

- `db_instance`
  - 用于保存 `workflow_task_records` 的数据库实例名。
  - 不配置时默认走 `orm.DB`。
  - 如果希望流程办理记录与业务库分离，可以单独指定一个工作流元数据库实例。

### 5.3 业务程序代码如何接入

只配 `config.yaml` 还不够，业务后端还必须在代码里把工作流标准路由注册进自己的 HTTP 服务。

最常见的接入方式就是：

1. 初始化你自己的业务系统配置
2. 初始化登录、RBAC、数据库、Redis 等基础能力
3. 创建业务系统自己的 `HTTPServer`
4. 先注册你自己的业务接口
5. 再调用 `workflowapi.MustRegisterFromConfig(server)` 注册工作流标准接口
6. 最后执行 `server.Prepare()` 和 `server.Run(...)`

#### 最小后端接入示例

```go
package main

import (
    commonhttp "github.com/goodbye-jack/go-common/http"
    rbacbase "github.com/goodbye-jack/go-common/rbac"
    workflowapi "github.com/goodbye-jack/go-common/workflow/api"
)

func main() {
    rbacbase.InitRoleMapping([]string{
        "ADMIN_ROLE",
        "DEFAULT_ROLE",
    })

    server := commonhttp.NewHTTPServer("demo-service")

    // 这里注册你自己的业务接口
    registerBizRoutes(server)

    // 这里注册 go-common 提供的标准工作流接口
    workflowapi.MustRegisterFromConfig(server)

    server.Prepare()
    server.Run(":9080")
}
```

#### 这段代码实际做了什么

- `workflowapi.MustRegisterFromConfig(server)`
  - 按 `config.yaml` 里的 `workflow.*` 配置，自动创建目录服务、分配服务、Flowable REST 客户端、表单引用服务
  - 自动把标准工作流接口挂到业务系统自己的服务里
  - 配置有问题时直接启动失败，避免系统“带病启动”
- `server.Prepare()`
  - 让 go-common 完成路由整理、鉴权元数据准备、权限策略落库等准备动作
- `server.Run(...)`
  - 正式对外提供业务接口和工作流接口

#### 什么时候调用最合适

推荐放在“业务路由注册完成之后、`server.Prepare()` 之前”。

推荐顺序如下：

```go
server := commonhttp.NewHTTPServer(serviceName)

// 1. 注册业务接口
biz.RouteAPI(server)

// 2. 注册工作流标准接口
workflowapi.MustRegisterFromConfig(server)

// 3. 准备并启动
server.Prepare()
server.Run(addr)
```

不推荐下面这种顺序：

- `server.Run()` 之后再注册
- 没有调用 `server.Prepare()`
- 业务接口还没注册完就中途启动

#### 如果要自定义目录服务或分配服务怎么办

默认推荐直接用：

```go
workflowapi.MustRegisterFromConfig(server)
```

只有在下面这些场景，才需要改用带 `options` 的注册方式：

- 你不想使用配置文件里的默认 LDAP 目录服务
- 你要注入自定义 `AssignmentService`
- 你要接入自定义表单引用解析服务
- 你要覆盖默认契约校验策略

示例：

```go
workflowapi.MustRegisterFromConfigWithOptions(server, workflowapi.RegisterOptions{
    DirectoryService:  myDirectoryService,
    AssignmentService: myAssignmentService,
    FormRefService:    myFormRefService,
    ContractPolicy:    myContractPolicy,
})
```

#### 业务后端除了注册工作流接口，还必须自己实现什么

工作流标准接口不替代你的业务接口。业务系统至少还要自己提供下面几类接口：

- 业务保存接口
  - 例如：`POST /api/v1/relic-preservation-alarms`
  - 负责写业务表、生成 `bizId`
- 业务详情接口
  - 例如：`GET /api/v1/relic-preservation-alarms/{bizId}`
  - 负责返回表单详情、附件、状态等业务数据
- 业务列表接口
  - 例如：草稿列表、我的申请列表、业务查询列表
- 业务状态更新逻辑
  - 例如：流程启动成功后，把业务单状态更新为“审批中”
  - 例如：收到回调后，把业务单状态更新为“已完成”“已驳回”“待补正”

要点只有一句话：

- `go-common/workflow` 负责流程能力
- 业务系统自己负责业务数据

### 5.4 前后端怎么配合使用

标准用法不是“前端只调一个万能业务接口”，而是“业务保存”和“流程启动”分开。

#### 标准发起顺序

1. 前端填写业务表单
2. 前端调用业务保存接口
3. 业务后端保存业务表，返回 `bizId`
4. 前端或业务后端调用 `POST /api/v1/workflow/process/start`
5. 启动成功后，业务系统记录 `bizId` 与 `processInstanceId` 的关系
6. 后续待办、已办、进度、办理动作都走标准工作流接口
7. 业务详情、附件、业务状态展示继续走业务系统自己的接口

#### 推荐的前端调用拆分

- 发起页
  - 先调业务保存接口
  - 再调工作流启动接口
- 我的待办页
  - 调 `GET /api/v1/workflow/me/tasks/todo`
- 我的已办页
  - 调 `GET /api/v1/workflow/me/tasks/done`
- 任务处理页
  - 调业务详情接口
  - 调 `GET /api/v1/workflow/tasks/{id}/context`
  - 调 `GET /api/v1/workflow/tasks/{id}/form-ref`
  - 调 `GET /api/v1/workflow/biz/{bizId}/progress-timeline`
  - 调 `GET /api/v1/workflow/biz/{bizId}/task-records`
- 提交处理
  - 根据动作调用 `claim / unclaim / delegate / resolve / transfer / complete`

#### 一个最小的发起流程示例

前端或网关层发起时，推荐拆成两次 HTTP 调用。

第一步，保存业务：

```http
POST /api/v1/relic-preservation-alarms
Content-Type: application/json

{
  "title": "文保业务告警测试",
  "alarmLevel": "YELLOW",
  "alarmType": "SECURITY"
}
```

假设返回：

```json
{
  "data": {
    "bizId": "ALARM-20260611-0001"
  },
  "message": "success"
}
```

第二步，启动流程：

```http
POST /api/v1/workflow/process/start
Content-Type: application/json

{
  "processDefinitionKey": "relic_preservation_alarm_test",
  "bizId": "ALARM-20260611-0001",
  "bizType": "relic_preservation_alarm",
  "title": "文保业务告警测试",
  "name": "文保业务告警测试",
  "variables": {
    "starterId": "test1",
    "starterName": "测试用户1",
    "nextCandidateGroups": ["role_LEGACY_ROLE_17"],
    "callbackUrl": "http://127.0.0.1:9080/flowable/callback",
    "callbackEvents": "PROCESS_STARTED,NODE_STARTED,NODE_ENDED,PROCESS_ENDED"
  }
}
```

这样做的好处是：

- 业务表和流程实例职责清晰
- 任何一个环节失败都更容易补偿
- 开发人员更容易理解问题出在业务接口还是流程接口
- 不会把“保存业务”和“启动流程”硬耦合成一个黑盒接口

### 5.5 `flowable-ui`、`flowable-rest`、业务程序分别负责什么

很多人接入时会把这三层混在一起。这里直接拆开说明。

#### `flowable-ui` 负责什么

- 画流程图
- 配节点属性
- 配用户任务、候选人、候选组
- 配表单模型
- 发布流程定义
- 查看流程模型和管理后台信息

一句话概括：

- `flowable-ui` 是“流程设计和发布平台”

#### `flowable-rest` 负责什么

- 接收 go-common 发起的 REST 调用
- 启动流程实例
- 查询运行中任务
- 查询历史任务和历史流程
- 执行认领、委派、转办、完成等运行态动作

一句话概括：

- `flowable-rest` 是“流程引擎运行时服务”

#### 业务程序负责什么

- 保存业务数据
- 返回业务详情
- 注册 `go-common/workflow` 标准接口
- 提供业务页面
- 处理业务状态同步
- 决定哪些业务字段映射为流程变量

一句话概括：

- 业务程序是“真正对外提供业务功能的系统”

#### 最容易搞错的一点

业务系统不是直接接 `flowable-ui`。

正确关系是：

1. 流程设计人员在 `flowable-ui` 里画图并发布
2. 业务系统通过 `go-common/workflow` 调用 `flowable-rest`
3. 前端页面调用的是“业务系统接口”和“业务系统里暴露出来的标准工作流接口”

也就是说：

- 前端不直接调 `flowable-ui`
- 前端通常也不直接调 `flowable-rest`
- 前端应该调业务系统暴露出来的统一接口

## 8. 流程启动接口

### 8.1 `POST /api/v1/workflow/process/start`

#### 作用

- 启动一个新的流程实例。

#### 使用场景

- 业务保存成功后启动流程
- 管理后台手工补发流程
- 联调 BPMN 时直接验证流程定义 key 是否可用

#### 路径参数

- 无

#### 请求体字段

- `processDefinitionId`
  - 流程定义 ID
  - 与 `processDefinitionKey` 二选一
- `processDefinitionKey`
  - 流程定义 Key
  - 正式接入时更推荐使用这个
- `businessKey`
  - Flowable 业务键
  - 不传时一般可与 `bizId` 保持一致
- `bizId`
  - 业务主键
- `bizType`
  - 业务类型
- `title`
  - 业务标题
- `name`
  - Flowable 实例名称
- `variables`
  - 启动时写入流程实例的变量集合

#### 请求体示例

```json
{
  "processDefinitionKey": "asset_approval_process",
  "bizId": "BIZ-20260610-0001",
  "bizType": "asset_approval",
  "title": "资产审批申请示例",
  "name": "资产审批申请示例",
  "variables": {
    "starterId": "test1",
    "starterName": "测试用户1",
    "tenantId": "test-demo",
    "systemCode": "sysA",
    "nextAssignee": "test2",
    "callbackUrl": "http://127.0.0.1:9080/flowable/callback"
  }
}
```

#### 返回字段

- `processInstanceId`
  - 流程实例 ID
- `processDefinitionId`
  - 实际启动的流程定义 ID
- `businessKey`
  - Flowable 业务键
- `tenantId`
  - 租户编码

#### 请求示例

```bash
curl -X POST http://127.0.0.1:9080/api/v1/workflow/process/start \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <token>" \
  -d '{
    "processDefinitionKey":"asset_approval_process",
    "bizId":"BIZ-20260610-0001",
    "bizType":"asset_approval",
    "title":"资产审批申请示例",
    "variables":{"starterId":"test1"}
  }'
```

## 11. 任务动作接口

### 11.1 职责边界

这一节非常重要，开发、测试、流程设计人员必须统一理解。

`go-common/workflow` 当前把任务相关接口明确拆成两类职责：

第一类，任务办理权管理接口：

- `claim`
- `unclaim`
- `delegate`
- `resolve`
- `transfer`

这 5 个接口只负责处理“这件任务现在由谁来办”。

它们解决的是：

- 候选任务是否已经被认领
- 当前办理人是否要把任务放回候选池
- 当前办理人是否要临时委派给别人
- 受托人处理完成后是否归还给原办理人
- 当前任务是否要彻底转交给另一个办理人

它们不解决的是：

- 流程是否流转到下一节点
- BPMN 网关走哪条线
- 退回、驳回、重审、补正应该走哪个分支

第二类，流程推进接口：

- `complete`

`complete` 只负责处理“流程往哪走”。

它解决的是：

- 当前节点处理完成
- 本次处理结果写入流程变量
- Flowable 根据变量和 BPMN 条件表达式决定下一节点

它不负责的是：

- 抢占候选任务
- 释放任务
- 委派
- 转办

可以这样理解：

- `claim / unclaim / delegate / resolve / transfer`
  - 管的是“谁来办”
- `complete`
  - 管的是“办完以后流程往哪走”

如果只记一句话，请记住：

- 想改办理人，用任务动作接口。
- 想推进流程，用 `complete`。

## 12. 进度与动作时间线接口

### 12.1 `GET /api/v1/workflow/process-instances/{id}/progress-view`

#### 作用

- 返回流程摘要、BPMN 结构化步骤视图、当前节点总览。

#### 使用场景

- 顶部流程摘要
- 流程图高亮
- 当前节点总览
- 结构化步骤分析

#### 路径参数

- `id`
  - 流程实例 ID

#### 返回字段重点

- `summary`
  - 其中：
    - `summary.startUserId` 表示流程发起人 ID
    - `summary.startUserName` 表示流程发起人显示名
- `steps`
  - 结构顺序仍然按 BPMN 正向步骤展示，不会被并行分支的时间先后打乱
- `completedActivityIds`
- `currentActivityIds`

#### `steps[]` 字段重点

- `steps[].activityId`
  - BPMN 节点 ID
- `steps[].activityName`
  - BPMN 节点名称
- `steps[].status`
  - 当前步骤状态
- `steps[].executions`
  - 该步骤真实发生过的办理执行记录列表

#### `steps[].executions[]` 字段重点

- `taskId`
  - 该次执行对应的任务 ID
- `status`
  - 该次执行当前状态，常见值：`ACTIVE`、`COMPLETED`
- `assignee`
  - 当前或最终办理人
- `owner`
  - 任务 owner
- `candidateUsers`
  - 候选用户
- `candidateGroups`
  - 候选组
- `records`
  - 该次执行下真实发生过的办理记录

#### `steps[].executions[].records[]` 字段重点

- `actionType`
  - 办理动作类型
- `operatorUserId`
  - 动作发起人
- `fromAssignee`
  - 变更前办理人
- `toAssignee`
  - 变更后办理人
- `time`
  - 动作时间

#### 说明

- 这个接口的核心定位是“结构视图”，不是“用户最终看到的审批进度主列表”。
- `steps[]` 顺序按 BPMN 结构展开，因此可能包含：
  - 未来节点
  - 回退节点
  - 尚未走到的分支节点
- 所以前端如果要做“审批进度弹窗 / 审批进度页”，不要直接把 `steps[]` 原样当主列表渲染。
- 推荐用法：
  - 用 `summary` 做顶部摘要
  - 用 `completedActivityIds/currentActivityIds` 做流程图高亮
  - 如果要做“结构分析页”，再使用 `steps[]`

#### 请求示例

```bash
curl -H "Authorization: Bearer <token>" \
  http://127.0.0.1:9080/api/v1/workflow/process-instances/process-001/progress-view
```

- `items[].assignee`

## 14. 回调接口

### 14.1 3 分钟看懂：`POST /flowable/callback`

这一节已经整合了《Flowable Callback 接口 3 分钟速读版》的核心内容。
如果你现在只想快速弄明白这个接口怎么配、谁来调、收到后做什么，直接看完这一节即可。

#### Q1. 我需要前端调用这个接口吗？

不需要。

直接结论：

1. `POST /flowable/callback` 不是前端调的。
2. 它也不是业务页面按钮直接调的。
3. 它通常是 Flowable 在流程运行过程中自动回调到业务系统的一个内部接口。
4. 你要让它生效，关键不是“手动调这个接口”，而是“启动流程时把 `callbackUrl` 和 `callbackEvents` 传给 Flowable”。
5. 当前 `go-common` 标准实现收到回调后，只会刷新工作流侧读模型，不会自动更新你的业务主表。
6. 如果你想在“节点完成后同步改业务表状态”，要基于这个回调事件做业务扩展处理。

#### Q2. 那到底是谁来调用它？

- 正常调用方：Flowable
- 非正常调用方：前端页面、业务按钮、人工日常操作

也就是说，正常链路是：

1. 业务系统调用 `POST /api/v1/workflow/process/start`
2. 请求里通过 `variables.callbackUrl` 告诉 Flowable：“后面事件发生时，请回调这个地址”
3. Flowable 在流程运行过程中，自动 `POST /flowable/callback`
4. `go-common` 收到后刷新待办、已办、流程摘要等工作流侧数据

#### Q3. 它什么时候会被调用？

最常见是 4 类事件：

- `PROCESS_STARTED`
  - 流程刚启动
- `NODE_STARTED`
  - 某个节点刚进入运行态
- `NODE_ENDED`
  - 某个节点刚处理完成
- `PROCESS_ENDED`
  - 整个流程结束

如果你在启动流程时没有传 `callbackUrl` 或没有声明 `callbackEvents`，那就不会有这类自动回调。

#### Q4. 我真正应该怎么接这个接口？

真正的使用方式不是“去写一个页面点这个接口”，而是下面这条链路：

第 1 步，启动流程时配置回调地址和事件：

```json
{
  "processDefinitionKey": "asset_approval_process",
  "bizId": "BIZ-20260610-0001",
  "bizType": "asset_approval",
  "title": "资产审批申请示例",
  "variables": {
    "callbackUrl": "http://127.0.0.1:9080/flowable/callback",
    "callbackEvents": "PROCESS_STARTED,NODE_STARTED,NODE_ENDED,PROCESS_ENDED"
  }
}
```

第 2 步，Flowable 在事件发生时自动回调：

```http
POST /flowable/callback
Content-Type: application/json
```

```json
{
  "eventId": "evt-001",
  "eventType": "NODE_ENDED",
  "eventTime": "2026-06-10T10:00:00+08:00",
  "processInstanceId": "process-001",
  "processDefinitionId": "asset_approval_process:3:abc",
  "activityId": "task_review",
  "taskId": "task-001",
  "bizId": "BIZ-20260610-0001",
  "variables": {
    "bizType": "asset_approval",
    "title": "资产审批申请示例",
    "result": "APPROVED"
  }
}
```

第 3 步，`go-common` 返回成功：

```json
{
  "data": {
    "accepted": true
  },
  "message": "success"
}
```

这就是 `flowable/callback` 的标准使用方式。

#### Q5. 启动流程时到底要传什么？

你真正需要关心的是启动流程时下面两个变量：

- `callbackUrl`
  - 含义：Flowable 以后要把事件回调到哪个地址
  - 示例：`http://127.0.0.1:9080/flowable/callback`
- `callbackEvents`
  - 含义：哪些事件发生时要回调
  - 示例：`PROCESS_STARTED,NODE_STARTED,NODE_ENDED,PROCESS_ENDED`

最推荐的启动示例：

```json
{
  "processDefinitionKey": "asset_approval_process",
  "bizId": "BIZ-20260610-0001",
  "bizType": "asset_approval",
  "title": "资产审批申请示例",
  "name": "资产审批申请示例",
  "variables": {
    "starterId": "test1",
    "starterName": "测试用户1",
    "callbackUrl": "http://127.0.0.1:9080/flowable/callback",
    "callbackEvents": "PROCESS_STARTED,NODE_STARTED,NODE_ENDED,PROCESS_ENDED"
  }
}
```

如果你现在问“我到底该不该配回调”，可以直接这样判断：

- 只想用待办、已办、流程进度，并希望这些信息实时刷新：建议配
- 还想在节点完成后联动更新业务表状态：必须配
- 只是做极简演示，不关心回调和状态同步：可以先不配

#### Q6. 为什么我没有收到回调？

先按下面顺序排查：

1. 启动流程时有没有传 `variables.callbackUrl`
2. 启动流程时有没有传 `variables.callbackEvents`
3. `callbackUrl` 地址是否能被 Flowable 所在环境访问到
4. 你订阅的事件里是否包含当前发生的事件
5. 如果配置了 `callback_secret`，调用方是否按要求带了签名头

最常见的原因就两个：

- 根本没配 `callbackUrl / callbackEvents`
- 地址配置了，但 Flowable 根本打不到这个地址

#### Q7. 请求头要不要签名？

默认情况下：

- `Content-Type: application/json`

如果你配置了 `workflow.api.callback_secret`，还必须带下面 3 个头：

- `X-Timestamp`
  - 当前时间戳
- `X-Nonce`
  - 随机串
- `X-Signature`
  - 签名结果

签名规则是：

```text
timestamp + "\n" + nonce + "\n" + body
```

签名算法是：

```text
HMAC-SHA256
```

也就是说：

- 没配 `callback_secret`
  - 不验签
- 配了 `callback_secret`
  - 少任何一个签名头都会返回 `401 Unauthorized`

#### Q8. 请求体每个字段到底是什么意思？

- `eventId`
  - 本次回调事件 ID
  - 建议业务扩展侧拿它做幂等去重
- `eventType`
  - 本次事件类型
  - 最常见就是 `PROCESS_STARTED`、`NODE_STARTED`、`NODE_ENDED`、`PROCESS_ENDED`
- `eventTime`
  - 本次事件发生时间
- `processInstanceId`
  - 流程实例 ID
  - 查流程进度、查流程图、更新业务流程状态时很常用
- `processDefinitionId`
  - 流程定义 ID
  - 主要用于调试、区分具体部署版本
- `activityId`
  - 当前节点 ID
  - 例如：`task_review`
  - 业务扩展时经常用它判断“是哪个节点结束了”
- `taskId`
  - 当前任务 ID
  - 对任务级事件最有用
- `bizId`
  - 你的业务主键
  - 回写业务表最关键的关联字段之一
- `variables`
  - 当时的流程变量快照
  - 里面通常会带标题、结果、发起人、下一节点变量等

#### Q9. 返回 `accepted=true` 到底代表什么？

返回示例：

```json
{
  "data": {
    "accepted": true
  },
  "message": "success"
}
```

字段含义：

- `accepted=true`
  - 表示这个回调请求已经被当前标准处理链路接收
  - 包含：验签通过、JSON 解析通过、标准回调处理通过

这个返回值的重点不是给前端展示，而是告诉回调方：“这条事件我收到了”。

#### Q10. 当前 `go-common` 收到回调后到底做了什么？

当前标准实现只做工作流平台侧的同步：

- `PROCESS_STARTED`
  - 刷新流程摘要
- `NODE_STARTED`
  - 同步运行中待办
  - 刷新流程摘要
- `NODE_ENDED`
  - 同步已办
  - 删除运行中待办
  - 刷新流程摘要
- `PROCESS_ENDED`
  - 清理流程实例下的运行中待办
  - 刷新流程摘要

一句话：

- 它主要解决“待办、已办、流程进度要及时更新”

#### Q11. 为什么回调成功了，但业务表状态没变？

当前标准实现不会自动帮你做下面这些业务动作：

- 不会自动更新业务主表状态
- 不会自动更新业务审核字段
- 不会自动插入业务审核日志
- 不会自动发短信、发站内信、发邮件
- 不会自动做你的行业逻辑补偿

因为当前标准实现不会自动改你的业务主表。

所以千万不要误解成：

- “我把 `callbackUrl` 配上，业务表状态就会自动变”

当前标准实现不是这个语义。

#### Q12. 那它支不支持“节点完成后更新业务表字段”？

支持，但这是“基于回调事件做业务扩展”，不是“平台已经默认帮你写好”。

最典型的扩展方式如下：

- 收到 `PROCESS_STARTED`
  - 把 `processInstanceId` 回写到你的业务主表
  - 记录 `workflow_started_at`
- 收到 `NODE_STARTED`
  - 更新当前处理节点编码
  - 更新当前处理节点名称
- 收到 `NODE_ENDED`
  - 更新本节点处理结果
  - 记录最近处理时间
  - 追加业务审核日志
- 收到 `PROCESS_ENDED`
  - 更新业务流程总状态
  - 记录流程完成时间

#### Q13. 我应该在哪一层写业务状态更新？

应该写在业务系统自己的事件处理层，不应该误以为写在前端，也不应该误以为 `go-common` 默认已经帮你写好了。

推荐分层：

1. `go-common` 标准回调接收层
   - 负责验签
   - 负责解析回调请求
   - 负责刷新工作流侧读模型
2. 业务系统事件处理层
   - 负责更新业务主表
   - 负责写业务日志
   - 负责通知、归档、统计等扩展逻辑

一句话：

- 工作流平台负责“事件到了”
- 业务系统负责“业务字段怎么变”

#### Q14. 给我一个最实际的业务扩展示例

假设你的业务表叫 `biz_order`。

收到 `NODE_ENDED` 时，可以这样理解：

```text
根据 bizId 找到业务单
根据 activityId 判断是哪一个节点完成
根据 variables.result 更新本次处理结果
根据 eventTime 更新最后处理时间
插入一条业务审核日志
```

可能对应的业务字段语义是：

- `biz_order.current_node_code = 'task_review'`
- `biz_order.current_node_status = 'DONE'`
- `biz_order.last_result = 'APPROVED'`
- `biz_order.last_operated_at = eventTime`

收到 `PROCESS_ENDED` 时，可以这样理解：

- `biz_order.workflow_status = 'COMPLETED'`
- `biz_order.completed_at = eventTime`

#### Q15. 回调重复推送怎么办？

因为回调有可能重复投递，所以建议至少这样处理：

1. 优先用 `eventId` 做幂等键
2. 如果上游没有稳定 `eventId`，就组合：
   - `processInstanceId`
   - `taskId`
   - `activityId`
   - `eventType`
   - `eventTime`
3. 业务日志表最好建唯一索引，避免重复写入

#### Q16. 联调时我能不能手工测？

正式环境不应该手工调用，但联调时可以用 `curl` 模拟一条回调：

```bash
curl -X POST http://127.0.0.1:9080/flowable/callback \
  -H "Content-Type: application/json" \
  -d '{
    "eventId":"evt-001",
    "eventType":"NODE_ENDED",
    "eventTime":"2026-06-10T10:00:00+08:00",
    "processInstanceId":"process-001",
    "processDefinitionId":"asset_approval_process:3:abc",
    "activityId":"task_review",
    "taskId":"task-001",
    "bizId":"BIZ-20260610-0001",
    "variables":{
      "bizType":"asset_approval",
      "result":"APPROVED"
    }
  }'
```

如果配置了 `callback_secret`，还要额外补签名头，否则会直接返回 `401 Unauthorized`。

#### Q17. 最后一句话怎么记？

- `flowable/callback` 不是“给页面调用的业务接口”。
- 它是“Flowable 自动回调过来的内部事件接口”。
- 当前标准能力负责刷新工作流侧数据。
- 业务表要不要跟着更新，要由业务系统基于回调事件自己扩展。

## 15. 前端推荐页面结构

前端推荐至少拆成下面几类页面：

1. 发起页
2. 我的申请页
3. 我的待办页
4. 我的已办页
5. 业务详情页
6. 任务处理页
7. 流程进度页

### 15.1 处理页建议分三块

1. 业务详情
2. 任务管理动作
   - 认领
   - 释放
   - 委派
   - 转办
   - 解决委派
3. 业务处理动作
   - 通过
   - 退回
   - 驳回
   - 补正

### 15.2 时间线建议分两块

1. 主进度链路
   - `progress-timeline`
   - 用来展示真正走过或当前正在处理的步骤
2. 节点补充动作
   - `task-records`
   - 用来展示某一步里发生的转办、委派、解决委派、完成等动作
3. 动作审计页
   - `action-timeline`
   - 只在需要单独审计“办理权如何变化”时使用

### 15.3 审批进度页推荐组合

推荐组合：

1. 顶部摘要和流程图
   - `progress-view`
2. 主步骤列表
   - `progress-timeline`
3. 每个步骤下的动作补充
   - `task-records`

不要这样组合：

1. 直接把 `progress-view.steps[]` 当审批进度主列表
2. 直接把 `action-timeline` 当审批进度主列表

推荐展示逻辑：

1. 主列表一行只表示一个真实步骤
2. 该步骤的处理人、创建时间、完成时间、状态来自 `progress-timeline.items[]`
3. 该步骤下的转办 / 委派 / 解决委派 / 完成意见来自 `task-records`
4. `START_PROCESS` 不建议重复挂入第一个步骤的动作列表

前端伪代码示例：

```ts
const [progressView, progressTimeline, taskRecords] = await Promise.all([
  getBizProgressView(bizId),
  getBizProgressTimeline(bizId),
  getBizTaskRecords(bizId),
]);

const stepList = progressTimeline.items.map((step) => {
  const records = taskRecords.items.filter((record) => {
    if (record.actionType === "START_PROCESS") {
      return false;
    }
    if (step.taskId && record.taskId === step.taskId) {
      return true;
    }
    return !step.taskId && step.activityId && record.activityId === step.activityId;
  });

  return {
    step,
    records,
  };
});
```
