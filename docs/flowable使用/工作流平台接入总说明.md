# 工作流平台接入总说明

## 1. 文档定位

当前 `go-common/workflow` 对外只保留 3 份正式工作流文档：

1. `工作流平台接入总说明.md`
   - 面向后端、前端、测试、实施
   - 负责说明架构、配置、标准接口、调用顺序、字段语义
2. `工作流BPMN建模与变量规范.md`
   - 面向流程设计人员、平台架构人员
   - 负责说明 BPMN 如何建模、变量如何命名、哪些写法允许、哪些写法禁止
3. `Flowable UI与REST本地启动说明.md`
   - 面向本地联调和环境运维人员
   - 负责说明 UI/REST/数据库如何启动、如何验证、如何切换 LDAP

除这 3 份外，其余旧的工作流说明、发布通知、版本说明、模板类文档都不再作为正式接入依据。

---

## 2. 先说结论

业务系统接入工作流时，只需要记住下面 8 条：

1. 业务数据必须落自己的业务表，不能指望 Flowable 表单代替业务存储。
2. 发起流程必须分两步：
   - 先调业务保存接口
   - 再调工作流启动接口
3. `claim / unclaim / delegate / resolve / transfer` 只负责“谁来办”。
4. `complete` 只负责“流程往哪走”。
5. 退回、重审、驳回、补正必须通过 BPMN 网关和变量表达，不新增“任意跳转”接口。
6. BPMN 中不要直接写业务私有变量如 `${customApproverId}` 作为长期平台契约，统一用标准变量。
7. 流程轨迹现在分三类：
   - `progress-view` 看流程摘要、BPMN 结构、当前节点总览
   - `progress-timeline` 看真正走过或当前正在处理的主链路步骤
   - `task-records` 看每个步骤下发生过哪些办理动作
   - `action-timeline` 只看办理权动作审计，不建议直接拿来做主进度页
8. “我的已办”和“我的办理记录”不是一回事：
   - `done` 只看真正完成过的任务
   - `task-records` 看认领、委派、转办、完成等实际办理动作

## 3. 总体架构与职责边界

推荐架构固定为三层：

1. `go-common/workflow`
   - 标准流程接口
   - 目录能力
   - 身份归一
   - 自动分配能力
   - Flowable REST 适配
2. `flowable-ui + flowable-rest`
   - `flowable-ui` 负责流程图、表单模型、部署发布、权限管理
   - `flowable-rest` 负责流程运行态和历史态
3. 业务系统
   - 业务表
   - 业务接口
   - 业务页面
   - 业务状态
   - 业务字段校验

### 3.1 `go-common/workflow` 负责什么

- 当前用户工作流上下文解析
- 启动流程
- 我的待办
- 我的已办
- 任务上下文
- 任务动作
- 流程进度视图
- 流程进度时间线
- 流程动作时间线
- 流程办理记录
- 我的办理记录
- 流程图视图
- 目录查询
- 自动分配

### 3.2 `go-common/workflow` 不负责什么

- 业务主表设计
- 业务详情组装
- 业务表单渲染
- 业务状态机
- 业务权限模型
- 行业专属字段语义

### 3.3 业务系统必须自己负责什么

- 业务保存接口
- 业务详情接口
- 业务列表接口
- 业务字段校验
- 附件管理
- 业务状态流转
- 流程变量与业务字段映射

## 4. 标准接入顺序

标准调用链路如下：

1. 用户登录，业务系统在请求上下文里放入：
   - `userId`
   - `userName`
   - `tenantId`
   - `systemCode`
   - `groups`
   - `roles`
2. 前端点击“提交申请”。
3. 前端先调用业务保存接口。
4. 业务后端写自己的业务表，生成 `bizId`。
5. 前端或业务后端调用 `POST /api/v1/workflow/process/start`。
6. 工作流返回 `processInstanceId`。
7. 业务后端把 `bizId` 与 `processInstanceId` 绑定。
8. 后续：
   - 我的待办、我的已办走工作流标准接口
   - 业务详情走业务系统自己的详情接口
   - 处理页同时加载业务详情 + 工作流上下文 + 进度 + 动作时间线

## 5. 推荐配置

下面只保留工作流接入真正需要关心的配置。

### 5.1 最小配置示例

```yaml
workflow:
  flowable:
    base_url: "http://127.0.0.1:8087/flowable-rest/service"
    username: "rest-admin"
    password: "PLEASE_CHANGE"
    timeout_seconds: 15
    group_prefix: ""
    role_prefix: "role_"
    summary_cache_ttl_seconds: 120
    action_timeline_cache_ttl_seconds: 60
    action_timeline_cache_prefix: "workflow:process-action-timeline:"
  api:
    enabled: true
    prefix: "/api/v1/workflow"
    sso_enabled: true
    callback_path: "/flowable/callback"
    callback_secret: ""
  context:
    user_name_header: "X-Workflow-User-Name"
    system_code_header: "X-System-Code"
    groups_header: "X-Workflow-Groups"
    roles_header: "X-Workflow-Roles"
  identity:
    role_aliases: {}
    group_aliases: {}
  directory:
    provider: "ldap"
  assignment:
    provider: "directory"
    variable_keys:
      assignee: "nextAssignee"
      candidate_users: "nextCandidateUsers"
      candidate_groups: "nextCandidateGroups"
  formref:
    db_instance: "flowable_meta"
  records:
    db_instance: "flowable_meta"
```

### 5.2 配置项含义

#### `workflow.flowable.*`

- `base_url`
  - Flowable REST 根地址。
  - 示例：`http://127.0.0.1:8087/flowable-rest/service`
- `username`
  - Flowable REST 服务账号。
- `password`
  - Flowable REST 服务密码。
- `timeout_seconds`
  - 调用 Flowable REST 的超时时间。
- `group_prefix`
  - 候选组前缀。
- `role_prefix`
  - 角色型候选组前缀。
- `summary_cache_ttl_seconds`
  - 流程摘要缓存秒数。
- `action_timeline_cache_ttl_seconds`
  - 动作时间线缓存秒数。
- `action_timeline_cache_prefix`
  - 动作时间线缓存前缀。

#### `workflow.api.*`

- `enabled`
  - 是否注册标准工作流接口。
- `prefix`
  - 工作流接口前缀，正式环境推荐统一为 `/api/v1/workflow`。
- `sso_enabled`
  - 是否要求登录态。
- `callback_path`
  - Flowable 回调接收路径。
- `callback_secret`
  - 回调签名密钥。

#### `workflow.context.*`

- `user_name_header`
  - 读取当前用户显示名的 Header。
- `system_code_header`
  - 读取系统编码的 Header。
- `groups_header`
  - 读取用户组列表的 Header。
- `roles_header`
  - 读取角色列表的 Header。

#### `workflow.identity.*`

- `role_aliases`
  - 用于把 LDAP 角色、JWT 角色、业务角色统一归一到 BPMN 使用的角色码。
- `group_aliases`
  - 用于统一非角色型组编码。

#### `workflow.directory.*`

- `provider`
  - 目录服务来源。
  - 常见值：`none`、`ldap`

#### `workflow.assignment.*`

- `provider`
  - 自动分配来源。
  - 常见值：`none`、`directory`、`ldap`
- `variable_keys.assignee`
  - 标准下一办理人变量名，推荐 `nextAssignee`
- `variable_keys.candidate_users`
  - 标准下一候选用户变量名，推荐 `nextCandidateUsers`
- `variable_keys.candidate_groups`
  - 标准下一候选组变量名，推荐 `nextCandidateGroups`

#### `workflow.formref.*`

- `db_instance`
  - 用于解析表单引用的数据库实例名。

#### `workflow.records.*`

- `db_instance`
  - 用于保存 `workflow_task_records` 的数据库实例名。
  - 不配置时默认走 `orm.DB`。
  - 如果希望流程办理记录与业务库分离，可以单独指定一个工作流元数据库实例。

## 6. 标准接口总览

正式对外接口分为 6 组：

1. 当前用户与目录接口
2. 流程启动接口
3. 待办已办接口
4. 任务处理接口
5. 进度与动作时间线接口
6. 流程图与回调接口

以下说明默认前缀为：

```text
/api/v1/workflow
```

## 7. 当前用户与目录接口

### 7.1 `GET /api/v1/workflow/me`

#### 作用

- 返回当前登录人的工作流上下文和目录信息。

#### 使用场景

- 前端初始化当前用户信息
- 发起页、工作台页显示当前登录人
- 调试当前用户身份是否已正确映射到工作流用户

#### 参数

- 无路径参数
- 无查询参数
- 无请求体

#### 返回字段重点

- `userId`
  - 工作流用户 ID
- `userName`
  - 当前用户显示名
- `tenantId`
  - 租户编码
- `systemCode`
  - 系统编码
- `groups`
  - 当前用户组
- `roles`
  - 当前用户角色
- `directoryResolved`
  - 是否成功解析目录信息
- `directory`
  - 目录详情

#### 请求示例

```bash
curl -H "Authorization: Bearer <token>" \
  http://127.0.0.1:9080/api/v1/workflow/me
```

#### 参数

- 无路径参数
- 无查询参数

#### 返回字段重点

- `userId`
- `displayName`
- `email`
- `dn`

#### 请求示例

## 8. 流程启动接口

### 8.1 `POST /api/v1/workflow/process/start`

#### 作用

- 启动一个新的流程实例。

#### 使用场景

- 业务保存成功后启动流程
- 管理后台手工补发流程
- 联调 BPMN 时直接验证流程定义 key 是否可用

#### 路径参数

- 无

#### 请求体字段

- `processDefinitionId`
  - 流程定义 ID
  - 与 `processDefinitionKey` 二选一
- `processDefinitionKey`
  - 流程定义 Key
  - 正式接入时更推荐使用这个
- `businessKey`
  - Flowable 业务键
  - 不传时一般可与 `bizId` 保持一致
- `bizId`
  - 业务主键
- `bizType`
  - 业务类型
- `title`
  - 业务标题
- `name`
  - Flowable 实例名称
- `variables`
  - 启动时写入流程实例的变量集合

#### 请求体示例

```json
{
  "processDefinitionKey": "asset_approval_process",
  "bizId": "BIZ-20260610-0001",
  "bizType": "asset_approval",
  "title": "资产审批申请示例",
  "name": "资产审批申请示例",
  "variables": {
    "starterId": "test1",
    "starterName": "测试用户1",
    "tenantId": "test-demo",
    "systemCode": "sysA",
    "nextAssignee": "test2",
    "callbackUrl": "http://127.0.0.1:9080/flowable/callback"
  }
}
```

#### 返回字段

- `processInstanceId`
  - 流程实例 ID
- `processDefinitionId`
  - 实际启动的流程定义 ID
- `businessKey`
  - Flowable 业务键
- `tenantId`
  - 租户编码

#### 请求示例

```bash
curl -X POST http://127.0.0.1:9080/api/v1/workflow/process/start \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <token>" \
  -d '{
    "processDefinitionKey":"asset_approval_process",
    "bizId":"BIZ-20260610-0001",
    "bizType":"asset_approval",
    "title":"资产审批申请示例",
    "variables":{"starterId":"test1"}
  }'
```

## 9. 待办、已办与办理记录接口

### 9.1 公共查询参数

以下 3 个列表接口共用同一套基础查询参数：

- `start`
  - 分页起始下标
- `size`
  - 分页大小
- `includeProgress`
  - 是否附带轻量进度摘要
- `title`
  - 按标题模糊过滤
- `bizType`
  - 按业务类型过滤
- `processDefinitionKey`
  - 按流程定义 key 过滤
- `activityId`
  - 按当前或历史节点 ID 过滤
- `actionType`
  - 仅办理记录接口有意义
  - 可选值例如：`START_PROCESS`、`CLAIM`、`UNCLAIM`、`DELEGATE`、`RESOLVE`、`TRANSFER`、`COMPLETE`
- `status`
  - 状态过滤
- `createdAfter`
  - 创建时间起始
- `createdBefore`
  - 创建时间结束
- `completedAfter`
  - 完成时间起始，仅已办更有意义
- `completedBefore`
  - 完成时间结束，仅已办更有意义

### 9.2 `GET /api/v1/workflow/me/tasks/todo`

#### 作用

- 查询当前登录人的待办任务。

#### 使用场景

- 我的待办列表
- 工作台卡片
- 待办搜索页

#### 查询参数示例

start=0&size=20&includeProgress=true&bizType=asset_approval

- `items[].activityId`
- `items[].activityName`
- `items[].processInstanceId`
- `items[].rootProcessInstanceId`
- `items[].bizId`
- `items[].bizType`
- `items[].title`
- `items[].assignee`
- `items[].owner`
- `items[].delegationState`
- `items[].formKey`
- `items[].progress`
- `total`
- `start`
- `size`

#### 请求示例

```bash
curl -H "Authorization: Bearer <token>" \
  "http://127.0.0.1:9080/api/v1/workflow/me/tasks/todo?start=0&size=20&includeProgress=true"
```

### 9.3 `GET /api/v1/workflow/me/tasks/done`

#### 作用

- 查询当前登录人的已办任务。

#### 使用场景

- 我的已办列表
- 审批历史回看
- 工作量统计前置查询

#### 查询参数

- 与待办接口一致

#### 返回字段重点

- `items[].completedAt`
  - 任务完成时间
- 其余字段与待办列表保持同风格

#### 请求示例

```bash
curl -H "Authorization: Bearer <token>" \
  "http://127.0.0.1:9080/api/v1/workflow/me/tasks/done?start=0&size=20"
```

### 9.4 `GET /api/v1/workflow/me/task-records`

#### 作用

- 查询“我实际做过哪些办理动作”。

#### 和 `done` 的区别

- `done`
  - 只表示“我完成过的任务”
- `task-records`
  - 表示“我参与过的办理记录”
  - 包括：
    - `START_PROCESS`
    - `CLAIM`
    - `UNCLAIM`
    - `DELEGATE`
    - `RESOLVE`
    - `TRANSFER`
    - `COMPLETE`

#### 使用场景

- 查看我是否转办过某单
- 查看我是否委派过某单
- 查看我是否只是认领但未完成
- 做办理动作审计

#### 查询参数

- 与待办、已办共用基础查询参数
- 其中 `actionType` 在本接口最常用

#### 返回字段重点

- `items[].actionType`
- `items[].time`
- `items[].taskId`
- `items[].taskName`
- `items[].activityId`
- `items[].activityName`
- `items[].bizId`
- `items[].bizType`
- `items[].title`
- `items[].operatorUserId`
- `items[].operatorUserName`
- `items[].fromAssignee`
- `items[].toAssignee`
- `items[].reason`

#### 请求示例

```bash
curl -H "Authorization: Bearer <token>" \
  "http://127.0.0.1:9080/api/v1/workflow/me/task-records?start=0&size=20&actionType=TRANSFER"
```

## 11. 任务动作接口

### 11.1 职责边界

这一节非常重要，开发、测试、流程设计人员必须统一理解。

`go-common/workflow` 当前把任务相关接口明确拆成两类职责：

第一类，任务办理权管理接口：

- `claim`
- `unclaim`
- `delegate`
- `resolve`
- `transfer`

这 5 个接口只负责处理“这件任务现在由谁来办”。

它们解决的是：

- 候选任务是否已经被认领
- 当前办理人是否要把任务放回候选池
- 当前办理人是否要临时委派给别人
- 受托人处理完成后是否归还给原办理人
- 当前任务是否要彻底转交给另一个办理人

它们不解决的是：

- 流程是否流转到下一节点
- BPMN 网关走哪条线
- 退回、驳回、重审、补正应该走哪个分支

第二类，流程推进接口：

- `complete`

`complete` 只负责处理“流程往哪走”。

它解决的是：

- 当前节点处理完成
- 本次处理结果写入流程变量
- Flowable 根据变量和 BPMN 条件表达式决定下一节点

它不负责的是：

- 抢占候选任务
- 释放任务
- 委派
- 转办

可以这样理解：

- `claim / unclaim / delegate / resolve / transfer`
  - 管的是“谁来办”
- `complete`
  - 管的是“办完以后流程往哪走”

如果只记一句话，请记住：

- 想改办理人，用任务动作接口。
- 想推进流程，用 `complete`。

### 11.2 `POST /api/v1/workflow/tasks/{id}/claim`

#### 作用

- 把候选任务认领到当前登录人名下。

#### 使用场景

- 候选任务先认领再处理
- 多人候选池中先明确谁来办

#### 路径参数

- `id`
  - 运行时任务 ID

#### 请求体

- 无

#### 返回字段

- `taskId`
  - 当前任务 ID
- `status`
  - 固定为 `claimed`
- `assignee`
  - 认领后的办理人
- `owner`
  - 任务所有者；普通认领场景下一般为空
- `delegationState`
  - 委派状态；普通认领场景下一般为空

#### 返回示例

```json
{
  "data": {
    "taskId": "task-001",
    "status": "claimed",
    "assignee": "test1",
    "owner": "",
    "delegationState": ""
  },
  "message": "success"
}
```

#### 请求示例

```bash
curl -X POST \
  -H "Authorization: Bearer <token>" \
  http://127.0.0.1:9080/api/v1/workflow/tasks/<taskId>/claim
```

- `owner`
  - 任务所有者
- `delegationState`

#### 返回示例

```json
{
  "data": {
    "taskId": "task-001",
    "status": "unclaimed",
    "assignee": "",
    "owner": "",
    "delegationState": ""
  },
  "message": "success"

```

#### 请求示例

```bash

  -H "Authorization: Bearer <token>" \
  http://127.0.0.1:9080/api/v1/workflow/tasks/<taskId>/unclaim
```

### 11.4 `POST /api/v1/workflow/tasks/{id}/delegate`

#### 作用

- 临时委派给别人处理。

#### 使用场景

- 当前办理人需要别人协助处理
- 处理完仍要回原办理人拍板

#### 路径参数

- `id`
  - 运行时任务 ID

#### 请求体字段

- `assignee`
  - 受托人的工作流用户 ID

#### 请求体示例

```json
{
  "assignee": "test2"
}
```

#### 返回字段

- `taskId`
  - 当前任务 ID
- `status`
  - 固定为 `delegated`
- `assignee`
  - 受托人
- `owner`
  - 原办理人
- `delegationState`
  - 通常为 `PENDING`

#### 返回示例

```json
{
  "data": {
    "taskId": "task-001",
    "status": "delegated",
    "assignee": "test2",
    "owner": "test1",
    "delegationState": "PENDING"
  },
  "message": "success"
}
```

#### 请求示例

```bash
curl -X POST http://127.0.0.1:9080/api/v1/workflow/tasks/<taskId>/delegate \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <token>" \
  -d '{"assignee":"test2"}'
```

### 11.5 `POST /api/v1/workflow/tasks/{id}/resolve`

#### 作用

- 受托人提交“解决委派”。

#### 使用场景

- 委派处理完成后，把任务还回原办理人

#### 路径参数

- `id`
  - 运行时任务 ID

#### 请求体字段

- `variables`
  - 解决委派时回写的变量集合
  - 这里只回写变量，不推进流程

#### 请求体示例

```json
{
  "variables": {
    "comment": "已完成委派处理",
    "reviewResult": "PASS"
  }
}
```

#### 返回字段

- `taskId`
  - 当前任务 ID
- `status`
  - 固定为 `resolved`
- `assignee`
  - 解决委派后，正常会回到原办理人
- `owner`
  - 原办理人
- `delegationState`
  - 解决后一般清空

#### 返回示例

```json
{
  "data": {
    "taskId": "task-001",
    "status": "resolved",
    "assignee": "test1",
    "owner": "test1",
    "delegationState": ""
  },
  "message": "success"
}
```

#### 请求示例

```bash
curl -X POST http://127.0.0.1:9080/api/v1/workflow/tasks/<taskId>/resolve \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <token>" \
  -d '{
    "variables":{
      "comment":"已完成委派处理",
      "alarmLevel":"ORANGE"
    }
  }'
```

### 11.7 `POST /api/v1/workflow/tasks/{id}/complete`

#### 作用

- 完成当前任务并推动流程进入下一节点。

#### 使用场景

- 审核通过
- 退回重审
- 补充材料
- 专家复核
- 流程结束

#### 路径参数

- `id`
  - 运行时任务 ID

#### 请求体字段

- `activityId`
  - 当前节点 ID
- `result`
  - 业务处理结果
- `comment`
  - 办理备注
- `reworkComment`
  - 退回补充说明
- `payloadRef`
  - 业务数据引用 ID
- `needExpert`
  - 是否需要专家复核
- `variables`
  - 最终写入流程实例的变量集合

#### 请求体示例

```json
{
  "activityId": "task_review",
  "result": "REWORK",
  "comment": "退回申请人补充资料",
  "reworkComment": "缺少关键附件",
  "payloadRef": "BIZ-20260610-0001",
  "needExpert": false,
  "variables": {
    "result": "REWORK",
    "nextCandidateGroups": [
      "role_REVIEW_GROUP"
    ]
  }
}
```

#### 边界说明

- `complete` 才会推动 BPMN
- 已委派且 `delegationState=PENDING` 的任务不能直接 `complete`
- 退回、驳回、重审都通过变量和 BPMN 条件表达完成

#### 请求示例

```bash
curl -X POST http://127.0.0.1:9080/api/v1/workflow/tasks/<taskId>/complete \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <token>" \
  -d '{
    "activityId":"task_review",
    "result":"REWORK",
    "comment":"退回申请人补充资料",
    "variables":{"result":"REWORK"}
  }'
```

#### 返回示例

```json
{
  "data": {
    "taskId": "task-001",
    "status": "completed"
  },
  "message": "success"
}
```

## 12. 进度与动作时间线接口

### 12.1 `GET /api/v1/workflow/process-instances/{id}/progress-view`

#### 作用

- 返回流程摘要、BPMN 结构化步骤视图、当前节点总览。

#### 使用场景

- 顶部流程摘要
- 流程图高亮
- 当前节点总览
- 结构化步骤分析

#### 路径参数

- `id`
  - 流程实例 ID

#### 返回字段重点

- `summary`
  - 其中：
    - `summary.startUserId` 表示流程发起人 ID
    - `summary.startUserName` 表示流程发起人显示名
- `steps`
  - 结构顺序仍然按 BPMN 正向步骤展示，不会被并行分支的时间先后打乱
- `completedActivityIds`
- `currentActivityIds`

#### `steps[]` 字段重点

- `steps[].activityId`
  - BPMN 节点 ID
- `steps[].activityName`
  - BPMN 节点名称
- `steps[].status`
  - 当前步骤状态
- `steps[].executions`
  - 该步骤真实发生过的办理执行记录列表

#### `steps[].executions[]` 字段重点

- `taskId`
  - 该次执行对应的任务 ID
- `status`
  - 该次执行当前状态，常见值：`ACTIVE`、`COMPLETED`
- `assignee`
  - 当前或最终办理人
- `owner`
  - 任务 owner
- `candidateUsers`
  - 候选用户
- `candidateGroups`
  - 候选组
- `records`
  - 该次执行下真实发生过的办理记录

#### `steps[].executions[].records[]` 字段重点

- `actionType`
  - 办理动作类型
- `operatorUserId`
  - 动作发起人
- `fromAssignee`
  - 变更前办理人
- `toAssignee`
  - 变更后办理人
- `time`
  - 动作时间

#### 说明

- 这个接口的核心定位是“结构视图”，不是“用户最终看到的审批进度主列表”。
- `steps[]` 顺序按 BPMN 结构展开，因此可能包含：
  - 未来节点
  - 回退节点
  - 尚未走到的分支节点
- 所以前端如果要做“审批进度弹窗 / 审批进度页”，不要直接把 `steps[]` 原样当主列表渲染。
- 推荐用法：
  - 用 `summary` 做顶部摘要
  - 用 `completedActivityIds/currentActivityIds` 做流程图高亮
  - 如果要做“结构分析页”，再使用 `steps[]`

#### 请求示例

```bash
curl -H "Authorization: Bearer <token>" \
  http://127.0.0.1:9080/api/v1/workflow/process-instances/process-001/progress-view
```

### 12.2 `GET /api/v1/workflow/process-instances/{id}/progress-timeline`

#### 作用

- 返回流程真正走过或当前正在处理的节点推进主链路。

#### 使用场景

- 审批进度弹窗
- 业务详情页的“流程步骤”
- 用户希望看懂 `A -> B -> C` 实际流转主线

#### 路径参数

- `id`
  - 流程实例 ID

#### 返回字段重点

- `summary`
- `summary.startUserId`
- `summary.startUserName`
- `items[].itemType`
- `items[].status`
- `items[].sequence`
- `items[].occurrence`
- `items[].activityId`
- `items[].activityName`
- `items[].assignee`
- `items[].candidateUsers`
- `items[].candidateGroups`
- `items[].startTime`
- `items[].endTime`

#### 说明

- 这是前端“流程步骤主列表”推荐优先使用的接口。
- `items[]` 只返回：
  - 已经真实发生过的步骤
  - 当前正在处理的步骤
- 不会像 `progress-view.steps[]` 一样把整张 BPMN 的未来节点都平铺出来。
- 如果前端需要进一步展示：
  - “谁转办给了谁”
  - “谁委派给了谁”
  - “这一环节的意见是什么”
  则应继续结合 `task-records`，而不是直接改用 `action-timeline` 代替主列表。

#### 请求示例

```bash
curl -H "Authorization: Bearer <token>" \
  http://127.0.0.1:9080/api/v1/workflow/process-instances/process-001/progress-timeline
```

### 12.3 `GET /api/v1/workflow/process-instances/{id}/action-timeline`

#### 作用

- 返回任务办理权动作时间线。

#### 使用场景

- 展示认领、委派、转办、解决委派轨迹
- 单独补足“谁来办”的历史
- 审计排查

#### 路径参数

- `id`
  - 流程实例 ID

#### 返回字段重点

- `summary`
- `items[].actionType`
- `items[].operatorUserId`
- `items[].operatorUserName`
- `items[].fromAssignee`
- `items[].toAssignee`
- `items[].fromOwner`
- `items[].toOwner`
- `items[].reason`
- `items[].time`

#### 请求示例

```bash
curl -H "Authorization: Bearer <token>" \
  http://127.0.0.1:9080/api/v1/workflow/process-instances/process-001/action-timeline
```

#### 说明

- 这个接口现在仍然保留，但推荐定位是“动作审计辅助接口”。
- 它不负责表达完整 BPMN 节点推进顺序。
- 前端如果直接把它拿来做“审批进度主列表”，通常会出现两个问题：
  - 用户看不出主步骤顺序
  - 节点推进和办理权变化混在一起，容易重复、难读
- 推荐：
  - 主链路看 `progress-timeline`
  - 节点补充动作看 `task-records`
  - 仅在需要单独审计“谁把任务交给了谁”时，再额外展示 `action-timeline`

### 12.7 `GET /api/v1/workflow/process-instances/{id}/task-records`

#### 作用

- 按流程实例查询该流程完整的办理记录列表。

#### 使用场景

- 审计一条流程里到底谁认领、谁转办、谁完成过
- 给 `progress-timeline` 的每个步骤补充动作细节
- 排查为什么某一环节最后到了某个人手里

#### 路径参数

- `id`
  - 流程实例 ID

#### 返回字段重点

- `summary`
- `items[].actionType`
- `items[].time`
- `items[].taskId`
- `items[].activityId`
- `items[].operatorUserId`
- `items[].fromAssignee`
- `items[].toAssignee`

#### 说明

- 这个接口不是“主步骤接口”，而是“步骤补充记录接口”。
- 推荐搭配方式：
  - `progress-timeline` 决定主列表顺序
  - `task-records` 决定该步骤下展示哪些动作细节
- 常见动作类型：
  - `START_PROCESS`
  - `CLAIM`
  - `UNCLAIM`
  - `DELEGATE`
  - `RESOLVE`
  - `TRANSFER`
  - `COMPLETE`
- 前端展示时建议注意：
  - `START_PROCESS` 通常不要再重复挂到第一个步骤下面，否则容易出现“两个发起流程”的重复展示
  - `TRANSFER / DELEGATE / RESOLVE / CLAIM / UNCLAIM` 更适合作为步骤下的“流转补充文案”
  - `COMPLETE` 更适合作为“办理意见 / 已处理完成”来源之一

#### 请求示例

```bash
curl -H "Authorization: Bearer <token>" \
  http://127.0.0.1:9080/api/v1/workflow/process-instances/process-001/task-records
```

#### 作用

- 兼容旧前端的复合图接口。

#### 使用场景

- 老页面未切到统一 `diagram-view`

#### 请求示例

```bash
curl -H "Authorization: Bearer <token>" \
  http://127.0.0.1:9080/api/v1/workflow/process-instances/process-001/composite-diagram
```

## 14. 回调接口

### 14.1 3 分钟看懂：`POST /flowable/callback`

这一节已经整合了《Flowable Callback 接口 3 分钟速读版》的核心内容。
如果你现在只想快速弄明白这个接口怎么配、谁来调、收到后做什么，直接看完这一节即可。

#### Q1. 我需要前端调用这个接口吗？

不需要。

直接结论：

1. `POST /flowable/callback` 不是前端调的。
2. 它也不是业务页面按钮直接调的。
3. 它通常是 Flowable 在流程运行过程中自动回调到业务系统的一个内部接口。
4. 你要让它生效，关键不是“手动调这个接口”，而是“启动流程时把 `callbackUrl` 和 `callbackEvents` 传给 Flowable”。
5. 当前 `go-common` 标准实现收到回调后，只会刷新工作流侧读模型，不会自动更新你的业务主表。
6. 如果你想在“节点完成后同步改业务表状态”，要基于这个回调事件做业务扩展处理。

#### Q2. 那到底是谁来调用它？

- 正常调用方：Flowable
- 非正常调用方：前端页面、业务按钮、人工日常操作

也就是说，正常链路是：

1. 业务系统调用 `POST /api/v1/workflow/process/start`
2. 请求里通过 `variables.callbackUrl` 告诉 Flowable：“后面事件发生时，请回调这个地址”
3. Flowable 在流程运行过程中，自动 `POST /flowable/callback`
4. `go-common` 收到后刷新待办、已办、流程摘要等工作流侧数据

#### Q3. 它什么时候会被调用？

最常见是 4 类事件：

- `PROCESS_STARTED`
  - 流程刚启动
- `NODE_STARTED`
  - 某个节点刚进入运行态
- `NODE_ENDED`
  - 某个节点刚处理完成
- `PROCESS_ENDED`
  - 整个流程结束

如果你在启动流程时没有传 `callbackUrl` 或没有声明 `callbackEvents`，那就不会有这类自动回调。

#### Q4. 我真正应该怎么接这个接口？

真正的使用方式不是“去写一个页面点这个接口”，而是下面这条链路：

第 1 步，启动流程时配置回调地址和事件：

```json
{
  "processDefinitionKey": "asset_approval_process",
  "bizId": "BIZ-20260610-0001",
  "bizType": "asset_approval",
  "title": "资产审批申请示例",
  "variables": {
    "callbackUrl": "http://127.0.0.1:9080/flowable/callback",
    "callbackEvents": "PROCESS_STARTED,NODE_STARTED,NODE_ENDED,PROCESS_ENDED"
  }
}
```

第 2 步，Flowable 在事件发生时自动回调：

```http
POST /flowable/callback
Content-Type: application/json
```

```json
{
  "eventId": "evt-001",
  "eventType": "NODE_ENDED",
  "eventTime": "2026-06-10T10:00:00+08:00",
  "processInstanceId": "process-001",
  "processDefinitionId": "asset_approval_process:3:abc",
  "activityId": "task_review",
  "taskId": "task-001",
  "bizId": "BIZ-20260610-0001",
  "variables": {
    "bizType": "asset_approval",
    "title": "资产审批申请示例",
    "result": "APPROVED"
  }
}
```

第 3 步，`go-common` 返回成功：

```json
{
  "data": {
    "accepted": true
  },
  "message": "success"
}
```

这就是 `flowable/callback` 的标准使用方式。

#### Q5. 启动流程时到底要传什么？

你真正需要关心的是启动流程时下面两个变量：

- `callbackUrl`
  - 含义：Flowable 以后要把事件回调到哪个地址
  - 示例：`http://127.0.0.1:9080/flowable/callback`
- `callbackEvents`
  - 含义：哪些事件发生时要回调
  - 示例：`PROCESS_STARTED,NODE_STARTED,NODE_ENDED,PROCESS_ENDED`

最推荐的启动示例：

```json
{
  "processDefinitionKey": "asset_approval_process",
  "bizId": "BIZ-20260610-0001",
  "bizType": "asset_approval",
  "title": "资产审批申请示例",
  "name": "资产审批申请示例",
  "variables": {
    "starterId": "test1",
    "starterName": "测试用户1",
    "callbackUrl": "http://127.0.0.1:9080/flowable/callback",
    "callbackEvents": "PROCESS_STARTED,NODE_STARTED,NODE_ENDED,PROCESS_ENDED"
  }
}
```

如果你现在问“我到底该不该配回调”，可以直接这样判断：

- 只想用待办、已办、流程进度，并希望这些信息实时刷新：建议配
- 还想在节点完成后联动更新业务表状态：必须配
- 只是做极简演示，不关心回调和状态同步：可以先不配

#### Q6. 为什么我没有收到回调？

先按下面顺序排查：

1. 启动流程时有没有传 `variables.callbackUrl`
2. 启动流程时有没有传 `variables.callbackEvents`
3. `callbackUrl` 地址是否能被 Flowable 所在环境访问到
4. 你订阅的事件里是否包含当前发生的事件
5. 如果配置了 `callback_secret`，调用方是否按要求带了签名头

最常见的原因就两个：

- 根本没配 `callbackUrl / callbackEvents`
- 地址配置了，但 Flowable 根本打不到这个地址

#### Q7. 请求头要不要签名？

默认情况下：

- `Content-Type: application/json`

如果你配置了 `workflow.api.callback_secret`，还必须带下面 3 个头：

- `X-Timestamp`
  - 当前时间戳
- `X-Nonce`
  - 随机串
- `X-Signature`
  - 签名结果

签名规则是：

```text
timestamp + "\n" + nonce + "\n" + body
```

签名算法是：

```text
HMAC-SHA256
```

也就是说：

- 没配 `callback_secret`
  - 不验签
- 配了 `callback_secret`
  - 少任何一个签名头都会返回 `401 Unauthorized`

#### Q8. 请求体每个字段到底是什么意思？

- `eventId`
  - 本次回调事件 ID
  - 建议业务扩展侧拿它做幂等去重
- `eventType`
  - 本次事件类型
  - 最常见就是 `PROCESS_STARTED`、`NODE_STARTED`、`NODE_ENDED`、`PROCESS_ENDED`
- `eventTime`
  - 本次事件发生时间
- `processInstanceId`
  - 流程实例 ID
  - 查流程进度、查流程图、更新业务流程状态时很常用
- `processDefinitionId`
  - 流程定义 ID
  - 主要用于调试、区分具体部署版本
- `activityId`
  - 当前节点 ID
  - 例如：`task_review`
  - 业务扩展时经常用它判断“是哪个节点结束了”
- `taskId`
  - 当前任务 ID
  - 对任务级事件最有用
- `bizId`
  - 你的业务主键
  - 回写业务表最关键的关联字段之一
- `variables`
  - 当时的流程变量快照
  - 里面通常会带标题、结果、发起人、下一节点变量等

#### Q9. 返回 `accepted=true` 到底代表什么？

返回示例：

```json
{
  "data": {
    "accepted": true
  },
  "message": "success"
}
```

字段含义：

- `accepted=true`
  - 表示这个回调请求已经被当前标准处理链路接收
  - 包含：验签通过、JSON 解析通过、标准回调处理通过

这个返回值的重点不是给前端展示，而是告诉回调方：“这条事件我收到了”。

#### Q10. 当前 `go-common` 收到回调后到底做了什么？

当前标准实现只做工作流平台侧的同步：

- `PROCESS_STARTED`
  - 刷新流程摘要
- `NODE_STARTED`
  - 同步运行中待办
  - 刷新流程摘要
- `NODE_ENDED`
  - 同步已办
  - 删除运行中待办
  - 刷新流程摘要
- `PROCESS_ENDED`
  - 清理流程实例下的运行中待办
  - 刷新流程摘要

一句话：

- 它主要解决“待办、已办、流程进度要及时更新”

#### Q11. 为什么回调成功了，但业务表状态没变？

当前标准实现不会自动帮你做下面这些业务动作：

- 不会自动更新业务主表状态
- 不会自动更新业务审核字段
- 不会自动插入业务审核日志
- 不会自动发短信、发站内信、发邮件
- 不会自动做你的行业逻辑补偿

因为当前标准实现不会自动改你的业务主表。

所以千万不要误解成：

- “我把 `callbackUrl` 配上，业务表状态就会自动变”

当前标准实现不是这个语义。

#### Q12. 那它支不支持“节点完成后更新业务表字段”？

支持，但这是“基于回调事件做业务扩展”，不是“平台已经默认帮你写好”。

最典型的扩展方式如下：

- 收到 `PROCESS_STARTED`
  - 把 `processInstanceId` 回写到你的业务主表
  - 记录 `workflow_started_at`
- 收到 `NODE_STARTED`
  - 更新当前处理节点编码
  - 更新当前处理节点名称
- 收到 `NODE_ENDED`
  - 更新本节点处理结果
  - 记录最近处理时间
  - 追加业务审核日志
- 收到 `PROCESS_ENDED`
  - 更新业务流程总状态
  - 记录流程完成时间

#### Q13. 我应该在哪一层写业务状态更新？

应该写在业务系统自己的事件处理层，不应该误以为写在前端，也不应该误以为 `go-common` 默认已经帮你写好了。

推荐分层：

1. `go-common` 标准回调接收层
   - 负责验签
   - 负责解析回调请求
   - 负责刷新工作流侧读模型
2. 业务系统事件处理层
   - 负责更新业务主表
   - 负责写业务日志
   - 负责通知、归档、统计等扩展逻辑

一句话：

- 工作流平台负责“事件到了”
- 业务系统负责“业务字段怎么变”

#### Q14. 给我一个最实际的业务扩展示例

假设你的业务表叫 `biz_order`。

收到 `NODE_ENDED` 时，可以这样理解：

```text
根据 bizId 找到业务单
根据 activityId 判断是哪一个节点完成
根据 variables.result 更新本次处理结果
根据 eventTime 更新最后处理时间
插入一条业务审核日志
```

可能对应的业务字段语义是：

- `biz_order.current_node_code = 'task_review'`
- `biz_order.current_node_status = 'DONE'`
- `biz_order.last_result = 'APPROVED'`
- `biz_order.last_operated_at = eventTime`

收到 `PROCESS_ENDED` 时，可以这样理解：

- `biz_order.workflow_status = 'COMPLETED'`
- `biz_order.completed_at = eventTime`

#### Q15. 回调重复推送怎么办？

因为回调有可能重复投递，所以建议至少这样处理：

1. 优先用 `eventId` 做幂等键
2. 如果上游没有稳定 `eventId`，就组合：
   - `processInstanceId`
   - `taskId`
   - `activityId`
   - `eventType`
   - `eventTime`
3. 业务日志表最好建唯一索引，避免重复写入

#### Q16. 联调时我能不能手工测？

正式环境不应该手工调用，但联调时可以用 `curl` 模拟一条回调：

```bash
curl -X POST http://127.0.0.1:9080/flowable/callback \
  -H "Content-Type: application/json" \
  -d '{
    "eventId":"evt-001",
    "eventType":"NODE_ENDED",
    "eventTime":"2026-06-10T10:00:00+08:00",
    "processInstanceId":"process-001",
    "processDefinitionId":"asset_approval_process:3:abc",
    "activityId":"task_review",
    "taskId":"task-001",
    "bizId":"BIZ-20260610-0001",
    "variables":{
      "bizType":"asset_approval",
      "result":"APPROVED"
    }
  }'
```

如果配置了 `callback_secret`，还要额外补签名头，否则会直接返回 `401 Unauthorized`。

#### Q17. 最后一句话怎么记？

- `flowable/callback` 不是“给页面调用的业务接口”。
- 它是“Flowable 自动回调过来的内部事件接口”。
- 当前标准能力负责刷新工作流侧数据。
- 业务表要不要跟着更新，要由业务系统基于回调事件自己扩展。

## 15. 前端推荐页面结构

前端推荐至少拆成下面几类页面：

1. 发起页
2. 我的申请页
3. 我的待办页
4. 我的已办页
5. 业务详情页
6. 任务处理页
7. 流程进度页

### 15.1 处理页建议分三块

1. 业务详情
2. 任务管理动作
   - 认领
   - 释放
   - 委派
   - 转办
   - 解决委派
3. 业务处理动作
   - 通过
   - 退回
   - 驳回
   - 补正

### 15.2 时间线建议分两块

1. 主进度链路
   - `progress-timeline`
   - 用来展示真正走过或当前正在处理的步骤
2. 节点补充动作
   - `task-records`
   - 用来展示某一步里发生的转办、委派、解决委派、完成等动作
3. 动作审计页
   - `action-timeline`
   - 只在需要单独审计“办理权如何变化”时使用

### 15.3 审批进度页推荐组合

推荐组合：

1. 顶部摘要和流程图
   - `progress-view`
2. 主步骤列表
   - `progress-timeline`
3. 每个步骤下的动作补充
   - `task-records`

不要这样组合：

1. 直接把 `progress-view.steps[]` 当审批进度主列表
2. 直接把 `action-timeline` 当审批进度主列表

推荐展示逻辑：

1. 主列表一行只表示一个真实步骤
2. 该步骤的处理人、创建时间、完成时间、状态来自 `progress-timeline.items[]`
3. 该步骤下的转办 / 委派 / 解决委派 / 完成意见来自 `task-records`
4. `START_PROCESS` 不建议重复挂入第一个步骤的动作列表

前端伪代码示例：

```ts
const [progressView, progressTimeline, taskRecords] = await Promise.all([
  getBizProgressView(bizId),
  getBizProgressTimeline(bizId),
  getBizTaskRecords(bizId),
]);

const stepList = progressTimeline.items.map((step) => {
  const records = taskRecords.items.filter((record) => {
    if (record.actionType === "START_PROCESS") {
      return false;
    }
    if (step.taskId && record.taskId === step.taskId) {
      return true;
    }
    return !step.taskId && step.activityId && record.activityId === step.activityId;
  });

  return {
    step,
    records,
  };
});
```
